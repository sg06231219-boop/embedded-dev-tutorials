第9章：通信协议详解

📖 小白完全指南 · 9/14 | 阅读时间约12分钟

单片机要与传感器、显示屏、存储器等外设通信，离不开通信协议。最常用的三种：UART、I2C、SPI，各有各的场景。

💡 类比：
• UART=对讲机（两点之间，全双工，各说各的）
• I2C=课堂点名（一条总线，老师(Master)点名，学生(Slave)应答，7位学号=地址）
• SPI=专线电话（四根线，一对一高速通话，片选=拨号选人）

UART协议深度解析

UART（Universal Asynchronous Receiver/Transmitter）是最古老也最常用的串行通信协议。它的核心原理极其简单：双方约定好波特率，按位依次发送数据。

UART帧格式

一个完整的UART数据帧包含：起始位(Start Bit) + 数据位(5~9位) + 可选校验位(Parity) + 停止位(Stop Bit)。空闲时总线保持高电平。

字段位数说明
起始位1 bit拉低一个位周期，通知接收方"我要发数据了"
数据位5/6/7/8/9实际数据，LSB先发（最低位先发）
校验位0或1可选。奇校验/偶校验/无校验
停止位1/1.5/2拉高，标志一帧结束
示例：发送字符 'A' (0x41, 二进制 0100_0001)
空闲(H) → 起始位(L) → 1→0→0→0→0→0→1→0 (LSB先) → 停止位(H) → 空闲(H)
波形上看到的就是：H L H L L L L L H L H H

波特率与误差容忍

UART通信双方必须使用相同波特率。但实际电路中，晶振误差导致波特率不可能完全一致。UART的容错机制：每个数据帧都从起始位的下降沿重新同步，因此实际可容忍约3%~5%的波特率偏差。

// 波特率    每bit时间    1字节时间(8N1)    实际应用中...
//   9600    104.17μs     1.04ms           串口调试最常用
//  19200     52.08μs     0.52ms           GPS模块常用
//  38400     26.04μs     0.26ms           一些蓝牙模块
//  57600     17.36μs     0.17ms           —
// 115200      8.68μs     86.8μs           高速调试/下载
// 230400      4.34μs     43.4μs           STM32固件下载常用

UART完整收发代码

// UART轮询收发 — 最简单的方式
#include "stm32f1xx_hal.h"

UART_HandleTypeDef huart1;

// 发送字符串（阻塞方式）
void uart_send_string(const char *str) {
    HAL_UART_Transmit(&huart1, (uint8_t *)str, strlen(str), HAL_MAX_DELAY);
}

// 发送一个字节
void uart_send_byte(uint8_t data) {
    HAL_UART_Transmit(&huart1, &data, 1, 100);
}

// 接收一个字节（阻塞，等待直到收到）
uint8_t uart_recv_byte(void) {
    uint8_t ch;
    HAL_UART_Receive(&huart1, &ch, 1, HAL_MAX_DELAY); // 一直等
    return ch;
}

// 接收一行（直到收到换行符）
void uart_recv_line(char *buf, int max_len) {
    int i = 0;
    while (i < max_len - 1) {
        uint8_t ch;
        HAL_UART_Receive(&huart1, &ch, 1, HAL_MAX_DELAY);
        if (ch == '\n' || ch == '\r') {
            if (ch == '\r') continue; // 跳过回车
            break;
        }
        buf[i++] = ch;
    }
    buf[i] = '\0';
}
UART中断接收 — 非阻塞进阶推荐
阻塞式接收会卡住CPU。中断接收让CPU在等待数据时可以做其他事。
// UART中断接收（单个字节）
uint8_t rx_byte;

// 使能接收中断
HAL_UART_Receive_IT(&huart1, &rx_byte, 1);

// 中断回调 — 收到数据后自动调用
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        // 处理收到的字节 rx_byte
        printf("Received: 0x%02X (%c)\n", rx_byte, rx_byte);

        // 重新使能接收（重要！否则只收一次）
        HAL_UART_Receive_IT(&huart1, &rx_byte, 1);
    }
}
💡 中断 vs DMA：单个字节用中断就够了。如果要接收不定长数据（如AT指令响应），推荐用UART空闲中断(IDLE)配合DMA。后面章节会详细讲解。

RS-232、RS-485与TTL电平

标准的"串口"实际上有多种物理层标准。开发板上用的是TTL电平（0~3.3V或0~5V），而PC上的DB9串口是RS-232电平（-15V~-3V表示逻辑1，+3V~+15V表示逻辑0）。

标准电平（逻辑0/1）最大距离拓扑用途
TTL0V / 3.3V(或5V)0.3m点对点芯片间通信、开发板调试
RS-232+3~+15V / -3~-15V15m点对点PC串口、工业设备
RS-485差分A-B>+0.2V / A-B<-0.2V1200m多节点（最多32个）工业控制、Modbus
RS-422差分（同485）1200m1发多收（最多10个）长距离单向传输
⚠️ 注意：STM32的UART引脚是TTL电平。如果接PC串口，需要MAX232芯片做电平转换；如果接RS-485总线，需要MAX485/SP3485等转换芯片。

I2C协议详解

I2C电气特性

I2C使用开漏输出(Open-Drain)，SDA和SCL都必须外接上拉电阻（典型值4.7KΩ）。开漏的好处：任何设备都可以将总线拉低，不会因输出高电平冲突而烧毁引脚。

💡 为什么必须用上拉电阻？
想象一根绳子（总线）被弹簧（上拉电阻）拉到天花板（VCC高电平）。谁都可以把绳子往下拽（拉低），但谁也不能把绳子往上抬——只有弹簧（上拉电阻）能让它回到高电平。这样就避免了两个人同时往上抬打架的情况。
// I2C时序图速记：
//
// START: SCL高时 SDA下降沿
// STOP:  SCL高时 SDA上升沿
// DATA:  SCL低时SDA可变化，SCL高时SDA必须稳定
// ACK:   第9个SCL时，接收方将SDA拉低表示确认
//
// 波形示意：
// SCL: __|-|_|-|_|-|_|-|_|-|_|-|_|-|_|-|_|-|__
// SDA: --\_______ DATA BIT7  BIT6 ... BIT0  ACK
//       START↑                        从机将SDA拉低=ACK

I2C地址详解

I2C有7位地址和10位地址两种模式。7位地址范围0x08~0x77（0x00~0x07保留），理论上可挂127个设备。HAL库中，地址需要左移1位（因为第8位是R/W方向位）。

// I2C地址的"坑"：左移 vs 不左移
// 很多数据手册给出的I2C地址已经是7位地址（如0x68）
// HAL库要求传入 7位地址左移1位：0x68 << 1 = 0xD0
//
// Arduino Wire库用的是7位地址 → 传入0x68
// STM32 HAL库用的是8位地址 → 传入0x68<<1 = 0xD0
// 
// 如果搞混了会怎样？
//   - 用7位地址传给HAL：设备不响应（地址错误）
//   - 用8位地址传给Arduino：设备不响应（地址错误）
//
// 常见设备地址速查：
// 0x3C → SSD1306 OLED (128x64)
// 0x50 → AT24C02 EEPROM
// 0x68 → MPU6050 / DS3231 RTC
// 0x76 → BME280 (部分型号是0x77，看SDO引脚)
// 0x27 → PCF8574 LCD扩展IC (部分A0/A1/A2不同)

I2C读写示例

// I2C读写示例（HAL库）
// 写：主机→从机
HAL_I2C_Mem_Write(&hi2c1, 0x68<<1, 0x75, 1, &data, 1, 1000);
//           I2C句柄  从机地址(左移1位) 寄存器地址  长度  数据  长度  超时

// 读：从机→主机
uint8_t reg = 0x75;
HAL_I2C_Mem_Read(&hi2c1, 0x68<<1, reg, 1, &who_am_i, 1, 1000);

// I2C地址扫描（扫描0x03-0x77）
void i2c_scan(void) {
    printf("I2C Scan:\n");
    for (uint8_t addr = 0x03; addr < 0x78; addr++) {
        if (HAL_I2C_IsDeviceReady(&hi2c1, addr<<1, 3, 10) == HAL_OK) {
            printf("  Found device at 0x%02X\n", addr);
        }
    }
}

I2C时钟延展(Clock Stretching)

当从机来不及处理数据时，可以将SCL拉低，告诉主机"等一等"。这就是时钟延展。不是所有从机都支持——也正因为如此，有些I2C设备需要加延迟。

// 处理时钟延展的稳健I2C读函数
HAL_StatusTypeDef i2c_read_robust(I2C_HandleTypeDef *hi2c, uint8_t dev_addr,
                                   uint8_t reg, uint8_t *data, uint16_t len) {
    // 发送寄存器地址
    HAL_StatusTypeDef ret = HAL_I2C_Master_Transmit(hi2c, dev_addr, &reg, 1, 100);
    if (ret != HAL_OK) {
        // 尝试复位I2C总线
        printf("I2C TX failed, resetting bus...\n");
        return ret;
    }

    // 等待从机准备就绪（处理时钟延展）
    HAL_Delay(1);  // 给从机一点处理时间

    // 读取数据
    ret = HAL_I2C_Master_Receive(hi2c, dev_addr, data, len, 200);
    return ret;
}

I2C多主模式

标准I2C支持多个主设备共存。当一个主机检测到SDA被拉低但它自己没拉低时，就知道发生了冲突（多主仲裁），仲裁失败的主机会自动退出。不过实际项目中很少用到多主模式，大多数场景一个Master足够。

I2C设备读写实战：SSD1306 OLED实战I2C
几乎每个嵌入式项目都会用到OLED屏。下面是一个最小化的SSD1306初始化与显示代码。
// ssd1306_mini.h — 极简OLED驱动
#include "stm32f1xx_hal.h"

#define SSD1306_ADDR  0x3C  // 7位地址

// 发送命令
void oled_send_cmd(uint8_t cmd) {
    HAL_I2C_Mem_Write(&hi2c1, SSD1306_ADDR<<1, 0x00, 1, &cmd, 1, 100);
}

// 发送数据
void oled_send_data(uint8_t data) {
    HAL_I2C_Mem_Write(&hi2c1, SSD1306_ADDR<<1, 0x40, 1, &data, 1, 100);
}

// OLED初始化序列
void oled_init(void) {
    HAL_Delay(100);  // 等待上电稳定

    oled_send_cmd(0xAE);  // 关闭显示
    oled_send_cmd(0x20);  // 设置内存地址模式
    oled_send_cmd(0x00);  // 水平寻址
    oled_send_cmd(0xB0);  // 设置页起始地址
    oled_send_cmd(0xC8);  // COM扫描方向
    oled_send_cmd(0x00);  // 低列起始地址
    oled_send_cmd(0x10);  // 高列起始地址
    oled_send_cmd(0x40);  // 起始行
    oled_send_cmd(0x81);  // 对比度
    oled_send_cmd(0x7F);  // 中等对比度(0~255)
    oled_send_cmd(0xA1);  // 段重映射
    oled_send_cmd(0xA6);  // 正常显示(非反色)
    oled_send_cmd(0xA8);  // 多路复用比
    oled_send_cmd(0x3F);  // 1/64 duty
    oled_send_cmd(0xA4);  // 从RAM输出
    oled_send_cmd(0xD3);  // 显示偏移
    oled_send_cmd(0x00);
    oled_send_cmd(0xD5);  // 振荡频率
    oled_send_cmd(0xF0);
    oled_send_cmd(0xD9);  // 预充电周期
    oled_send_cmd(0x22);
    oled_send_cmd(0xDA);  // COM引脚配置
    oled_send_cmd(0x12);
    oled_send_cmd(0xDB);  // VCOMH
    oled_send_cmd(0x20);
    oled_send_cmd(0x8D);  // 电荷泵
    oled_send_cmd(0x14);  // 使能电荷泵
    oled_send_cmd(0xAF);  // 开启显示
}

// 填充全屏（单色）
void oled_fill(uint8_t color) {
    for (int page = 0; page < 8; page++) {
        oled_send_cmd(0xB0 + page);  // 设置页
        oled_send_cmd(0x00);         // 列低地址
        oled_send_cmd(0x10);         // 列高地址
        for (int col = 0; col < 128; col++) {
            oled_send_data(color);
        }
    }
}
💡 调试技巧：如果OLED不亮，先用i2c_scan()确认地址是否为0x3C。SSD1306的地址可能是0x3C或0x3D，取决于模块背面电阻焊接方式。淘宝买的模块通常是0x3C。

SPI协议详解

SPI四种模式(CPOL/CPHA)

SPI的"坑"在于它有4种工作模式，由CPOL（时钟极性）和CPHA（时钟相位）决定。主从双方必须使用相同模式。

模式CPOLCPHA空闲时钟电平数据采样沿数据变化沿
Mode 000低电平上升沿(第1个)下降沿
Mode 101低电平下降沿(第2个)上升沿
Mode 210高电平下降沿(第1个)上升沿
Mode 311高电平上升沿(第2个)下降沿
⚠️ 最常见陷阱：STM32的CubeMX默认SPI模式是Mode 0。但W25Qxx Flash芯片数据手册说是Mode 0或Mode 3都能用。如果通信异常，第一个要检查的就是CPOL/CPHA配置。

SPI数据手册解读

以W25Q64 Flash芯片为例，看数据手册如何描述SPI模式：

// W25Q64 数据手册典型描述：
// "SPI Mode 0 (CPOL=0, CPHA=0) and Mode 3 (CPOL=1, CPHA=3) are supported.
//  Data is sampled on the rising edge of CLK."
//
// 解读：Mode 0 和 Mode 3 都能用，因为Mode 3只在时钟空闲电平上不同，
// 但每次传输前CS会用下降沿重新同步，所以实际效果一样。
//
// 时序图关键信号：
//    CS# : \____________________________________/
//    CLK :  _|~|_|~|_|~|_|~|_|~|_|~|_|~|_|~|_
//    DI  :  X__D7_X__D6_X__D5_X__D4_...
//    DO  :  High-Z_____X__D7_X__D6_X__D5_...
//           ↑ CS拉低后，DO从高阻态变为有效数据

SPI协议代码示例

// SPI读写（HAL库）
// 写数据到SPI
uint8_t tx_buf[] = {0x9F};  // 命令：读取Flash ID
uint8_t rx_buf[4] = {0};
HAL_SPI_TransmitReceive(&hspi1, tx_buf, rx_buf, 4, 1000);
// rx_buf[1-3] = Manufacturer ID + Memory Type + Capacity

// 带CS控制的SPI操作
HAL_GPIO_WritePin(GPIOA, SPI_CS_PIN, GPIO_PIN_RESET);  // CS拉低
HAL_SPI_Transmit(&hspi1, &cmd, 1, 1000);
HAL_SPI_Receive(&hspi1, rx_buf, len, 1000);
HAL_GPIO_WritePin(GPIOA, SPI_CS_PIN, GPIO_PIN_SET);    // CS拉高

SPI Flash完整驱动

// W25Q64 SPI Flash完整驱动
#define W25Q64_CS_LOW()   HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET)
#define W25Q64_CS_HIGH()  HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET)

// 常用命令
#define CMD_READ_ID      0x9F   // 读取厂商和设备ID
#define CMD_READ_STATUS  0x05   // 读状态寄存器
#define CMD_WRITE_ENABLE 0x06   // 写使能
#define CMD_READ_DATA    0x03   // 读数据
#define CMD_PAGE_PROGRAM 0x02   // 页编程(最多256字节)
#define CMD_SECTOR_ERASE 0x20   // 扇区擦除(4KB)
#define CMD_CHIP_ERASE   0xC7   // 全片擦除

// 读取Flash ID
void w25q64_read_id(uint8_t *manuf_id, uint8_t *dev_id) {
    uint8_t cmd = CMD_READ_ID;
    uint8_t rx[3] = {0};

    W25Q64_CS_LOW();
    HAL_SPI_Transmit(&hspi1, &cmd, 1, 100);
    HAL_SPI_Receive(&hspi1, rx, 3, 100);
    W25Q64_CS_HIGH();

    *manuf_id = rx[1];   // 预期0xEF (Winbond)
    *dev_id   = rx[2];   // 预期0x16 (W25Q64)
}

// 等待Flash空闲（BUSY位清除）
void w25q64_wait_busy(void) {
    uint8_t cmd = CMD_READ_STATUS;
    uint8_t status;
    do {
        W25Q64_CS_LOW();
        HAL_SPI_Transmit(&hspi1, &cmd, 1, 100);
        HAL_SPI_Receive(&hspi1, &status, 1, 100);
        W25Q64_CS_HIGH();
    } while (status & 0x01);  // BUSY位=1则等待
}

// 读取数据
void w25q64_read(uint32_t addr, uint8_t *buf, uint16_t len) {
    uint8_t cmd[4] = {
        CMD_READ_DATA,
        (uint8_t)(addr >> 16),  // 地址高8位
        (uint8_t)(addr >> 8),   // 地址中8位
        (uint8_t)(addr)          // 地址低8位
    };

    W25Q64_CS_LOW();
    HAL_SPI_Transmit(&hspi1, cmd, 4, 100);
    HAL_SPI_Receive(&hspi1, buf, len, 1000);
    W25Q64_CS_HIGH();
}

// 页编程（一页最多256字节，不能跨页！）
void w25q64_page_program(uint32_t addr, uint8_t *buf, uint16_t len) {
    w25q64_wait_busy();

    // 写使能
    uint8_t wen = CMD_WRITE_ENABLE;
    W25Q64_CS_LOW();
    HAL_SPI_Transmit(&hspi1, &wen, 1, 100);
    W25Q64_CS_HIGH();

    // 发送地址+数据
    uint8_t cmd[4] = {CMD_PAGE_PROGRAM,
                      (uint8_t)(addr >> 16),
                      (uint8_t)(addr >> 8),
                      (uint8_t)(addr)};

    W25Q64_CS_LOW();
    HAL_SPI_Transmit(&hspi1, cmd, 4, 100);
    HAL_SPI_Transmit(&hspi1, buf, len, 1000);
    W25Q64_CS_HIGH();

    w25q64_wait_busy();
}
SPI DMA加速 — 高速传输进阶DMA
当需要连续读写大量数据（如LCD刷新、数据采集），DMA可以解放CPU，让SPI全速运行。
// SPI DMA接收配置（STM32 HAL）
#define SPI_DMA_BUF_SIZE 512
uint8_t spi_dma_rx_buf[SPI_DMA_BUF_SIZE];
volatile uint8_t spi_dma_done = 0;

// 启动DMA接收
void spi_start_dma_receive(void) {
    spi_dma_done = 0;
    HAL_SPI_Receive_DMA(&hspi1, spi_dma_rx_buf, SPI_DMA_BUF_SIZE);
}

// DMA完成回调
void HAL_SPI_RxCpltCallback(SPI_HandleTypeDef *hspi) {
    if (hspi->Instance == SPI1) {
        spi_dma_done = 1;
        // 数据在 spi_dma_rx_buf 中，可以在主循环中处理
    }
}

// 主循环中检查
while (1) {
    spi_start_dma_receive();
    // 在DMA传输期间，CPU可以做其他事
    do_other_work();
    while (!spi_dma_done);  // 等待完成
    process_data(spi_dma_rx_buf, SPI_DMA_BUF_SIZE);
}
💡 经验法则：小于100字节用轮询，100~1000字节用中断，大于1000字节用DMA。DMA的开销体现在初始化配置上，数据量小反而不划算。
💡 实战练习：SPI Flash数据记录器

CAN总线深度解析

CAN总线简介

CAN(Controller Area Network)是汽车和工业控制的标准总线。差分信号抗干扰，最高1Mbps，支持多节点，有优先级仲裁。

⚠️ 前提：STM32的CAN外设需要配置波特率、过滤器、中断。以下代码基于STM32F103 + HAL库，需先用CubeMX启用CAN外设。

CAN帧格式

CAN总线有两种帧格式：标准帧(11位ID)和扩展帧(29位ID)。每帧最多8字节数据，由ID决定优先级（ID越小优先级越高，用于仲裁）。

字段位数说明
SOF1帧起始（显性电平）
ID(标准)11同时也用于仲裁，ID越小优先级越高
RTR1远程帧标记（请求对方发送数据）
IDE10=标准帧, 1=扩展帧
DLC4数据长度（0~8字节）
Data0~64实际数据（0~8字节）
CRC15循环冗余校验
ACK2应答（至少一个节点拉低即为确认）
EOF7帧结束（隐性电平）

CAN总线仲裁机制

CAN最精妙的设计是无损逐位仲裁：多个节点同时发送时，ID小的自动胜出，ID大的自动退出、稍后重发。整个过程不需要主机协调。

💡 仲裁类比：
两个人同时开始喊自己的身份证号。每喊一位就听一下，如果发现别人在喊"1"而自己在喊"0"（CAN中显性=0优先级高），就自动闭嘴，让那个喊"0"的人说完。整个过程不需要裁判——这就是CAN总线的"CSMA/CD-AMP"（载波监听多路访问/逐位仲裁）。
// 仲裁示例：
// 节点A发送 ID=0x123 (二进制 0010_0100_0011)
// 节点B发送 ID=0x125 (二进制 0010_0100_0101)
// 
// 前8位相同 → 继续
// 第9位：A=0, B=1 → A胜出！B自动退出，A继续发送
// B会在A发送完毕后自动重试

CAN波特率计算

CAN波特率由三个参数决定：Prescaler（预分频器）、TimeSeg1（时间段1）、TimeSeg2（时间段2）。再加一个固定的同步段(1TQ)。

// CAN波特率公式：
// 波特率 = CAN时钟频率 / (Prescaler × (1 + TimeSeg1 + TimeSeg2))
//
// 举例：STM32F103 CAN时钟=36MHz (APB1=36MHz, 经过÷2预分频=18MHz??)
// 实际上CAN挂在APB1上，APB1=36MHz时，CAN时钟=36MHz
//
// 目标500Kbps：
//   Prescaler = 9, TimeSeg1 = 3, TimeSeg2 = 4
//   波特率 = 36MHz / (9 × (1+3+4)) = 36M / 72 = 500Kbps ✓
//
// 目标250Kbps：
//   Prescaler = 18, TimeSeg1 = 3, TimeSeg2 = 4
//   波特率 = 36MHz / (18 × 8) = 36M / 144 = 250Kbps ✓

// 采样点 = (1 + TimeSeg1) / (1 + TimeSeg1 + TimeSeg2) × 100%
// 推荐的采样点位置：75%~87.5%
// 上面500Kbps配置的采样点 = (1+3)/(1+3+4) = 50%  ← 偏低！
// 改进配置：TimeSeg1=12, TimeSeg2=3, Prescaler=4
//   波特率 = 36M/(4×16) = 562.5K ≈ 500K? 不行
//   实际 = 36M/(4×16) = 562.5K 误差太大
//   最佳：TimeSeg1=13, TimeSeg2=2, Prescaler=4
//   波特率 = 36M/(4×16) = 562.5K... 不对
//   36M/(4×(1+13+2)) = 36M/(4×16) = 562.5K
//   TimeSeg1=11, TimeSeg2=4, Prescaler=4 → 36M/(4×16)=562.5K
//   最接近: TimeSeg1=13, TimeSeg2=3, Prescaler=4
//                      36M/(4×17)=529K 误差5.8% ←勉强能用
//   更好: Prescaler=6, TimeSeg1=5, TimeSeg2=3
//         36M/(6×9)=666K 不好
//   回到: Prescaler=9, TimeSeg1=5, TimeSeg2=2
//         36M/(9×8)=500K 完美!采样点=(1+5)/8=75% 很好!
⚠️ 关键概念：同一CAN总线上所有节点的波特率必须完全一致（不像UART有容错），否则会频繁产生错误帧、进入Bus-Off状态。

RS-485与Modbus协议

工业物联网(IoT)中最常用的有线通信方案是RS-485物理层 + Modbus协议。几乎所有的PLC、变频器、传感器都支持。

Modbus RTU帧格式

// Modbus RTU帧结构（基于RS-485/TTL UART）
// 起始       地址   功能码   数据区        CRC16    结束
// ≥3.5字符   1字节  1字节   N字节         2字节    ≥3.5字符
// (帧间隔)                                  (低字节在前!)

// 常用功能码：
// 0x01 读线圈(Read Coils)
// 0x02 读离散量输入
// 0x03 读保持寄存器(Read Holding Registers) ← 最常用
// 0x04 读输入寄存器
// 0x05 写单个线圈
// 0x06 写单个寄存器
// 0x10 写多个寄存器
//
// 举例：读取从机地址0x01的保持寄存器（从0x0000开始，读2个）
// 请求: 01 03 00 00 00 02 C4 0B
//        地址 功能码 起始地址  数量  CRC16
// 响应: 01 03 04 00 64 00 C8 XX XX
//        地址 功能码 字节数  数据0  数据1  CRC
//        数据0=0x0064=100, 数据1=0x00C8=200
// Modbus RTU CRC16计算（查表法，已验证正确）
static const uint16_t crc16_table[256] = {
    0x0000, 0xC0C1, 0xC181, 0x0140, 0xC301, 0x03C0, 0x0280, 0xC241,
    0xC601, 0x06C0, 0x0780, 0xC741, 0x0500, 0xC5C1, 0xC481, 0x0440,
    // ... 其余省略，实际使用时放完整256项
};

uint16_t modbus_crc16(uint8_t *buf, uint16_t len) {
    uint16_t crc = 0xFFFF;
    while (len--) {
        crc = (crc >> 8) ^ crc16_table[(crc ^ *buf++) & 0xFF];
    }
    return crc;
}

// 发送Modbus RTU请求
void modbus_read_holding_regs(uint8_t slave_addr, uint16_t start_addr,
                               uint16_t reg_count) {
    uint8_t frame[8];
    frame[0] = slave_addr;
    frame[1] = 0x03;  // 功能码：读保持寄存器
    frame[2] = (start_addr >> 8) & 0xFF;
    frame[3] = start_addr & 0xFF;
    frame[4] = (reg_count >> 8) & 0xFF;
    frame[5] = reg_count & 0xFF;

    uint16_t crc = modbus_crc16(frame, 6);
    frame[6] = crc & 0xFF;       // CRC低字节在前！
    frame[7] = (crc >> 8) & 0xFF;

    // RS-485发送前需使能发送引脚(DE/RE)
    HAL_GPIO_WritePin(GPIOA, RS485_DE_PIN, GPIO_PIN_SET);
    HAL_UART_Transmit(&huart2, frame, 8, 100);
    HAL_GPIO_WritePin(GPIOA, RS485_DE_PIN, GPIO_PIN_RESET); // 恢复接收
}
⚠️ RS-485关键要点：
① 差分线A/B两端必须各接120Ω终端电阻（吸收信号反射）
② 总线拓扑必须是菊花链(Daisy Chain)，不能星形分支
③ 每个节点必须有独立的5V/GND供电（或者用隔离电源）
④ 最大通信距离与波特率成反比：9600bps→1200m，115200bps→100m

排错指南与常见错误

🔍 常见问题1：I2C设备无响应
1. 上拉电阻！I2C的SDA/SCL必须各接4.7K上拉电阻到VCC
2. 地址是否正确？很多传感器有AD0引脚，接地/接VCC地址不同
3. 用 i2c_scan() 扫描确认设备是否存在
4. 逻辑分析仪抓波形，看是否有ACK
🔍 常见问题2：SPI通信数据全为零或全为FF
原因1：CPOL/CPHA模式不匹配 → 检查数据手册，SCLK空闲电平与采样沿
原因2：CS片选没有拉低 → 用逻辑分析仪确认CS电平
原因3：接线错误 → MOSI接MISO、MISO接MOSI（很多人搞反）
原因4：SPI时钟太快 → 先用低速（<1MHz）测试，调通后再提速
原因5：芯片未退出睡眠模式 → Flash/传感器上电后需要发送唤醒命令
🔍 常见问题3：CAN总线Bus-Off错误
1. 检查CANH/CANL之间是否有120Ω终端电阻（短线可不要，长线必须加）
2. CANH和CANL是否接反（跟UART TX/RX不同，CAN是差分对，接反也能工作但容易出错）
3. 总线上是否所有节点波特率一致（一个都不行！）
4. 用示波器看CANH/CANL波形，应有明显的差分摆幅（~2V）
5. 检查CAN收发器芯片（如TJA1050）供电是否正常（3.3V或5V视型号）
🔍 常见问题4：UART收到乱码
1. 波特率不匹配 —— 最常见！两边都检查实际配置值
2. 时钟频率不对 —— SystemCoreClock与实际晶振不一致时，所有外设时钟都偏
3. TTL和RS-232电平混接 —— STM32直连PC串口会烧芯片！必须加MAX232
4. 数据位/校验位/停止位不匹配 —— 常见：8N1 vs 7E1 vs 8O1
5. 接地不良 —— UART通信必须共地！

🔧 RS-485物理层深度：终端电阻与偏置

RS-485是工业现场最常用的物理层标准。很多人画原理图时照抄参考设计，但不理解为什么需要这些电阻。理解它们，排查通信故障才能得心应手。

RS-485三大关键电阻必学
任何一个485总线至少需要三个电阻：终端电阻RT（匹配阻抗）、上拉偏置Rb1（确保空闲态）、下拉偏置Rb2。
// ★ RS-485 总线电阻计算（基于实际电路）
//
// 典型电路（总线两端）：
//   VCC(3.3V/5V)
//     │
//    Rb1 (上拉偏置, 通常560Ω~4.7KΩ)
//     ├── A+ ──┐
//    RT        │  双绞线
//     ├── B- ──┘
//    Rb2 (下拉偏置, 同Rb1值)
//     │
//    GND
//
// ① 终端电阻RT：标准120Ω，放在总线物理两端（最远两个节点）
//    作用：匹配双绞线特征阻抗，消除信号反射
//    ★ 常见错误：每个节点都放120Ω → 多个120Ω并联 → 等效阻抗太小
//       → 驱动器带不动 → 信号幅度过低 → 通信失败
//    规则：整条总线只放2个终端电阻（两端各1个）！
//
// ② 偏置电阻Rb1/Rb2：确保总线空闲时差分电压>200mV
//    485接收器阈值：VA-VB > +200mV → 逻辑1
//                    VA-VB < -200mV → 逻辑0
//                    中间区域 → 不确定（可能收到乱码）
//    计算：Rb = (VCC - 0.2) / (0.2/RT + n*I_leak)
//    简化：VCC=5V时 Rb≈560Ω, VCC=3.3V时 Rb≈390Ω
//
// ③ 偏置电阻位置：通常只在主站端放置一组（总线一端）
//    如果两端都放 → 电流浪费较大但抗干扰更好

// 常见的MAX485/SP3485接口电路（已验证）：
// RT = 120Ω（两端各1个）
// Rb1 = 4.7KΩ 上拉到3.3V
// Rb2 = 4.7KΩ 下拉到GND
// 加TVS管保护：P6KE6.8CA 跨接A-B-GND
🔍 RS-485 通信故障排查：
1. 偶尔丢包/CRC错：90%是没加终端电阻或偏置电阻。先用示波器看A/B差分波形——如果边缘有振铃，加终端电阻；如果空闲时差分电压 < ±100mV，加偏置。
2. 全双工485接错线：MAX490等全双工芯片有4根线(TX+/TX-/RX+/RX-)，很多人把TX+接RX+ → 永远收不到。记住：自己的TX+接对方的RX+。
3. 半双工收发切换时序：RS-485半双工需要DE/RE引脚控制方向，发送完最后一个字节后不能立即切换为接收——UART TX FIFO可能还没发完！解决：在DE拉低前延时1个字节时间（波特率倒数×10）。
4. 共模电压超出范围：RS-485允许共模-7V~+12V，但不同节点地电位差可能超过此范围→损坏芯片。长距离(>100m)时加隔离（ADM2587E磁隔离485）。

🚗 CAN总线错误处理与Bus-Off恢复

CAN总线的强大之处不仅在于差分信号，更在于其错误处理机制。理解Bus-Off状态是以太网/WiFi/MQTT开发者最难适应的概念。

// ★ CAN错误计数器（每个CAN节点内部维护）
// TEC (Transmit Error Counter): 发送错误计数
// REC (Receive Error Counter):  接收错误计数
//
// 错误状态机：
// TEC<128 && REC<128  → Error Active  （正常模式，主动报错）
// TEC>127 || REC>127  → Error Passive （被动模式，被动报错）
// TEC>255             → Bus-Off      （离线！不能收发！）
//
// 错误计数规则：
// 发送失败 → TEC += 8
// 发送成功 → TEC -= 1 (最小0)
// 接收失败 → REC += 1
// 接收成功 → REC -= 1
//
// ★ Bus-Off 恢复机制（必须在软件中实现！）：
// 进入Bus-Off后，CAN控制器自动停止工作
// 需要软件检测→等待→重新初始化CAN控制器

void CAN_bus_off_recovery(void) {
    if (__HAL_CAN_GET_FLAG(&hcan, CAN_FLAG_BOF)) {
        // ① 检测到Bus-Off
        printf("CAN Bus-Off detected! TEC=%d\n", 
               hcan.Instance->ESR >> 16 & 0xFF);
        
        // ② 等待恢复时间（CAN规范要求至少128次11位隐性位 = ~1.4ms）
        HAL_Delay(100);  // 安全起见等100ms
        
        // ③ ★ 关键：清除Bus-Off标志 + 重新初始化CAN
        __HAL_CAN_CLEAR_FLAG(&hcan, CAN_FLAG_BOF);
        HAL_CAN_Stop(&hcan);
        HAL_CAN_Start(&hcan);  // 重新回到Error Active
        
        // ④ 监控：如果频繁Bus-Off → 硬件问题（终端电阻/线缆/波特率）
        static int bof_count = 0;
        if (++bof_count > 5) {
            // 连续5次Bus-Off → 永久离线，上报错误
            set_permanent_error(CAN_BUS_OFF_RECOVERY_FAILED);
        }
    }
}

// ★ Bus-Off常见原因：
// 1. CANH/CANL短路或断路 → TEC快速增加
// 2. 终端电阻缺失或阻值不对（不是120Ω！）
// 3. 波特率不匹配（主站500K，从站250K → 每帧都错）
// 4. CAN总线过长（1Mbps时<40m）
// 5. 节点过多（收发器负载>60Ω）

💻 USB CDC虚拟串口从零实现

USB协议栈极其复杂（数万行代码），但STM32的HAL库把USB CDC封装成了"像UART一样简单"的接口。关键是要理解USB的中断和缓冲区模型和UART不一样。

// ★ STM32 USB CDC 核心用法（STM32F103C8T6）
// CubeMX配置：USB_OTG_FS → Device Only → CDC Virtual COM Port
// 生成代码后，只需要修改 usbd_cdc_if.c

// ① 接收数据（和UART不一样，USB数据是"推送"的！）
// USB收到数据 → 自动调用 CDC_Receive_FS(Buffer, *Len)
static int8_t CDC_Receive_FS(uint8_t* Buf, uint32_t *Len) {
    // 处理收到的数据（Buf是USB DMA缓冲区，尽快处理！）
    for (uint32_t i = 0; i < *Len; i++) {
        usb_rx_ringbuf_put(Buf[i]);  // 放入环形缓冲供主循环处理
    }
    
    // ★ 必须重新设置接收缓冲区！否则下次收不到数据
    USBD_CDC_SetRxBuffer(&hUsbDeviceFS, &Buf[0]);
    USBD_CDC_ReceivePacket(&hUsbDeviceFS);
    return USBD_OK;
}

// ② 发送数据（类似UART，但有最大包长限制）
void usb_cdc_send(uint8_t* data, uint16_t len) {
    // USB CDC最大包长64字节（全速模式）
    while (len > 0) {
        uint16_t chunk = (len > 64) ? 64 : len;
        CDC_Transmit_FS(data, chunk);
        data += chunk;
        len -= chunk;
        // 等上一包发完（USB使用IN端点轮询）
        while (hUsbDeviceFS.dev_state != USBD_STATE_CONFIGURED);
    }
}

// ★ USB CDC vs UART比较：
// 相同点：都是串行通信、都能用printf重定向
// 不同点：USB有包概念(64字节/包)、自动流控、即插即用、速度可达1MB/s
// USB陷阱：PC端没打开串口软件 → 发送会阻塞！
//   解决：检查连接状态 (hUsbDeviceFS.dev_state == USBD_STATE_CONFIGURED)

📡 CAN FD：CAN的下一代

CAN FD (Flexible Data-rate) 是CAN 2.0的升级版，解决了传统CAN每帧只能传8字节数据的限制，同时支持更高的数据速率。

特性CAN 2.0CAN FD
每帧最大数据8字节64字节
最高速率1Mbps8Mbps（数据段）
CRC长度15位17/21位（更可靠）
兼容性-可混合组网（FD节点+2.0节点）
典型MCUSTM32F103/F407STM32G4/H7, ESP32-S3
适用场景传统汽车/工业ADAS/OTA/高带宽传感器
💡 CAN FD选择建议：如果是新设计且MCU支持CAN FD，优先选CAN FD。8字节限制在传统CAN上是常见的痛点（如发送GPS坐标需要分两帧）。不过CAN FD芯片略贵，且CAN FD和CAN 2.0混用时速率受限于最慢节点。

协议选型指南

面对一个新项目，该用哪个通信协议？没有银弹，但有以下经验法则：

场景推荐协议原因
调试输出、GPS模块、蓝牙串口模块UART最简单，PC端直接COM口读取
板上多个传感器/存储器（<10个）I2C两根线挂所有设备，省引脚

单总线(1-Wire)协议进阶省引脚

// 复位脉冲：主机拉低480μs → 释放 → 从机拉低60~240μs应答

本章总结

📋 知识点回顾：

UART：异步、全双工、2线(TX/RX)。波特率需双方一致，有~3%容错。TTL/RS-232/RS-485是不同物理层标准。

I2C：同步、半双工、2线(SDA/SCL)。开漏输出，必须上拉电阻(4.7K)。7位地址0x08~0x77。HAL库地址要左移1位。

SPI：同步、全双工、4线(MOSI/MISO/SCK/CS)。4种工作模式(CPOL/CPHA)。速度最快可达80Mbps。无标准协议层，各芯片命令格式不同。

CAN：差分、多主、有仲裁机制。标准帧11位ID，扩展帧29位ID。每帧8字节数据。需要终端电阻120Ω。广泛用于汽车和工业。

RS-485/Modbus：差分远距离、多节点(最多32/256)。Modbus是应用层协议，RTU格式最常用。CRC16校验。

调试神器：逻辑分析仪(¥20-50) + PulseView软件 = 协议问题终结者。波形不会说谎。

选型原则：近距离多设备=I2C，高速数据=SPI，调试=UART，工业/汽车=CAN或RS-485。
