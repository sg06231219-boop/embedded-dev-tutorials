第4章：串口通信 — 单片机与电脑对话

📖 小白完全指南 · 4/14 | 阅读时间约3分钟

串口(UART)是嵌入式开发最基础、最常用的通信方式。单片机的printf输出、调试信息、AT指令控制，全靠串口。可以说：不会串口，嵌入式寸步难行。

💡 类比：串口就像两个人用对讲机通话——波特率=说话速度（必须一致才能听懂），数据位=每个词的长度，停止位=说完了喘口气，校验位=确认对方听清了。速度不一样就全是乱码，就像一个人说中文、一个说英文。

串口参数

参数常见值说明
波特率(Baud Rate)9600 / 115200通信速度，双方必须一致
数据位(Data Bits)8每个字节的有效位数
停止位(Stop Bits)1帧结束标志
校验位(Parity)None简单错误检测，通常不用
流控(Flow Control)None硬件/软件流控，通常不用

日常最常用的配置：115200 8N1（115200波特率，8数据位，无校验，1停止位）

串口通信原理深入

UART（Universal Asynchronous Receiver/Transmitter）是一种异步串行通信协议。"异步"的意思是没有时钟线——收发双方靠约定的波特率来同步。这与I2C/SPI这类有时钟线的同步协议完全不同。

💡 类比：异步通信就像两个人在黑暗中用闪光灯交流——没有"开始"的视觉线索，全靠约定的节奏。如果一个人以每秒2次的速度闪，另一个人以每秒10次的速度看，那就像波特率不匹配（一方9600，一方115200），看到的全是乱码。

UART数据帧结构

一个完整的UART字节帧包含以下部分：

组成部分说明位数
起始位(Start Bit)拉低1位时间，通知接收方"数据来了"1 bit
数据位(Data Bits)实际数据，LSB在前（最低位先发）5-9 bit（常用8）
校验位(Parity Bit)奇偶校验，用于检测传输错误0-1 bit（通常不用）
停止位(Stop Bits)拉高，表示这一帧结束1-2 bit（常用1）
💡 关键理解：发送方和接收方的波特率误差必须 < 2%。如果一方是115200，另一方是9600，误差超过90%，收到的数据完全是乱码。这就是为什么串口通信的第一步永远是"对波特率"。

波特率与通信距离

波特率最大理论距离推荐距离适用场景
9600~15m≤10m长距离、抗干扰要求高
115200~3m≤1m日常调试、开发板连接
921600~50cm≤30cm高速数据传输
2000000+~20cmPCB板内高速需求，需优质线缆
⚠️ 实际经验：USB转TTL模块（CH340/CP2102）在115200下非常稳定，但921600以上可能掉数据——这和USB转串芯片质量、线缆质量、电脑USB口供电都有关。日常开发用115200最靠谱。
串口多字节数据帧设计进阶
串口是字节流，没有"包"的概念。如果要发送多字节数据（如温度值=25.3，占4字节float），需要自己设计帧格式来分割每一帧。以下是几种常用方案：
// 方案1：固定帧头+长度+数据+校验（最推荐）
// 帧格式：[0xAA 0x55] [长度N] [N字节数据] [CRC8校验]
// 示例：发送温度25.3°C
uint8_t frame[] = {
    0xAA, 0x55,  // 帧头（两个字节，降低误判概率）
    0x04,         // 数据长度=4字节（float）
    0x00, 0x00, 0xCA, 0x41,  // 25.3的IEEE 754表示
    0x00          // CRC8校验（此处简化）
};

// 方案2：固定帧头+固定长度（简单但浪费带宽）
// 帧格式：[0xAA] [固定8字节数据包]
// 优点：解析简单 缺点：不够灵活

// 方案3：字符协议（人类可读，最常用）
// 以换行符\\n为分隔，JSON格式
// 如：{"temp":25.3,"hum":62.1}\\n
// 方案4：Modbus-RTU（工业标准）
// 帧格式：[地址][功能码][数据][CRC16]
// 帧间间隔≥3.5字符时间作为帧分隔

Python串口高级用法

串口接收：三种模式对比实用技巧
Python pyserial提供了多种接收数据的方式，选择合适的方式对程序稳定性至关重要：
import serial
ser = serial.Serial('COM3', 115200, timeout=0.1)

# 方式1：readline() — 按行读取（最常用）
# 适用：数据以\\n结尾的场景
line = ser.readline().decode('utf-8').strip()
print(f"行数据: {line}")

# 方式2：read(size) — 读取指定字节数
# 适用：固定长度数据包
data = ser.read(8)  # 读8字节
if len(data) == 8:
    # 解析数据包
    header, value = data[0], data[1:]
    print(f"Header: {header:02X}, Value: {value.hex()}")

# 方式3：in_waiting — 非阻塞读取所有可用数据
# 适用：实时数据流
if ser.in_waiting > 0:
    data = ser.read(ser.in_waiting)
    print(f"收到 {len(data)} 字节: {data.hex()}")

# 方式4：iter() — 迭代器模式（适合持续读取）
# 适用：持续监听串口数据
for line in ser:

串口接收线程（生产级代码）进阶
在实际项目中，串口接收通常放在独立线程中，避免阻塞主程序。以下是带帧解析和自动重连的完整示例：
import serial
import threading
import time
import queue

class SerialReader:
    """串口读取器：线程化接收+帧解析+数据队列"""
    def __init__(self, port, baud=115200):
        self.ser = serial.Serial(port, baud, timeout=0.5)
        self.data_queue = queue.Queue()
        self.running = True
        self.buffer = b''
        # 启动接收线程
        self.thread = threading.Thread(target=self._read_loop, daemon=True)
        self.thread.start()

    def _read_loop(self):
        """接收线程：持续读取串口数据"""
        while self.running:
            try:
                if self.ser.in_waiting:
                    data = self.ser.read(self.ser.in_waiting)
                    self.buffer += data
                    # 按换行符分割帧
                    while b'\\n' in self.buffer:
                        line, self.buffer = self.buffer.split(b'\\n', 1)
                        self.data_queue.put(line)
            except serial.SerialException as e:
                print(f"串口异常: {e}")
                time.sleep(1)  # 等待重连
            except Exception as e:
                print(f"未知错误: {e}")
                break

    def get_data(self, timeout=1.0):
        """从队列获取一帧数据（非阻塞）"""
        try:
            return self.data_queue.get(timeout=timeout)
        except queue.Empty:
            return None

    def send(self, data):
        """发送数据"""
        if isinstance(data, str):
            data = data.encode('utf-8')
        self.ser.write(data)

    def close(self):
        self.running = False
        self.ser.close()

串口DMA收发（STM32进阶）进阶
轮询方式每收发一个字节都要占用CPU，大流量时CPU全忙在串口上。DMA（Direct Memory Access）可以在不占用CPU的情况下完成数据搬运，CPU只需在传输完成时处理一次中断：
// STM32 HAL库 DMA串口发送（不阻塞CPU）
// 前提：CubeMX已启用USART1_TX的DMA通道
// 特点：调用后立即返回，CPU可做其他事

uint8_t tx_buf[256];

// 发送数据（DMA自动完成，函数立即返回）
sprintf((char*)tx_buf, "Temp: %.1f\\r\\n", temperature);
HAL_UART_Transmit_DMA(&huart1, tx_buf, strlen((char*)tx_buf));
// CPU不需要等待！可以立即执行其他代码

// DMA发送完成回调
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        // 发送完成，可以准备下一次发送
        tx_ready = 1;
    }
}

// DMA循环接收（空闲中断：检测帧间隔）
// 结合DMA+UART_IDLE中断，实现不定长数据接收
uint8_t rx_dma_buf[256];

// 初始化：启动DMA接收（在main初始化中调用一次）
HAL_UART_Receive_DMA(&huart1, rx_dma_buf, sizeof(rx_dma_buf));

// UART空闲中断回调（一帧接收完成）
void HAL_UARTEx_RxEventCallback(UART_HandleTypeDef *huart, uint16_t Size) {
    if (huart->Instance == USART1) {
        // Size = 本帧实际接收的字节数
        printf("收到 %d 字节: ", Size);
        for (int i = 0; i < Size; i++) {
            printf("%02X ", rx_dma_buf[i]);
        }
        printf("\\r\\n");
        // 重新启动DMA接收（重要！否则下次收不到）
        HAL_UART_Receive_DMA(&huart1, rx_dma_buf, sizeof(rx_dma_buf));
    }
}
⚠️ DMA注意事项：DMA缓冲区必须是全局变量或静态变量（不能是栈上的局部变量）。DMA在后台异步工作，如果缓冲区被释放了，DMA会写入野指针导致HardFault。

串口协议实战：Modbus-RTU从入门到通信

Modbus-RTU协议详解进阶工业通信
Modbus是最广泛使用的工业串口协议之一，几乎所有PLC、变频器、传感器、电表都支持Modbus-RTU。它虽然协议简单，但有严格的帧格式和时序要求。
// === Modbus-RTU 帧格式 ===
// [地址1B] [功能码1B] [数据N B] [CRC16 2B]
//  地址: 0-247（0=广播, 247=保留）
//  功能码: 03=读保持寄存器, 06=写单寄存器, 16=写多寄存器
//  CRC: 低字节在前（Little-Endian）
//
// 帧间间隔: ≥3.5个字符时间（115200下≈3.5×87μs≈305μs）
// 帧内间隔: ≤1.5个字符时间
//   → 接收方通过间隔来分割帧！

// === 示例：读温度传感器（从地址1的设备读2个寄存器）===
// 发送: 01 03 00 00 00 02 C4 0B
//  01       = 设备地址
//  03       = 功能码（读保持寄存器）
//  00 00    = 起始寄存器地址（从0号寄存器开始）
//  00 02    = 读取数量（2个寄存器=4字节）
//  C4 0B    = CRC16（低字节在前）
//
// 回复: 01 03 04 00 C0 00 64 8B 8A
//  01       = 设备地址（原样返回）
//  03       = 功能码（原样返回）
//  04       = 数据字节数（4字节）
//  00 C0    = 寄存器0的值（温度×10→192→19.2°C）
//  00 64    = 寄存器1的值（湿度×10→100→10.0%RH）
//  8B 8A    = CRC16

// === CRC16-Modbus 计算函数 ===
uint16_t Modbus_CRC16(uint8_t *data, uint16_t len) {
    uint16_t crc = 0xFFFF;
    for (uint16_t i = 0; i < len; i++) {
        crc ^= data[i];
        for (uint8_t j = 0; j < 8; j++) {
            if (crc & 0x0001) {
                crc = (crc >> 1) ^ 0xA001;
            } else {
                crc = crc >> 1;
            }
        }
    }
    return crc;  // 注意：已经是低字节在前
}

// === Modbus从机接收（基于帧间隔检测）===
#define MODBUS_FRAME_TIMEOUT_MS  5  // 帧间隔超时（>3.5字符时间）

// 在定时器中检测帧间隔
void MODBUS_RxTimeout_Callback(void) {
    if (modbus_rx_len >= 4) {  // 至少地址+功能码+CRC
        // 校验CRC
        uint16_t calc_crc = Modbus_CRC16(modbus_rx_buf, modbus_rx_len - 2);
        uint16_t recv_crc = modbus_rx_buf[modbus_rx_len - 1] << 8 |
                            modbus_rx_buf[modbus_rx_len - 2];
        
        if (calc_crc == recv_crc) {
            // CRC正确，处理帧
            uint8_t addr = modbus_rx_buf[0];
            uint8_t func = modbus_rx_buf[1];
            
            if (addr == MY_SLAVE_ADDR || addr == 0) {
                MODBUS_ProcessFrame(func, modbus_rx_buf + 2,
                                    modbus_rx_len - 4);
            }
        }
    }
    modbus_rx_len = 0;  // 重置接收计数
}

// === 常见Modbus坑 ===
// 1. CRC字节顺序：计算结果是16位值，但发送时要低字节在前
//    如CRC=0x0BC4，发送顺序是 C4 0B
//    很多人搞反顺序导致校验失败
//
// 2. 地址偏移：Modbus寄存器地址从1开始，但协议中写0
//    读"40001"寄存器 → 协议中写"0000"
//
// 3. 帧间隔必须足够：STM32在115200下，3.5字符≈305μs
//    如果定时器精度不够或中断延迟过大，帧边界识别失败
//
// 4. RS-485方向控制：Modbus通常走RS-485半双工
//    发送前需先切换485方向为TX，发送完成后等到TC标志再切回RX
//    如果不等待TC（传输完成）就切回RX，最后一个字节会丢失

串口故障诊断：从乱码到通信的完整修复流程

🔍 故障诊断流程图

第1层：物理层检查
• TX-RX交叉连接正确？（TX→RX, RX→TX, GND→GND，不要TX→TX）
• GND线接了吗？（没有GND参考，电平毫无意义）
• 用万用表测：模块TX引脚应稳定在高电平（空闲态=高），RX类似
• 示波器看：TX线空闲时应为高电平，发数据时有脉冲

第2层：波特率/参数检查
• 双方波特率、数据位、停止位、校验位完全一致？
• 用示波器测量1个bit的宽度推算实际波特率：
  bit宽度(秒) = 1 / 波特率
  115200 → 1bit ≈ 8.68μs
  ​如果测到bit宽度=104μs → 实际波特率≈9600
• 检查系统时钟：HSE是否起振？PLL配置是否正确？

第3层：数据验证
• 用示波器/逻辑分析仪抓取波形，逐bit解码
• 检查起始位（1bit低电平）、停止位（1bit高电平）
• 如果发现停止位是低电平 → 波特率偏高（接收方采样比发送方快）
• 如果发现起始位后数据太快 → 波特率偏低

第4层：软件验证
• STM32回环测试：TX和RX短接，发送"TEST"，接收应收到"TEST"
• Python回环测试：USB转TTL的TX和RX短接，发送数据确认自收自发正常
• 交叉测试：用逻辑分析仪同时接TX和RX，确认双方都在发送

串口通信在嵌入式中的角色图谱

串口在项目中的典型用途全景图
理解串口在完整项目中的位置有助于做架构决策。以下是串口的几种经典使用场景：
// === 场景1：调试输出（开发阶段最常用）===
// [STM32] → UART → [CH340] → USB → [PC/串口助手]
// 用途：输出日志、变量值、执行路径
// 速度：115200足够
// 替代：SWO/ITM、RTT、USB CDC（速度更快）

// === 场景2：AT指令控制（WiFi/蓝牙/GPRS模块）===
// [STM32] ← UART → [ESP8266/EC20/SIM800]
// 示例：
//   STM32发送 → "AT+CWMODE=1\\r\\n"
//   模块回复 ← "\\r\\nOK\\r\\n"
// 重点：注意模块的应答延迟（可能几百ms），别用死等

// === 场景3：传感器直读（串口传感器）===
// [STM32] ← UART → [传感器: PM2.5/GPS/激光测距]
// 许多传感器自带MCU，通过串口输出格式化数据
// 如PMS7003 PM2.5传感器：主动发送32字节数据帧
//   → 用DMA+空闲中断接收最合适

// === 场景4：工业总线（RS-485 + Modbus）===
// [主机] ← RS-485总线 → [从机1] [从机2] ... [从机N]
// 特点：多节点、长距离、半双工

MCU端串口接收：中断+环形缓冲区

环形缓冲区（Ring Buffer）详解必学实用技巧
串口接收中断中不能做耗时操作（如printf、复杂计算），否则会丢数据。最佳实践是：中断中只负责存数据到环形缓冲区，主循环中取数据处理。环形缓冲区是嵌入式开发的最核心数据结构之一。
// === 环形缓冲区实现（生产级代码）===
// ring_buffer.h
#ifndef __RING_BUFFER_H
#define __RING_BUFFER_H

#include <stdint.h>
#include <stdbool.h>

typedef struct {
    uint8_t *buffer;       // 缓冲区指针
    uint16_t size;         // 缓冲区总大小（必须是2的幂）
    volatile uint16_t head; // 写指针（ISR中修改）
    volatile uint16_t tail; // 读指针（主循环中修改）
} RingBuffer_t;

// 初始化环形缓冲区
void RingBuffer_Init(RingBuffer_t *rb, uint8_t *buf, uint16_t size);

// 写入一个字节（ISR中调用）
bool RingBuffer_Put(RingBuffer_t *rb, uint8_t data);

// 读取一个字节（主循环中调用）
bool RingBuffer_Get(RingBuffer_t *rb, uint8_t *data);

// 获取当前缓冲区中的数据量
uint16_t RingBuffer_Available(RingBuffer_t *rb);

// 清空缓冲区
void RingBuffer_Flush(RingBuffer_t *rb);

#endif

// ring_buffer.c
void RingBuffer_Init(RingBuffer_t *rb, uint8_t *buf, uint16_t size) {
    rb->buffer = buf;
    rb->size = size;
    rb->head = 0;
    rb->tail = 0;
}

bool RingBuffer_Put(RingBuffer_t *rb, uint8_t data) {
    uint16_t next = (rb->head + 1) & (rb->size - 1);  // 2的幂取模优化
    if (next == rb->tail) {
        return false;  // 缓冲区满！丢数据（生产环境应记录溢出次数）
    }
    rb->buffer[rb->head] = data;
    rb->head = next;
    return true;
}

bool RingBuffer_Get(RingBuffer_t *rb, uint8_t *data) {
    if (rb->head == rb->tail) {
        return false;  // 缓冲区空
    }
    *data = rb->buffer[rb->tail];
    rb->tail = (rb->tail + 1) & (rb->size - 1);
    return true;
}

uint16_t RingBuffer_Available(RingBuffer_t *rb) {
    return (rb->head - rb->tail) & (rb->size - 1);
}

// === 在串口中断中使用环形缓冲区 ===
static uint8_t uart1_rx_buf[256];
static RingBuffer_t uart1_rb;

// 初始化（main函数中调用）
RingBuffer_Init(&uart1_rb, uart1_rx_buf, 256);

// 串口接收中断回调（每收到一个字节触发）
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        static uint8_t rx_byte;
        
        // 1. 读取刚收到的字节
        rx_byte = huart->Instance->DR & 0xFF;  // 直接读DR寄存器
        // 或: HAL_UART_Receive(huart, &rx_byte, 1, 0);
        
        // 2. 存入环形缓冲区
        RingBuffer_Put(&uart1_rb, rx_byte);
        
        // 3. 重新启动接收中断（每次只能收1字节）
        HAL_UART_Receive_IT(huart, &rx_byte, 1);
    }
}

// 主循环中处理接收数据
void ProcessSerialData(void) {
    uint8_t byte;
    while (RingBuffer_Get(&uart1_rb, &byte)) {
        // 按行处理（检测换行符）
        static uint8_t line_buf[128];
        static uint8_t line_idx = 0;
        
        if (byte == '\\n' || byte == '\\r') {
            if (line_idx > 0) {
                line_buf[line_idx] = '\\0';
                // 处理完整行数据
                ParseCommand((char*)line_buf);
                line_idx = 0;
            }
        } else if (line_idx < sizeof(line_buf) - 1) {
            line_buf[line_idx++] = byte;
        }
    }
}
💡 为什么buffer大小必须是2的幂？用& (size - 1)替代% size取模运算，只需1个CPU周期（按位与），而除法需要2-12个周期。在ISR中每微秒都很宝贵。
⚠️ 环形缓冲区注意事项：head只被ISR修改，tail只被主循环修改——单生产者单消费者模型天然无锁，不需要关中断。如果主循环也要写（如发送缓冲区），则需要临界区保护。

USART同步模式：像SPI一样通信

USART同步模式进阶
很多人不知道，STM32的USART除了异步模式（最常用），还支持同步模式。在同步模式下，USART会输出一路时钟信号（CK），与SPI非常相似。同步模式的主要优势：不需要精确的波特率匹配（时钟由主机提供），速度可以更高。
// === USART同步模式 vs 异步模式 ===
// 异步模式（标准UART）：
//   TX: 只发数据，没有时钟
//   RX: 通过过采样(16倍)恢复时钟
//   缺点：双方波特率误差必须<2%
//
// 同步模式：
//   TX: 发送数据 + 时钟信号（CK引脚）
//   RX: 在时钟沿采样数据（像SPI一样准确）
//   优点：
//     - 不受波特率误差影响
//     - 速度可达MCU频率/2（STM32F103最高4.5Mbps）
//     - 支持半双工/全双工
//   缺点：
//     - 需要额外时钟线（CK）
//     - 两者都必须是同步模式

// === CubeMX配置同步模式 ===
// 1. USART1 Mode: Synchronous
// 2. 引脚自动分配: TX, RX, CK
// 3. 时钟参数：
//    - 时钟极性(CPOL): Low(默认)
//    - 时钟相位(CPHA): 1st Edge(第1个沿采样)
//    - 最后位脉冲(LastBit): Disable

// === 代码中使用 ===
// 发送（同步模式，函数相同）
HAL_UART_Transmit(&huart1, data, len, 1000);

// 接收也相同
HAL_UART_Receive(&huart1, data, len, 1000);

// 实际应用场景：
// - 板间高速通信（两块STM32之间）
// - 驱动SPI-like外设（某些SPI设备可以接USART同步模式）
// - 替代SPI（如果SPI接口不够用）
⚠️ 实际使用建议：同步模式在实际项目中使用较少，因为SPI接口通常够用，且更加标准化。但了解这个特性有助于在资源紧张时"一接口多用"。另外，STM32F1的USART同步模式最高4.5Mbps，而SPI可达18Mbps——SPI更快。

硬件流控（RTS/CTS）详解

硬件流控 RTS/CTS进阶
当一方发送速度超过另一方处理速度时，会丢数据。软件流控（XON/XOFF）通过发送特殊字符来暂停/恢复传输，但会与数据内容冲突。硬件流控（RTS/CTS）使用专用引脚信号，不占用数据带宽。
// === 硬件流控工作原理 ===
// RTS (Request To Send) = 发送方的"准备好"信号（输出）
// CTS (Clear To Send)   = 接收方的"可以发"信号（输入）
//
// 场景：MCU_A 向 MCU_B 高速发送数据
//
// MCU_A(TX/RTS)           MCU_B(RX/CTS)
//    |                        |
//    |-- RTS=低(请求发送) --> |  (MCU_B检测到RTS低)
//    |                        |-- 检查缓冲区
//    |<-- CTS=低(允许发送) -- |  (缓冲区有空间，拉低CTS)
//    |                        |
//    |--数据数据数据数据...-> |  (MCU_A开始发送)
//    |                        |-- 缓冲区快满了！
//    |<-- CTS=高(暂停!) ----- |  (拉高CTS，停止发送)
//    |   MCU_A暂停发送        |-- 处理缓冲区的数据
//    |<-- CTS=低(继续发送) -- |  (处理完了，拉低CTS)
//    |-- 数据数据数据...->    |  (继续发送)

// === CubeMX配置硬件流控 ===
// USART1 → Hardware Flow Control (RS232):
//   - None (默认，无流控)
//   - CTS Only
//   - RTS Only
//   - CTS/RTS (全双工流控)
//
// 引脚自动分配：
//   USART1_RTS → PA12
//   USART1_CTS → PA11

// === 代码中使用（HAL库自动处理）===
// 无需任何额外代码！HAL库在底层自动检测CTS信号：
// - CTS=高时，HAL_UART_Transmit会自动等待（直到超时）
// - 不需要在用户代码中判断CTS状态
HAL_UART_Transmit(&huart1, large_buffer, 4096, 5000);
// 如果接收方来不及处理，硬件自动暂停发送

// === 实际应用场景 ===
// • 使用ESP8266/ESP32 AT指令时（模块内部缓冲区小，需要流控）
// • 高速日志输出（发太快串口助手处理不过来）
// • 无线模块通信（空中速率 << 串口速率）
// • GPRS/4G模块（AT指令+数据混合传输）
⚠️ 常见坑：很多USB转TTL模块（CH340/CP2102）不支持硬件流控引脚。如果你配置了CTS/RTS，但USB转TTL上没有对应的CTS/RTS信号，CTS会一直"悬空"（电平不确定），导致STM32永远无法发送。解决方案：① 用支持硬件流控的FT232模块 ② 关闭流控 ③ 外接上拉/下拉电阻固定CTS电平。

波特率误差计算：为什么有时候115200不行？

波特率误差计算踩坑经验
UART对波特率误差非常敏感。当系统时钟源不是标准值的整数倍时，实际波特率会偏离标称值。误差>2%就开始出现乱码，>5%基本无法通信。
// === 波特率误差计算 ===
// STM32 USART波特率 = f_CK / (16 * USARTDIV)
// 其中f_CK是USART的时钟（来自APB1/APB2）
// USARTDIV是分频系数

// 示例1: STM32F103, USART1 (APB2=72MHz), 目标115200
//   72,000,000 / (16 * 115200) = 39.0625
//   → 整数部分INTDIV=39, 小数部分FRAC=0.0625*16=1
//   → 实际波特率 = 72,000,000 / (16 * 39.0625) = 115200.0
//   → 误差 = 0%（完美！因为72MHz可被整除）

// 示例2: STM32F103用内部HSI (8MHz, PLL=64MHz), USART1, 目标115200
//   64,000,000 / (16 * 115200) = 34.7222
//   → 整数部分=34, 小数部分=11
//   → 实际波特率 = 64,000,000 / (16 * 34.6875) = 115235.294
//   → 误差 = (115235 - 115200) / 115200 ≈ 0.03% ← OK

// 示例3: 如果用了8MHz晶振但PLL配成64MHz, USART1, 目标921600
//   64,000,000 / (16 * 921600) = 4.340
//   → 取整≈4.34375
//   → 实际 = 64,000,000 / (16 * 4.34375) ≈ 921176
//   → 误差 = 0.046%（在高速下仍可接受）

// === 常见导致波特率偏差的原因 ===
// 1. 晶振/时钟配置错误（最常见！）
//    - 实际晶振是8MHz，但CubeMX配成了25MHz
//    - 结果：所有波特率偏离 8/25 = 32%，完全无法通信
//
// 2. HSE起振失败，自动fallback到HSI
//    - CubeMX配的是72MHz(HSE+PLL)
//    - 实际HSE没起振，系统用HSI的8MHz
//    - 结果：波特率只有目标值的 8/72 ≈ 11% ← 全是乱码
//    - 检测方法：在main函数开头读RCC寄存器确认时钟源
//    if (__HAL_RCC_GET_SYSCLK_SOURCE() != RCC_SYSCLKSOURCE_PLLCLK) {
//        // 时钟源不对！HSE可能没起振
//    }
//
// 3. PLL配置错误
//    - CubeMX中PLLM/PLLN/PLLP参数与实际代码不一致

// === 波特率容差表（参考值）===
// 波特率 | 推荐时钟频率(MHz)      | 说明
// 9600   | 任何常见频率            | 几乎无误差
// 115200 | 7.3728, 14.7456, 72     | 能被整数除尽的频率
// 230400 | 7.3728, 14.7456         | 高速下需精确匹配
// 460800 | 7.3728, 14.7456         | 同上
// 921600 | 7.3728, 14.7456         | 常用晶振中只有这两个
//
// ★ 7.3728MHz晶振是串口专用频率！几乎所有标准波特率都能整除

USB CDC虚拟串口：告别USB转TTL

USB CDC ACM（虚拟串口）实用进阶
STM32F103C8T6自带USB外设，可以模拟成虚拟串口（CDC ACM）。插上USB线后，电脑直接识别为COM口，不需要CH340/USB转TTL模块。而且速度远超物理串口（USB全速=12Mbps）。
// === STM32 USB CDC实现（CubeMX配置）===
// 步骤1: CubeMX启用USB
//   Connectivity → USB → Device (FS)
//
// 步骤2: 选择USB设备类
//   Middleware → USB_DEVICE → Class: Communication Device Class (CDC)
//
// 步骤3: 配置时钟
//   USB需要精确的48MHz时钟
//   → HSE=8MHz → PLL → 72MHz(SYSCLK) + 48MHz(USB)
//   → 或直接用内部HSI48（F0/L0系列有）
//
// 步骤4: 生成代码后在usbd_cdc_if.c中实现数据收发

#include "usbd_cdc_if.h"

// === USB CDC 发送（和串口几乎一样）===
uint8_t tx_data[] = "Hello from USB CDC!\\r\\n";
uint8_t result = CDC_Transmit_FS(tx_data, sizeof(tx_data) - 1);
// result == USBD_OK: 发送成功
// result == USBD_BUSY: USB忙，需等待重试

// === USB CDC 接收（通过回调）===
// usbd_cdc_if.c 中已有接收缓冲区 UserRxBufferFS[]

static int8_t CDC_Receive_FS(uint8_t* Buf, uint32_t *Len) {
    // Buf: 收到的数据
    // *Len: 数据长度
    
    // 处理接收到的数据（例如复制到环形缓冲区）
    for (uint32_t i = 0; i < *Len; i++) {
        RingBuffer_Put(&usb_rb, Buf[i]);
    }
    
    // 重新启动接收（重要！）
    USBD_CDC_SetRxBuffer(&hUsbDeviceFS, &Buf[0]);
    USBD_CDC_ReceivePacket(&hUsbDeviceFS);
    
    return USBD_OK;
}

// === USB CDC速度对比 ===
// 物理串口 @115200:  ~11.5 KB/s
// 物理串口 @921600:  ~92   KB/s
// USB CDC (FS):       ~800  KB/s (实测，接近USB Full Speed极限)
// USB CDC (HS):       ~8000 KB/s (F4/H7系列)
//
// → USB CDC比115200串口快 ~70倍！

// === 电脑端看到的效果 ===
// 设备管理器 → 端口 → STMicroelectronics Virtual COM Port (COM7)
// 和普通串口完全一样使用！pyserial/matplotlib直接连COM7即可

• USB CDC需要精确48MHz时钟 → 必须使用外部晶振(HSE)或HSI48

  USB_DM=PA11, USB_DP=PA12 ← 与USART1_CTS_RTS冲突！

1. 波特率不匹配（最常见！）——检查单片机代码和串口工具是否一致
2. 时钟配置错误——STM32用HSE(8MHz)还是HSI(8MHz)？HSE没起振会导致波特率偏

✅ 实用工具：
• pyserial：Python串口库，miniterm 是最常用的串口终端
• PuTTY：Windows通用串口工具，注意勾选Local echo才能看到输入
• USB转TTL芯片：CH340(最便宜)/CP2102(稳定)/FT232(最贵最稳)

✅ 编程要点：
• Python：serial.Serial() 打开串口 → readline() 按行读 → write() 发送
• 生产级代码：接收放在独立线程中，用队列传递数据
• 多字节数据需要设计帧格式（帧头+长度+数据+校验）
• STM32：用中断+环形缓冲区实现高效接收（ISR只存不处理）
• STM32进阶：用DMA+空闲中断实现零CPU开销的不定长接收

✅ 高级话题（新增）：
• 环形缓冲区：单生产者单消费者模型，无锁高效，比DMA方案更灵活
• USART同步模式：输出时钟线（像SPI），不受波特率误差影响
• 硬件流控（RTS/CTS）：专用引脚信号控流，不占用数据带宽
• USB CDC虚拟串口：用STM32 USB模拟COM口，速度~800KB/s
• 波特率误差：时钟源配置错误是乱码的最常见根源
• Modbus-RTU：工业标准协议，帧间隔检测+CRC16校验

✅ 工业应用：
• RS-485适用于长距离（1200m）/多节点（256个）工业场景
• Modbus-RTU协议底层通常走RS-485
• TTL与RS-232/485电平不兼容，需要电平转换芯片

⚠️ 常见陷阱（扩展）：
• 忘记接GND线（TX+RX+GND三根线缺一不可）
• 3.3V MCU直连5V设备，烧IO口
• 波特率不匹配（最最常见的问题）
• 时钟源配置错误导致波特率偏差（HSE不起振是第一大坑）
• 发送缓冲区溢出（发送太快，来不及发）
• 串口中断中做耗时操作（printf/复杂计算）→丢数据
• USB CDC与USART1引脚冲突（PA11/PA12同时被占用）
