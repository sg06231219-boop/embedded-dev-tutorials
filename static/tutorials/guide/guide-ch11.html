第11章：电源管理与低功耗设计

📖 小白完全指南 · 11/14 | 阅读时间约14分钟

嵌入式设备的"续航力"决定了产品体验。从纽扣电池跑一年的传感器节点，到电池供电的手持设备，电源管理是电子工程师的必修课。

💡 类比：电源管理就像家庭理财——收入(电池能量)有限，支出(系统功耗)要精打细算。LDO=稳定但浪费的固定月薪，DC-DC=高效但复杂的投资理财，低功耗模式=省钱过日子。

🔋 电池供电系统设计

嵌入式设备大部分是电池供电，电池管理做不好，产品体验就废了一半。

锂电池放电特性重要
Li-Po放电曲线（关键电压点）：
• 4.20V = 100% 满电
• 4.00V = 80%（刚用不久）
• 3.85V = 50%（中间段最平）
• 3.70V = 20%（电量告急！请充电）
• 3.50V = 5%（接近截止）
• 3.00V = 0%（必须停止放电，低于此电压永久损坏）

注：锂电池在3.7-4.0V之间放电曲线较平，电压法测电量准确度有限。精确测量需库仑计。

锂电池充电管理

TP4056模块（最常用单节锂电充电芯片）必备
TP4056核心参数：
• 输入：5V USB/Micro-USB/Type-C
• 充电电流：可编程（PROG引脚电阻设置，默认1.2K=1A）
• 充满电压：4.2V ±1%
• 充电阶段：涓流(＜2.9V)→恒流(CC)→恒压(CV)→截止(C/10即100mA)
• 状态输出：CHRG(STDBY)引脚——充电时低电平，充满高电平
• 温度保护：TEMP引脚接NTC，过热自动降电流

PROG电阻与充电电流对照：
• 1.2KΩ → 1000mA（默认）   2KΩ → 580mA
• 5KΩ → 240mA（USB100标准）   10KΩ → 120mA（小电池）
锂电池保护电路（DW01+8205A）安全必备
任何锂电池电路必须有保护板！裸电池充电/过放=起火爆炸风险。

DW01保护IC + 8205A双MOSFET 经典组合：
• 过充保护：电压＞4.25V → 断开充电MOS（COUT=低）
• 过放保护：电压＜2.4V → 断开放电MOS（DOUT=低）
• 过流保护：放电电流＞3A → 断开（短路/堵转）
• 短路保护：电压瞬间跌落 → 立刻断开
• 延迟时间：过充~1s、过放~100ms、过流~10ms

典型接线（3个元件）：DW01(6脚SOT23-6) + 8205A(8脚TSSOP8) + 100Ω电阻

电池电量测量

// 电池电量测量：电压法（最简单，精度~10%）
// Li-Po: 4.2V=100%  3.85V=50%  3.0V=0%（截止！）

float measure_battery_voltage(void) {
    // 假设电池经1:2分压后接ADC（100K+100K分压）
    HAL_ADC_Start(&hadc1);
    HAL_ADC_PollForConversion(&hadc1, 10);
    uint16_t adc = HAL_ADC_GetValue(&hadc1);
    // 3.3V参考，12位ADC
    float v_adc = adc * 3.3f / 4096.0f;
    float v_bat = v_adc * 2.0f;  // 分压1:2，实际=(R1+R2)/R2
    return v_bat;
}

uint8_t battery_percent(float voltage) {
    // Li-Po放电曲线近似（分段线性化）
    if (voltage >= 4.15f) return 100;
    if (voltage >= 4.06f) return 90;
    if (voltage >= 3.98f) return 80;
    if (voltage >= 3.92f) return 70;
    if (voltage >= 3.87f) return 60;
    if (voltage >= 3.82f) return 50;
    if (voltage >= 3.77f) return 40;
    if (voltage >= 3.73f) return 30;
    if (voltage >= 3.68f) return 20;
    if (voltage >= 3.60f) return 10;
    return 5;  // 快没电了！
}

// 更精确的方法：库仑计（电量积分）
// 芯片选择：MAX17048(1-cell I2C)、BQ27427(TI)、LTC2942
// 库仑计通过测量电流的积分来计算实际消耗的电量
// MAX17048使用示例（I2C读取，无需校准！）
float max17048_read_soc(void) {
    uint8_t buf[2];
    HAL_I2C_Mem_Read(&hi2c1, 0x36<<1, 0x04, 1, buf, 2, 100);
    return (float)((buf[0] << 8) | buf[1]) / 256.0f;
    // 返回值0-100.0%，自动学习电池容量
}

🔌 DC-DC电源拓扑

💡 类比：LDO像一个可变电阻降压——电流不变，差值功率全变成热量。DC-DC像一个变压器+开关——高速通断存储能量，效率高但纹波大。

LDO（低压差线性稳压器）

LDO的核心优势是无开关噪声、外围简单。劣势是效率低，大压差场景发热严重。

// LDO选型关键参数（按重要度排序）
// 1. 压差(Dropout Voltage): Vin-Vout的最小值，越小越好
// 2. 最大输出电流: 需>负载峰值电流
// 3. 静态电流(Iq): 芯片自身耗电——电池设备选Iq<10μA！
// 4. PSRR(电源纹波抑制比): 抑制输入噪声能力
// 5. 噪声: 输出端噪声幅度——模拟电路关注

// 常用LDO芯片速查
// AMS1117-3.3: 1A, 1.1V压差@1A, 5-10mA Iq (⚠不推荐电池设备)
// ME6206A33: 500mA, 100mV压差@200mA, 7μA Iq (适合电池)
// HT7333-1:   250mA, 90mV压差@40mA,  5μA Iq (经典低功耗)
// XC6206P332: 200mA, 250mV压差,      1μA Iq (超低功耗！)
// TPS7A03:    200mA, 足压差,         200nA Iq (TI超低功耗)
// TPS799:     200mA, 100mV压差, 40μV噪声 (超低噪声-ADC用)

// LDO功耗计算（发热评估）
// P_loss = (Vin - Vout) × I_load + Vin × Iq
// AMS1117@5V→3.3V,100mA: (5-3.3)×0.1 = 0.17W ✅
// AMS1117@12V→3.3V,100mA: (12-3.3)×0.1 = 0.87W 🔥烫手！
// AMS1117@12V→3.3V,500mA: (12-3.3)×0.5 = 4.35W 💥烧毁!
⚠ LDO散热估算：LDO芯片热阻通常50-200°C/W。0.87W × 100°C/W = 87°C温升！加上室温25°C = 112°C结温，超过大多数芯片的125°C上限。大压差务必用DC-DC。

Buck（降压型DC-DC）

// Buck转换器：高效降压，效率85-95%
// 核心原理：开关管高速通断 → 电感储能/释放 → 电容平滑

// 常用Buck芯片与模块
// MP1584模块: 3A, 4.5-28V→0.8-25V, 1.5MHz (¥3, 最常用)
// MP2307模块: 3A, 4.75-23V→1.0-20V, 340KHz (¥2)
// TPS5430:    3A, 5.5-36V→1.22-31V, 500KHz (TI工业级)
// MT3608:     升压, 2-24V→最高28V, 1.2MHz (¥2, 升压万用)
// XL6009:     升降压SEPIC, 5-32V→可调 (万用电源方案)

// Buck关键元件选型
// 1. 电感 L = (Vin-Vout)×D/(ΔI×fsw)
//    D = Vout/Vin (占空比), ΔI ≈ 0.3×Iload (纹波电流)
//    例子: 12V→5V@1A, fsw=1.5MHz
//    D=5/12=0.417, L=(12-5)×0.417/(0.3×1×1.5M) = 6.49μH → 选6.8μH
//    饱和电流必须 > Iload+ΔI/2 = 1+0.15 = 1.15A → 选2A
//
// 2. 输出电容：决定输出电压纹波
//    ΔVout ≈ ΔI/(8×fsw×Cout)
//    例子: Cout=22μF, ΔV=0.3/(8×1.5M×22μ)=1.1mV
//
// 3. 输入电容：应对输入电流脉冲（MLCC 10μF+0.1μF）
// 4. 续流二极管：肖特基SS34/SS54/B340A（低VF值=高效率）
⚠ PCB布局黄金法则：DC-DC的输入电容必须尽可能靠近IC引脚(<5mm)！远放=大电流环路=EMI地狱=不稳定。开关节点(SW/LX)面积越小越好——它是最大的辐射天线。反馈线(FB)远离电感和开关节点走线。地平面完整不断裂。

LDO vs DC-DC 选择决策树

// 选择流程（按此顺序判断）
// 1. 压差 < 1V ? → LDO（简单无噪声）
// 2. 电流 < 100mA ? → LDO（发热可接受）
// 3. 对噪声敏感（ADC/音频）? → LDO + π型滤波
// 4. 电池供电+超低功耗 ? → LDO（Iq小）
// 5. 以上都不满足 → DC-DC Buck
//
// 常见的"LDO+DC-DC"组合：DC-DC做一级粗降压(12V→5V)
// + LDO做二级精降压(5V→3.3V)，兼具效率和噪声

⚡ STM32低功耗模式详解

电池供电设备的核心竞争力就是功耗。STM32提供多层睡眠模式，选对模式能省90%以上的电。

三种睡眠模式对比

模式唤醒方式电流(F103)唤醒时间唤醒后RAM保持
Sleep任意中断/事件~5mA即时(0)继续执行下一条✅全保持
StopEXTI线/RTC唤醒~20μA~10μs从WFI处继续✅全保持
StandbyWKUP引脚/RTC/NRST~2μA~50μs相当于复位❌丢失
// Sleep模式：最简单，CPU停下来但外设继续运行
// 适合：等待DMA完成、等待UART接收
void enter_sleep_mode(void) {
    HAL_SuspendTick();  // 暂停SysTick（可选）
    HAL_PWR_EnterSLEEPMode(PWR_MAINREGULATOR_ON, PWR_SLEEPENTRY_WFI);
    HAL_ResumeTick();   // 恢复SysTick
}
// 或直接：
__WFI();  // Wait For Interrupt，任意中断唤醒后继续
// 注意：SysTick中断也会唤醒Sleep！

// Sleep-on-Exit模式：中断退出后自动再睡
SCB->SCR |= SCB_SCR_SLEEPONEXIT_Msk;
// 典型：中断处理完数据 → 自动休眠 → 等待下一个中断
// Stop模式：最常用低功耗，大部分时钟停了
void enter_stop_mode(void) {
    // 1. 关闭不必要的外设时钟（省电大户）
    __HAL_RCC_GPIOB_CLK_DISABLE();
    __HAL_RCC_GPIOC_CLK_DISABLE();
    __HAL_RCC_USART1_CLK_DISABLE();  // 串口不需要了
    __HAL_RCC_TIM2_CLK_DISABLE();    // 定时器不需要了
    
    // 2. 将所有未用IO设为模拟输入（最低功耗状态）
    GPIO_InitTypeDef gpio = {0};
    gpio.Mode = GPIO_MODE_ANALOG;
    gpio.Pull = GPIO_NOPULL;
    // PB0-PB15设为AN...
    
    // 3. 挂起SysTick（Stop模式前必须！）
    HAL_SuspendTick();
    
    // 4. 进入Stop模式
    HAL_PWR_EnterSTOPMode(PWR_LOWPOWERREGULATOR_ON, PWR_STOPENTRY_WFI);
    
    // 5. 唤醒后重新配置
    HAL_ResumeTick();
    SystemClock_Config();  // 重新配置时钟（HSE/HIS停了）
    MX_GPIO_Init();        // IO恢复了
    MX_USART1_UART_Init(); // 串口恢复了
}

// Standby模式：电流最低，但唤醒=复位
void enter_standby_mode(void) {
    // 使能WKUP引脚（PA0）
    HAL_PWR_EnableWakeUpPin(PWR_WAKEUP_PIN1);
    __HAL_PWR_CLEAR_FLAG(PWR_FLAG_WU);
    
    // 进入Standby
    HAL_PWR_EnterSTANDBYMode();
    // ⚠ 永远不会到这里——唤醒后从头执行main()！
    // 所有RAM数据丢失，只能用备份域寄存器保存标志
}

低功耗设计黄金法则

功耗优化检查清单必备
✅ 外设：不用就关时钟！__HAL_RCC_XXX_CLK_DISABLE()。不要"不用就放那"——时钟开着=耗电
✅ GPIO：悬空引脚设为模拟输入（AN模式+NOPULL），避免浮空导致内部推挽振荡耗电
✅ 时钟：降频！72MHz→8MHz，电流降约80%。用HIS代替HSE也能省~1mA
✅ 看门狗：独立看门狗(IWDG)在Stop模式下仍运行，功耗~1μA，不影响低功耗
✅ LED：每个LED约1-5mA，不用就关（GPIO_PIN_RESET关灯）
✅ ADC：采样完立即关：__HAL_ADC_DISABLE()。ADC是耗电大户(~2mA)
✅ DMA：用DMA做数据搬运，CPU多睡觉。DMA功耗远低于CPU
✅ 通信：打包数据批量发送。减少无线电/收发器工作时间=节电
✅ 上拉/下拉：外部上拉电阻用100K以上（避免10K=330μA浪费）
✅ 供电电压：3.3V比5V省约40%功耗（动态功耗∝V²）

⏲ RTC闹钟唤醒实战

// 每5分钟自动唤醒采集一次温湿度，其余时间Stop模式
// 平均功耗 < 50μA！用CR2032可跑1年以上

void setup_rtc_wakeup(void) {
    // 禁用RTC写保护
    HAL_RTCEx_DeactivateWakeUpTimer(&hrtc);
    
    // 配置RTC闹钟：5分钟后唤醒
    RTC_AlarmTypeDef sAlarm = {0};
    sAlarm.Alarm = RTC_ALARM_A;
    sAlarm.AlarmTime.Hours = 0;
    sAlarm.AlarmTime.Minutes = 5;
    sAlarm.AlarmTime.Seconds = 0;
    sAlarm.AlarmTime.SubSeconds = 0;
    sAlarm.AlarmMask = RTC_ALARMMASK_NONE;
    sAlarm.AlarmSubSecondMask = RTC_ALARMSUBSECONDMASK_ALL;
    sAlarm.AlarmDateWeekDaySel = RTC_ALARMDATEWEEKDAYSEL_DATE;
    sAlarm.AlarmDateWeekDay = 1;
    
    HAL_RTC_SetAlarm_IT(&hrtc, &sAlarm, RTC_FORMAT_BIN);
}

// RTC唤醒回调
void HAL_RTC_AlarmAEventCallback(RTC_HandleTypeDef *hrtc) {
    // RTC闹钟触发！在中断上下文中
    // 设置标志或释放信号量，不要在中断中做复杂操作
}

void main_loop(void) {
    uint8_t wake_count = 0;
    
    while (1) {
        // 1. 采集数据（~10ms）
        float temp = read_temperature();
        float hum = read_humidity();
        float batt = measure_battery_voltage();
        
        // 2. 累加计数
        wake_count++;
        
        // 3. 每6次（即30分钟）发送一次BLE广播省电
        if (wake_count >= 6) {
            send_data_over_ble(temp, hum, batt);  // ~50ms
            wake_count = 0;
        }
        
        // 4. 设置下次唤醒闹钟
        setup_rtc_wakeup();
        
        // 5. 进入Stop模式 → 功耗降到20μA！
        enter_stop_mode();
        // ... 5分钟后RTC闹钟唤醒，回到步骤1
    }
}

📱 ESP32功耗优化实战

ESP32功能强大但功耗也大。用好睡眠模式是电池续航的关键。

// ESP32功耗等级（参考数据，实际因模块而异）
// 运行模式(WiFi TX 802.11b): ~350mA
// 运行模式(CPU 240MHz): ~68mA
// 运行模式(CPU 80MHz): ~30mA
// Modem Sleep: ~3mA @ DTIM=10
// Light Sleep: ~0.8mA
// Deep Sleep: ~5μA (RTC+ULP可用)
// Hibernation: ~2.5μA (仅RTC定时器)

// Deep Sleep 实战：每30秒唤醒采一次温湿度
#include "esp_sleep.h"
#include "esp_wifi.h"

#define uS_TO_S_FACTOR 1000000
#define TIME_TO_SLEEP  30

// 保存在RTC慢速内存中的数据（Deep Sleep后仍保留）
RTC_DATA_ATTR int boot_count = 0;
RTC_DATA_ATTR float last_temp = 0;

void enter_deep_sleep(void) {
    // 1. 关闭WiFi/BLE（功耗大头）
    esp_wifi_stop();
    esp_bt_controller_disable();
    
    // 2. 配置唤醒源：定时器
    esp_sleep_enable_timer_wakeup(TIME_TO_SLEEP * uS_TO_S_FACTOR);
    
    // 3. 可选：EXT0/EXT1唤醒（RTC GPIO）
    // esp_sleep_enable_ext0_wakeup(GPIO_NUM_4, 1);  // GPIO4高电平唤醒
    // esp_sleep_enable_ext1_wakeup(BIT(GPIO_NUM_4)|BIT(GPIO_NUM_5), ESP_EXT1_WAKEUP_ANY_HIGH);
    
    // 4. 可选：ULP协处理器唤醒（超低功耗模式下的简单逻辑）
    // 例如：ULP每1秒读一次ADC，超过阈值才唤醒主CPU
    
    // 5. 进入Deep Sleep
    esp_deep_sleep_start();
    // ⚠ 这行代码永远执行不到！Deep Sleep后CPU完全断电
}

// 快速WiFi连接（优化启动时间）
void wifi_quick_connect(void) {
    wifi_config_t wifi_cfg = {
        .sta = {
            .ssid = "myssid",
            .password = "mypassword",
            .scan_method = WIFI_FAST_SCAN,    // 跳过全量扫描
            .sort_method = WIFI_CONNECT_AP_BY_SIGNAL,
            .threshold.authmode = WIFI_AUTH_WPA2_PSK,
        },
    };
    esp_wifi_set_config(ESP_IF_WIFI_STA, &wifi_cfg);
    esp_wifi_connect();
    // 相比默认模式可节省2-3秒=节省~0.3mAh
}

void app_main(void) {
    boot_count++;
    
    // 检查唤醒原因
    esp_sleep_wakeup_cause_t cause = esp_sleep_get_wakeup_cause();
    
    if (cause == ESP_SLEEP_WAKEUP_TIMER) {
        // 定时器唤醒：采集数据
        float temp = read_dht11_temperature();
        ESP_LOGI("TEMP", "Boot #%d: %.1f°C", boot_count, temp);
        
        // 智能判断：温度没变化就不浪费电发WiFi
        if (fabs(temp - last_temp) < 1.0f && boot_count % 10 != 0) {
            last_temp = temp;
            enter_deep_sleep();  // 直接睡，不连WiFi
        }
        last_temp = temp;
    }
    
    // WiFi初始化 + 发送数据（约2秒，约70mAh消耗）
    nvs_flash_init();
    esp_netif_init();
    wifi_init_sta();
    wifi_quick_connect();        // ~1.5s
    send_data_to_server(temp);   // ~0.5s
    
    // 任务完成，洗澡睡觉
    enter_deep_sleep();
}

ESP32动态调频

// ESP32支持动态切换CPU频率，省电同时保持响应
#include "esp_pm.h"

void configure_dynamic_freq(void) {
    esp_pm_config_esp32_t pm_cfg = {
        .max_freq_mhz = 240,   // 最高240MHz（负载大时）
        .min_freq_mhz = 80,    // 最低80MHz（空闲时）
        .light_sleep_enable = true,  // 允许自动Light Sleep
    };
    esp_pm_configure(&pm_cfg);
    // CPU自动根据负载升降频，无需手动控制
}

// 手动设置固定频率（如果需要）
set_cpu_freq(80);  // 80MHz ~30mA
set_cpu_freq(160); // 160MHz ~45mA

模式1：事件驱动 + Tickless（扔掉while(1)里的轮询）

// ❌ 传统轮询风格（功耗杀手）
void main_loop_bad(void) {
    while(1) {
        if (button_pressed()) {          // 轮询按钮 → CPU永远在跑
            handle_button();
        }
        if (uart_data_available()) {     // 轮询串口 → 同上
            handle_uart();
        }
        // SysTick每1ms中断 → CPU不能深睡 → 至少5mA
        HAL_Delay(10);  // 延迟也是轮询！
    }
}

// ✅ 中断驱动风格（低功耗友好）
void setup_interrupts(void) {
    // 按钮用EXTI外部中断 → CPU只在按下时醒
    HAL_NVIC_EnableIRQ(EXTI0_IRQn);
    // 串口用RXNE中断或DMA+空闲中断 → 来了数据才醒
    HAL_UART_Receive_IT(&huart1, rx_byte, 1);
}

// main函数：做最少的事，睡最久的觉
int main(void) {
    HAL_Init();
    SystemClock_Config();
    setup_interrupts();

    while(1) {
        // 处理完所有待处理事件后进入Stop模式
        process_events();  // 一次性处理所有积压事件
        if (no_more_work()) {
            enter_stop_mode();  // 睡了！直到下一个中断
            // 某中断唤醒了CPU：EXTI or UART or RTC
        }
    }
}

// 关键转变：while(1)循环体从"高速轮询"变成"睡醒-干活-再睡"

模式2：延迟工作队列（Deferred Work）

// 思想：ISR只做最重要的事（设置标志/读寄存器），
// 耗时操作推迟到主循环批量处理，减少总唤醒时间

typedef struct {
    void (*callback)(void*);
    void *arg;
    uint32_t deadline;  // 可以延迟处理的截止时间
} deferred_work_t;

#define MAX_DEFERRED 16
deferred_work_t work_queue[MAX_DEFERRED];
volatile uint8_t work_count = 0;

// ISR中只入队，不处理
void EXTI0_IRQHandler(void) {
    if (__HAL_GPIO_EXTI_GET_IT(GPIO_PIN_0)) {
        __HAL_GPIO_EXTI_CLEAR_IT(GPIO_PIN_0);
        // 只做最小的操作：记录事件
        enqueue_work(handle_button_press, NULL, HAL_GetTick() + 50);
    }
}

// 主循环集中处理（一次醒来做完所有事，效率最高）
void process_deferred_work(void) {
    for (int i = 0; i < work_count; i++) {
        if (HAL_GetTick() >= work_queue[i].deadline) {
            work_queue[i].callback(work_queue[i].arg);
            // 移除已处理的工作项
            work_queue[i] = work_queue[--work_count];
            i--;
        }
    }
}
// 优势：ISR < 5μs，主循环批量处理一次20μs → 比每次ISR都处理省4倍时间

模式3：分级电源状态机

// 不要只有"全开"和"全关"两种状态！设计多级省电状态：
typedef enum {
    PWR_ACTIVE,      // 全速运行：240MHz, 所有外设开, ~68mA
    PWR_IDLE,        // 空闲：80MHz, 关SPI/I2C, 只留UART+Timer, ~30mA
    PWR_LOW_POWER,   // 低功耗：80MHz, 只留RTC+LPUART, 定期短暂醒, ~3mA
    PWR_DEEP_SLEEP,  // 深睡：Stop/Standby, RTC闹钟或GPIO唤醒, ~5μA
    PWR_OFF          // 全关：外部PMOS切断全部供电, 0μA
} power_state_t;

power_state_t decide_power_state(void) {
    uint32_t idle_time = HAL_GetTick() - last_activity_tick;
    float batt_voltage = read_battery_voltage();

    if (batt_voltage < 3.3f) {
        // 电池快没电了：强制深睡，只保留关键功能
        return PWR_DEEP_SLEEP;
    }
    if (idle_time > 60000) {  // 1分钟没操作
        return PWR_DEEP_SLEEP;
    }
    if (idle_time > 5000) {   // 5秒没操作
        return PWR_LOW_POWER;
    }
    if (idle_time > 1000) {   // 1秒没操作
        return PWR_IDLE;
    }
    return PWR_ACTIVE;
}

// 状态转换时正确开关外设（这是最容易被忽略的地方！）
void transition_power_state(power_state_t new_state) {
    if (current_state == PWR_ACTIVE && new_state == PWR_IDLE) {
        __HAL_RCC_SPI1_CLK_DISABLE();
        __HAL_RCC_I2C1_CLK_DISABLE();
        // UART+Timer保持开启（基本通信+心跳）
    }
    if (new_state == PWR_DEEP_SLEEP) {
        // 进入前保存上下文到备份寄存器
        HAL_RTCEx_BKUPWrite(&hrtc, RTC_BKP_DR1, boot_count);
        // 全关 → 进Stop
        deinit_all_peripherals();
        enter_stop_mode();
        // 醒来后init_all_peripherals()恢复
    }
    current_state = new_state;
}

模式4：STM32 Tickless 模式

标准FreeRTOS用SysTick每1ms中断一次——这意味MCU每1ms就要醒一次，即使无事可做！Tickless模式让OS只在"下一个任务到期"时醒来。

// FreeRTOS Tickless配置（FreeRTOSConfig.h）
#define configUSE_TICKLESS_IDLE  1  // 启用Tickless

// 然后实现这两个函数（FreeRTOS会调它们）：
// vPortSuppressTicksAndSleep()：OS空闲时计算下次唤醒时间 → 进Stop
// xPortSysTickHandler()替代品：用RTC/低功耗Timer做唤醒源

// 以STM32为例，用RTC闹钟做Tickless唤醒（HAL库实现）：
void vPortSuppressTicksAndSleep(TickType_t xExpectedIdleTime) {
    // 1. 关SysTick（Stop模式下它不跑）
    // 2. 设RTC闹钟 = 当前时间 + xExpectedIdleTime 对应的秒数
    // 3. 进Stop模式
    // 4. 醒来后修正FreeRTOS的tick计数（补偿睡眠的时间）

    if (xExpectedIdleTime > 5) {  // 至少睡5个tick才值得进Stop
        uint32_t sleep_ms = xExpectedIdleTime * portTICK_PERIOD_MS;
        HAL_RTCEx_DeactivateWakeUpTimer(&hrtc);
        HAL_RTCEx_SetWakeUpTimer_IT(&hrtc, sleep_ms * 10, RTC_WAKEUPCLOCK_RTCCLK_DIV16);

        __disable_irq();
        HAL_SuspendTick();
        HAL_PWR_EnterSTOPMode(PWR_LOWPOWERREGULATOR_ON, PWR_STOPENTRY_WFI);
        HAL_ResumeTick();
        __enable_irq();

        // 修正FreeRTOS时间
        vTaskStepTick(sleep_ms / portTICK_PERIOD_MS);
    }
}
// 效果：SysTick每1ms中断(5mA) → RTC每10-50ms中断(<100μA平均)

// LPBAM = Low Power Background Autonomous Mode
// 这是STM32U5最革命性的低功耗特性：
// LPDMA + LPTIM + LPUART/ADC可以在Stop 2模式下全自动运行
// CPU全程不醒，只有DMA传输完成后才唤CPU处理数据

// 场景：每1秒自动采集1次ADC → 自动通过LPUART发送 → 全部在Stop 2下完成
// CPU只在ADC缓冲区满（采集了32次=32秒）才醒一次批量处理

#include "stm32u5xx_hal.h"
#include "lpbm_config.h"  // STM32CubeMX生成

void setup_lpbam_acquisition(void) {
    // 1. LPTIM1每1秒触发ADC转换（全在低功耗域运行）
    // 2. ADC结果由LPDMA链式传输到SRAM4（Stop模式下保持供电的RAM区域）
    // 3. LPUART1由LPDMA推送数据（可选，需要发送时）
    // 全部在硬件层面完成，CPU在Stop 2模式睡觉

    HAL_LPBAM_Init(&hlpbam);
    HAL_LPBAM_Start(&hlpbam);

    // 进入Stop 2：CPU睡了，但LPTIM/LPDMA/LPUART仍在后台工作
    HAL_PWREx_EnterSTOP2Mode(PWR_STOPENTRY_WFI);

    // 唤醒后：LPDMA已经填充了32个ADC数据在SRAM4里
    // 直接处理，不需要重新初始化外设！
}

// 传统F1方案对比：
// Timer中断(CPU醒) → 启动ADC → EOC中断(CPU再醒) → 读结果 → 睡
// 每次采样CPU醒两次，共~50μs × 2 = 100μs
// 1秒周期：100μs/1s = 0.01%占空比，但中断开销累计不可忽略
// U5 LPBAM：CPU完全不用醒，功耗 < 3μA (包括LPTIM+LPDMA)

🛠 常见问题排查

🔍 问题1：Stop模式功耗远高于datasheet标称值
❌ 症状：datasheet说Stop模式20μA，实测>500μA。
✅ 排查：
1. GPIO浮空！未初始化的IO默认浮空输入→内部触发器振荡→耗电
2. 外设时钟没关：USART/TIM/ADC开着即使不用也耗电
3. 调试器连着：ST-Link/J-Link的SWD接口在Stop模式下拉电流
4. 外部上拉电阻太小：10K上拉到3.3V=330μA！
✅ 修复：
1. 所有未用IO设AN模式：GPIO_MODE_ANALOG + GPIO_NOPULL
2. __HAL_RCC_XXX_CLK_DISABLE()关所有不用的时钟
3. 测量时拔掉调试器
4. 上拉电阻换100K~1M
🔍 问题2：Standby唤醒后程序行为异常
❌ 症状：Standby唤醒后变量值不对、外设配置丢失。
✅ 原因：Standby=芯片复位！RAM全部丢失，所有外设复位到默认状态。
✅ 修复：
1. 唤醒后完整执行初始化（和上电行为一致）
2. 用备份域寄存器(BKP_DRx)保存标志判断是冷启动还是Standby唤醒
3. if(__HAL_PWR_GET_FLAG(PWR_FLAG_SB)) { /* Standby唤醒 */ }
4. 对时间敏感的应用用Stop模式代替Standby
🔍 问题3：DC-DC输出电压剧烈纹波/不稳定
❌ 症状：用示波器看到输出波形锯齿状，峰峰值>200mV。
✅ 排查：
1. 输入电容离IC太远(>5mm) → 输入电流环太大 → 不稳定
2. 电感饱和电流不够 → 接近满载时电感饱和 → 电流失控
3. 输出电容ESR太高 → 不能用普通电解电容
4. 反馈FB走线被干扰 → FB是模拟信号！
✅ 修复：
1. 输入电容贴IC引脚放（陶瓷10μF+0.1μF）
2. 确认电感标称电流>1.5倍负载
3. 输出电容用陶瓷MLCC，ESR尽量低
4. FB走线短而直，远离SW节点，包地
🔍 问题4：电池电量显示不准（跳跃/死机不降）
❌ 症状：电量从80%直接跳到20%，或长时间不动。
✅ 排查：
1. 电压法受温度影响大：低温时锂电池内阻高，带载电压跌落明显
2. 分压电阻温漂：普通电阻温漂100ppm/°C，50°C温差=0.5%误差
3. ADC参考电压不准：用内部1.2V参考代替VDDA
4. 测量时带载vs空载电压差很大：锂电池内阻~100mΩ
✅ 修复：
1. 多次采样+滑动平均滤波
2. 用内部参考电压校准：Vref = 1.2V * 4096/ADC_Vrefint
3. 结合电压+库仑计，或直接用MAX17048
4. 在CPU空闲时采样（负载稳定）
🔍 问题5：TP4056充电时芯片发烫
❌ 症状：充电过程中TP4056温度非常高(>70°C)。
✅ 排查：
1. 充电电流设置过大：默认1A对应1.2KΩ
2. 输入电压偏高：5V正常，>5.5V会加大热耗
3. PCB散热不够：TP4056底部焊盘没接地铜皮
4. 电池电压很低→大压差→高功耗
✅ 修复：
1. 降低充电电流：换2K电阻→580mA，或10K→120mA
2. USB供电用5V（不要用快充头，可能输出9V/12V！）
3. PCB底部铺大片铜皮做散热，最好用过孔到背面
4. 热保护是正常的：TP4056在120°C自动降电流

nRF52 低功耗设计的三个核心理念

理念1：PPI（可编程外设互联）—— 外设间不需要CPU介入
这是nRF52最革命性的设计。PPI允许Timer触发ADC采样、ADC完成后自动通过SPI发送数据、完成后自动触发RTC——全部在硬件层面完成，CPU全程睡眠。

// nRF52 PPI + EasyDMA：零CPU数据采集
// 配置：Timer每100ms触发 → ADC自动采样 → 结果DMA直传RAM → 完成后中断唤醒CPU
// CPU只在中断处理时醒100μs，其余99.99%时间在睡觉

// STM32等效方案：Timer触发 → ISR中启动ADC → 等EOC中断 → ISR中读数据
// CPU每次都要醒来处理中断，功耗高5-10倍

// STM32虽没有PPI，但可以用DMA+Timer触发来逼近：
// Timer TRGO → ADC触发(DMA循环模式) → 数据自动进RAM
// 只有DMA传输完成(缓冲区满)才中断唤醒CPU
理念2：EasyDMA —— 所有外设自带DMA
nRF52的UART/SPI/I2C/ADC/PWM/RADIO全部内置EasyDMA。传统MCU只有少量DMA通道(STM32F103只有7个通道)，外设间常抢DMA。nRF52每个外设自带DMA，完全不用争抢。

理念3：128MHz Cortex-M4F —— 做完就睡
为什么nRF52840运行在128MHz而非更低的64MHz？因为更高的频率意味着更短的运行时间。 一个SPI传输100字节的任务：
• 64MHz下CPU醒80μs → 24μA平均
• 128MHz下CPU醒50μs → 15μA平均（跑得快=睡得早=更省电！）

STM32上借鉴nRF52的设计理念

// 技巧1：用DMA+TIM触发做"伪PPI"数据采集
// Timer通道输出(OC/CH1)触发ADC规则组转换
// ADC用DMA循环模式，自动填充环形缓冲区
// CPU只在DMA半传输/全传输中断时短暂醒来处理

// 初始化代码示例（STM32F103）：
ADC_InitTypeDef adc = {0};
adc.ExternalTrigConv = ADC_EXTERNALTRIGCONV_T3_TRGO; // Timer3触发
HAL_ADC_Init(&hadc1);

HAL_ADC_Start_DMA(&hadc1, (uint32_t*)adc_buf, BUF_SIZE); // 自动DMA
// 现在Timer3溢出 → 自动触发ADC → DMA直写RAM → BTC/HT中断唤醒CPU
// 整个过程CPU在Stop模式睡觉，平均功耗 < 50μA

// 技巧2：用RTC闹钟替代SysTick做系统心跳
// SysTick依赖CPU运行时钟 → Sleep模式下停止 → 必须靠中断唤醒
// RTC用LSE 32.768KHz独立时钟 → Stop模式下仍然运行
HAL_RTCEx_SetWakeUpTimer_IT(&hrtc, 10, RTC_WAKEUPCLOCK_RTCCLK_DIV16);
// 每10/32768*16 ≈ 4.88ms唤醒一次（>1ms的延时可用RTC替代SysTick）

// 技巧3：异步串口替代轮询/中断串口
// STM32F1/F4的LPUART（低功耗UART）可在Stop模式下接收数据
// 收到完整字节后才唤醒CPU——不要用USART_IT_RXNE(每字节都中断)
if (IS_UART_INSTANCE(USART1) == 1) {

⚡ 电源监控与上电复位电路

电源管理不只是省电——让芯片在电压异常时安全运行同等重要。BOD（掉电检测）、POR（上电复位）和外部监控IC是嵌入式系统的最后一道防线。

BOD（Brown-Out Detector）：掉电检测

// STM32内置BOD：VDD低于阈值时自动复位MCU
// 防止低电压下Flash写入错误、GPIO状态不可控等问题

// STM32F103 BOD配置（在system_stm32f1xx.c或option bytes）
// 阈值可选：2.0V / 2.3V / 2.5V / 2.7V（通过Option Bytes设置）
// 2.7V阈值 → VDD低于2.7V就复位，保护Flash

// STM32L4 BOD更灵活：可产生中断而非复位
HAL_PWR_EnableBOR();  // 上电默认开启
// STM32L4有三个BOR阈值等级：Level 1(1.7V) Level 2(2.0V) Level 3(2.3V)

// BOD对低功耗的影响：
// STM32F103的BOD在Stop模式下消耗约10-20μA（不小！）
// STM32L4的BOR在Stop模式下消耗约0.5μA（好多了）
// ⚠ 低功耗应用要权衡：BOD关掉可省电，但电压跌落时可能死机或Flash损坏
⚠️ BOD的坑：电池供电设备在电机启动/GSM发送瞬间电压可能骤降。如果BOD阈值设太高(2.7V)，瞬间低压会触发复位。解决方案：① 输入端加大电容(>100μF)缓冲；② 用超级电容/法拉电容做蓄能；③ 降低BOD阈值到2.3V（但确保Flash工作电压允许）。

外部电压监控IC（比内置BOD更可靠）

📝 MAX809 典型接法（最简单可靠的复位方案） 
//                VCC
//                 │
//             ┌───┴───┐
// 100nF ───┤1     3├─── NRESET (接MCU的NRST脚)
//             │ MAX809│
//  GND  ───┤2      │
//             └───────┘
// 电容靠近引脚，NRST脚上拉10K到VCC（MCU内部一般已有上拉）
// 原理：VCC低于阈值 → 第3脚拉低 → MCU复位 → VCC恢复超阈值 + 延时240ms → 第3脚释放

上电时序：多电压轨系统的陷阱

现代MCU常有多路电源：VDD(1.8V核心) + VDDA(3.3V模拟) + VBAT(备份)。上电顺序不对会导致闩锁效应(Latch-up)烧毁芯片。

⚠️ STM32F4上电顺序要求（数据手册Section 5.1.1原话）：
VDD必须在VDDA之前或同时上电。若VDDA先上而VDD为0V，模拟引脚可能向数字域漏电，触发寄生SCR结构→芯片烧毁。

解决方案：
① 用带EN引脚的LDO控制上电顺序
② 用TPS3808等时序IC串联：先使能VDD的DC-DC，VDD稳定后使能VDDA的LDO
③ 低成本方案：VDD的LDO输出接VDDA的EN脚(分压到1.2V以上)，自然形成时序
④ 如果只有一路供电，VDDA和VDD直接短接，在VDD引脚上并联磁珠隔离（STM32参考设计推荐）

🌞 太阳能+锂电池充电电路设计

户外传感器/气象站/共享设备常需要太阳能供电。以下是经过验证的实用方案：

// 太阳能充电架构：
// 太阳能板 → CN3791(MPPT充电IC) → 18650锂电池 → ME6206(LDO) → MCU

// CN3791 关键参数（太阳能专用充电IC，国产替代CN3065/CN3165方案）：
// - 内置MPPT（最大功率点跟踪）：太阳能板电压低于设置值时自动降低充电电流
// - 输入电压：4.5V-28V
// - 充电电流：可编程，最大4A (RISET电阻设置)
// - 完整的涓流-恒流-恒压三段式充电
// - SOT-23-5封装：VIN, BAT, GND, ISET, 悬空(CE)

// 太阳能板选型指南：
// 5V/1W (200mA): 小型传感器节点，间歇工作(每天唤醒几次)
// 5V/2.5W (500mA): BLE网关，持续BLE扫描
// 12V/5W (400mA): LoRa网关，ESP32持续WiFi
// 公式：太阳能板功率(W) > (日耗电Wh) / (峰值日照小时×0.6效率)
// 例：ESP32平均50mA×3.3V=0.165W，日耗0.165×24=3.96Wh
// 日照4h → 3.96/(4×0.6) = 1.65W → 至少选2W太阳能板

// 低成本方案：TP4056太阳能充电（没有MPPT但有温度保护）
// TP4056的TEMP脚可接NTC做温度保护(电池过热停充)
// 太阳能板接5V输入端，板子输出接电池
// ⚠ TP4056无欠压锁定(UVLO)，低光时可能倒灌电池→串肖特基二极管防倒灌
💡 太阳能系统设计三件套：① 防倒灌二极管（SS34肖特基，VF≈0.3V）串在太阳能板和充电IC之间；② 过放保护：DW01+8205A保护板(<¥1)或硬件切断电路；③ MCU读取电池电压，低于3.3V时进入"省电模式"（只保留RTC+看门狗）。

🐛 电源管理踩坑实录

🔍 坑1：锂电池接LDO后MCU启动不了（瞬间电压跌落）

❌ 现象：电池电压3.8V完全正常，但MCU一上电就反复复位
✅ 根因：MCU启动瞬间需要大电流（STM32F103约30mA峰值），电池内阻(>200mΩ旧电池)+线路电阻导致LDO输入端瞬间跌落到3.1V以下，LDO无法维持输出
✅ 解决：
① LDO输入并联大电容：100μF电解 + 10μF MLCC + 0.1μF MLCC
② LDO输出也要并：22μF + 0.1μF（参考LDO数据手册的稳定性要求）
③ 换更低静态电流的LDO（如ME6206在电池设备远优于AMS1117）
④ 旧电池内阻大，测试时用新电池
🔍 坑2：Stop模式功耗远超数据手册声称的20μA

❌ 现象：进入Stop模式后电流约200-500μA，而非手册说的20μA
✅ 根因（按概率排序）：
① GPIO没有全设AN模式（最常见！）：悬空的输入引脚会产生额外的施密特触发器电流（每个引脚约10-50μA）。解决办法：所有未用引脚设为GPIO_MODE_ANALOG + GPIO_NOPULL
② 调试器( ST-Link/J-Link)还连着：调试接口本身耗电30-100μA。关掉调试器或断开SWD连线
③ 外设时钟没有全关：检查RCC寄存器，把所有不需要的外设时钟禁用
④ LDO/DC-DC静态电流：板上LDO本身的Iq也算在你的"MCU功耗"里——比如AMS1117的5-10mA Iq远大于MCU的20μA Stop电流！换ME6206(7μA Iq)后立即见效
⑤ 上拉/下拉电阻：10K上拉在3.3V下吃0.33mA！上拉电阻至少100K或1M
🔍 坑3：RTC闹钟在该唤醒的时候没有触发

❌ 现象：Stop模式下去后永远醒不过来，或者醒来的时间完全不对
✅ 根因：
① RTC时钟源选了HSI——Stop模式下HSI关闭了！必须用LSE(32.768KHz晶振)或LSI(内部低速RC)
② LSE晶振没有起振——加6pF负载电容(具体值查数据手册)，PCB走线尽量短
③ 闹钟中断优先级错误——中断分组配置忘了
✅ 解决：
① 确认RTC时钟源：__HAL_RCC_RTC_CONFIG(RCC_RTCCLKSOURCE_LSE)
② LSE起振检查：读RCC_BDCR的LSERDY位
③ 闹钟中断使能：HAL_RTC_SetAlarm_IT(&hrtc, &sAlarm, RTC_FORMAT_BIN)
④ 闹钟中断在NVIC中使能：HAL_NVIC_SetPriority(RTC_Alarm_IRQn, 0, 0); HAL_NVIC_EnableIRQ(RTC_Alarm_IRQn);
🔍 坑4：DC-DC模块在轻载时效率暴跌

❌ 现象：MP1584模块标称效率92%，但实测轻载(10mA)只有60%，电池很快没电
✅ 根因：DC-DC的开关损耗、驱动损耗、控制电路功耗在轻载时占主导。92%是在满载(1-2A)时的效率
✅ 解决：
① 选择支持轻载高效模式(PFM/Pulse Skipping)的DC-DC芯片：
- TPS62740(DCS-Control, 360nA Iq, 10μA负载效率>90%)
- TPS62800(Iq=150nA，轻载效率极佳)
② MP1584在低负载下有PFM模式(自动切换)，但仍不如专用低Iq芯片
③ 如果MCU大部分时间在睡眠(<1mA电流)，直接用超低Iq LDO比DC-DC更省电——LDO的Iq只有1μA，而DC-DC的Iq可能上百μA
🔍 坑5：ADC读数漂移——VREF基准电压不稳定

❌ 现象：电池电压测量值同一电压下忽高忽低，跳变几十mV
✅ 根因：ADC参考电压用了VDDA=直接接电池或非稳压电源。电池从4.2V跌到3.3V过程中，VDDA也一直在变——你测的"电池电压"其实是电池电压/电池电压=恒定的
✅ 解决：
① 使用MCU内置VREFINT通道：STM32有内部1.2V基准，先读VREFINT校准ADC：
float vdda = 1.2f * (*VREFINT_CAL_ADDR) / adc_read(ADC_CHANNEL_VREFINT);
② 外部基准IC：TL431(¥0.5, 1%), REF3033(¥2, 0.2%), LM4040(¥1, 0.1%)
③ STM32校准值在Flash系统存储区：*VREFINT_CAL_ADDR(0x1FFFF7BA-0x1FFFF7BB)
④ 关键提醒：STM32的数据手册里校准值是在VDDA=3.3V±10mV、TA=25°C条件下测的。如果你的VDDA偏离3.3V，VREFINT仍然稳定在1.2V(±1%)——所以VREFINT是真正的"锚点"

📊 功耗计算与电池续航估算

// 电池续航计算公式
// 续航(h) = 电池容量(mAh) / 平均电流(mA)
// 平均电流 = (运行电流×运行时间 + 睡眠电流×睡眠时间) / 总周期

// 实例：BLE温湿度信标
// 唤醒200ms: 采集10ms(5mA) + BLE广播50ms(15mA) + 其他40ms(5mA)
// 睡眠30s: Stop模式(20μA)
// 平均电流 = (5×10 + 15×50 + 5×40 + 20e-3×29900) / 30000
//          = (50 + 750 + 200 + 598) / 30000
//          = 1598 / 30000 = 0.0533 mA
// CR2032(230mAh)续航 = 230/0.0533 ≈ 4315小时 ≈ 180天

// 编写功耗计算器
float calc_avg_current(void) {
    float run_i = 15.0;   // 运行时电流(mA)
    float run_t = 200;    // 运行时间(ms)
    float sleep_i = 0.02; // 睡眠电流(mA)
    float period = 30000; // 总周期(ms)
    
    float sleep_t = period - run_t;
    return (run_i * run_t + sleep_i * sleep_t) / period;
}

float calc_battery_life(float capacity_mah, float avg_current_ma) {
    return capacity_mah / avg_current_ma;  // 小时
}

📋 知识点回顾

✅ 电源管理核心：
• LDO vs DC-DC：小压差(<1V)/低电流(<100mA)→LDO(简单无噪声)；大压差/大电流→DC-DC(高效)
• 锂电池：标称3.7V、充满4.2V、截止3.0V，必须有DW01+8205A保护板
• 电量测量：电压法(简单±10%) vs 库仑计(精确±1%)，MAX17048即插即用
• STM32低功耗：Sleep(~5mA)→Stop(~20μA)→Standby(~2μA)，Stop最常用
• ESP32 Deep Sleep：~5μA，唤醒源=定时器/GPIO/ULP协处理器
🛠 设计检查清单：
• PCB布局：DC-DC输入电容<5mm，开关节点面积最小，FB线远离SW
• 电池安全：保护板必须有(过充4.25V/过放2.4V/短路)，充电IC选TP4056
• 低功耗IO：悬空引脚→AN模式+NOPULL，上拉电阻≥100K
• 测试验证：万用表μA档/PPK2测量实际功耗，运行态+睡眠态都要测
📏 续航估算公式：
• 平均电流 = (I_run×t_run + I_sleep×t_sleep)/t_total
• 续航小时 = 电池容量(mAh)/平均电流(mA)
• 例：CR2032(230mAh) + 平均0.05mA = 4600小时 ≈ 191天
