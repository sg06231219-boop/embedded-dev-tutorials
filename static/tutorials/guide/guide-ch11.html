第11章：调试技巧大全

📖 小白完全指南 · 11/14 | 阅读时间约3分钟

写代码的时间占30%，调试的时间占70%。掌握高效的调试方法，是嵌入式工程师最重要的技能。

💡 类比：调试就像破案——printf=监控摄像头（全局观察），GDB=现场勘查（停下来仔细看），HardFault分析=验尸报告（分析死因），SWO=实时监控（不间断观察）。

printf调试 — 最简单最常用

// 重定向printf到UART
#include <stdio.h>

int fputc(int ch, FILE *f) {
    HAL_UART_Transmit(&huart1, (uint8_t *)&ch, 1, HAL_MAX_DELAY);
    return ch;
}

// 使用ITM（SWO引脚）重定向——比UART快100倍！
int fputc(int ch, FILE *f) {
    ITM_SendChar(ch);
    return ch;
}

// 调试宏
#define DEBUG_PRINT(fmt, ...) printf("[%lu] " fmt "
", HAL_GetTick(), ##__VA_ARGS__)
// 使用：DEBUG_PRINT("ADC value: %d", adc_val);
// 输出：[12345] ADC value: 2048

SWO/ITM 输出

ITM(Instrumentation Trace Macrocell)是Cortex-M内置的调试输出通道，通过SWO引脚输出数据，比UART快得多且不占用串口资源。

// 在OpenOCD配置中启用SWO
# openocd.cfg
swj_newdap $_CHIPNAME cpu -irlen 4 -ircapture 0x1 -irmask 0xf -expected-id $_CPUTAPID
$_TARGETNAME configure -event gdb-attach {
    itm port 0 on
}

// GDB中查看ITM输出
(gdb) monitor tpiu config internal /tmp/itm.log uart off 72000000
(gdb) monitor itm port 0 on

HardFault分析 — 程序崩溃急救

⚠️ 注意：下面的汇编代码使用了Thumb-2指令（ITE EQ），仅适用于Cortex-M3/M4/M7。Cortex-M0/M0+不支持，如STM32F0/G0系列需改用条件分支写法。
// HardFault处理函数（放在main.c中）
void HardFault_Handler(void) {
    __asm volatile (
        "TST LR, #4
"          // 检查使用哪个栈
        "ITE EQ
"
        "MRSEQ R0, MSP
"       // 主栈
        "MRSNE R0, PSP
"       // 进程栈
        "B Hard_Fault_Handler
" // 跳到C函数
    );
}

void Hard_Fault_Handler(uint32_t *stack_frame) {
    printf("
==== HardFault! ====
");
    printf("R0  = 0x%08X
", stack_frame[0]);
    printf("R1  = 0x%08X
", stack_frame[1]);
    printf("R2  = 0x%08X
", stack_frame[2]);
    printf("R3  = 0x%08X
", stack_frame[3]);
    printf("R12 = 0x%08X
", stack_frame[4]);
    printf("LR  = 0x%08X
", stack_frame[5]);   // 返回地址
    printf("PC  = 0x%08X
", stack_frame[6]);   // 崩溃地址！
    printf("PSR = 0x%08X
", stack_frame[7]);

    // 在.map文件或反汇编中查找PC地址对应的函数
    volatile uint32_t fault_addr = stack_frame[6];
    while (1);  // 停在这里，用GDB查看
}

// GDB中查看崩溃位置
(gdb) info line *0x08001234    // 把PC值换成实际的
(gdb) list *0x08001234         // 显示源码

常见Bug模式

Bug类型症状特征排查方法
空指针HardFaultPC在非法地址检查指针是否为NULL
栈溢出变量莫名变化、HardFault局部变量被覆盖uxTaskGetStackHighWaterMark()
中断竞争数据偶发错误中断中修改共享变量关中断保护或用原子操作
未初始化变量行为随机上电第一次不正常开启-Wall -Wuninitialized
看门狗复位程序周期性重启IWDG超时检查长阻塞代码
🔍 常见问题：程序偶尔跑飞，无法稳定复现
90%是中断竞争。排查：
1. 检查中断和主循环是否访问同一变量
2. 如果是，用 __disable_irq() / __enable_irq() 保护
3. 或用 volatile 关键字声明共享变量
4. 更好的方案：用FreeRTOS队列/信号量
💡 调试黄金法则：
① 先让Bug稳定复现（100%复现=解决了一半）
② 二分法注释代码，缩小Bug范围
③ 看.map文件确认代码/数据地址
④ 永远开启 -Wall -Werror 编译
