第0章：环境搭建 — Windows嵌入式开发环境

📖 小白完全指南 · 0/14 | 阅读时间约3分钟

嵌入式开发需要一套交叉编译工具链——因为你的电脑是x86架构，单片机是ARM架构，编译器必须在PC上运行、产出ARM机器码。就像在中国用电脑设计美国的零件，需要一套"跨国工具"。

💡 类比：嵌入式开发环境就像厨房——ARM GCC是菜刀（切菜=编译代码），CMake是菜谱模板（规划做菜流程），Make是自动炒菜机（按菜谱执行），串口工具是对讲机（和芯片对话的唯一方式）。没有这些，你连"hello world"都烧不进芯片。

🔧 PATH环境变量配置

把工具加到PATH，这样在任意目录的命令行里都能直接调用，不用每次输入完整路径。

步骤1：Win + S 搜索"环境变量" → 打开"编辑系统环境变量"
步骤2：点"环境变量" → 选中用户变量的"Path" → 点"编辑"
步骤3：点"新建"，加入以下路径（已自动配置的跳过）：
C:\Program Files (x86)\Arm GNU Toolchain arm-none-eabi\14.2 rel1\bin
C:\Program Files\CMake\bin
C:\Program Files (x86)\GnuWin32\bin
C:\Program Files\PuTTY
C:\Users\LYS\AppData\Local\Python\pythoncore-3.14-64\Scripts
💡 小技巧：加完PATH后，重新打开所有命令行窗口才会生效。验证：打开新的PowerShell，直接输入 arm-none-eabi-gcc --version，能运行就说明配好了。

🔍 环境一键验证脚本

复制下面的脚本到PowerShell运行，自动检测所有工具是否配置正确：

# 保存为 check_env.ps1 或直接复制运行
# 自动检测Python命令（python/python3/python3.14等）
$pyCmd = $null
foreach ($cmd in @('python3.14','python3','python')) {
    try { $null = & $cmd --version 2>&1; $pyCmd = $cmd; break } catch {}
}
if (-not $pyCmd) { Write-Host "❌ Python未找到，请确认已安装并加入PATH" -ForegroundColor Red }

$tools = @(
    @{name='ARM GCC'; cmd='arm-none-eabi-gcc --version'},
    @{name='CMake'; cmd='cmake --version'},
    @{name='Make'; cmd='make --version'},
    @{name='Python'; cmd="$pyCmd --version"},
    @{name='pyserial'; cmd="$pyCmd -m serial.tools.list_ports -h"},
    @{name='esptool'; cmd="$pyCmd -m esptool version"}
)

Write-Host "`n=== 嵌入式环境检测 ===`n" -ForegroundColor Cyan
foreach ($t in $tools) {
    Write-Host "检测 $($t.name)... " -NoNewline
    try {
        $result = Invoke-Expression $t.cmd 2>&1
        if ($LASTEXITCODE -eq 0 -or $result -match '\d') {
            Write-Host "✅ OK" -ForegroundColor Green
        } else {
            Write-Host "❌ 失败" -ForegroundColor Red
        }
    } catch {
        Write-Host "❌ 未安装" -ForegroundColor Red
    }
}
Write-Host "`n检测完成！" -ForegroundColor Cyan
⚠️ 注意事项：
• 如果Make检测失败，说明GnuWin32的bin目录未加入PATH，手动添加：C:\Program Files (x86)\GnuWin32\bin
• Python Scripts目录也需检查：C:\Users\你的用户名\AppData\Local\Programs\Python\Python3xx\Scripts
• pyserial和esptool装在哪个Python下，那个Scripts目录就要加PATH

🔧 Windows命令行环境优化

Windows自带的cmd.exe功能有限，强烈推荐使用PowerShell或Windows Terminal作为日常终端。以下是几个能让开发效率翻倍的配置技巧：

PowerShell Profile 配置

PowerShell的Profile（配置文件）类似Linux的.bashrc——每次启动终端自动加载。把你的常用alias和函数放进去：

# 创建/编辑Profile
notepad $PROFILE
# 如果文件不存在，先创建：
New-Item -Path $PROFILE -Type File -Force

# === 以下写入 Profile ===
# 嵌入式开发快捷键
function arm-gcc { arm-none-eabi-gcc $args }
function arm-size { arm-none-eabi-size $args }
function arm-objcopy { arm-none-eabi-objcopy $args }

# 快速构建函数（保存为 stm32-build）
function stm32-build {
    param([string]$target = "all")
    # 自动检测Makefile/CMakeLists.txt
    if (Test-Path "Makefile") {
        & make -j4 $target
    } elseif (Test-Path "CMakeLists.txt") {
        if (-not (Test-Path "build")) { mkdir build }
        cd build
        cmake -DCMAKE_TOOLCHAIN_FILE=../toolchain-arm.cmake -DCMAKE_BUILD_TYPE=Debug ..
        cmake --build . -j4
        cd ..
    } else {
        Write-Host "未找到Makefile或CMakeLists.txt" -ForegroundColor Red
    }
}

# 快速列出串口
function com-list {
    [System.IO.Ports.SerialPort]::GetPortNames()
    # 或者用 pyserial:
    # python -m serial.tools.list_ports
}

# 串口终端快捷函数
function com-term($port="COM3", $baud=115200) {
    python -m serial.tools.miniterm $port $baud
}
💡 PowerShell执行策略问题：如果运行脚本报"禁止运行脚本"，以管理员身份运行：
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

# 查看改动细节

git show HEAD               # 看最后一次commit改了啥

🐛 常见环境安装错误排解

🔍 错误1：arm-none-eabi-gcc 不是内部或外部命令
原因：ARM GCC的bin目录没有加入PATH，或加了PATH但没有重新打开命令行窗口。
解决步骤：
① 确认安装路径正确，打开文件管理器找到：C:\Program Files (x86)\Arm GNU Toolchain arm-none-eabi\14.2 rel1\bin
② 检查这个目录下是否有 arm-none-eabi-gcc.exe
③ 按Win+S搜索"环境变量" → 编辑用户PATH → 添加上面的bin路径
④ 关闭所有命令行窗口，重新打开（这步90%的人会漏掉！）
⑤ 新的命令行里运行 arm-none-eabi-gcc --version 验证
🔍 错误2：make --version 正常，但 make 命令报"make: *** No targets specified and no makefile found. Stop."
原因：这不是错误！这说明make找到了但没有Makefile。你需要在一个有Makefile的目录里运行make，或者创建一个Makefile。
🔍 错误3：cmake 报 "The C compiler identification is unknown"
原因：交叉编译器不在PATH中，CMake没法自动检测。
解决：
① 确保 arm-none-eabi-gcc --version 能正常执行
② 在CMake构建时显式指定工具链文件：
cmake -DCMAKE_TOOLCHAIN_FILE=../toolchain-arm.cmake ..
③ 确保toolchain-arm.cmake中的 CMAKE_C_COMPILER 路径正确
🔍 错误4：python -m serial.tools.miniterm 报"No module named serial"
原因：pyserial没有安装，或装在了其他Python版本下。
解决：
① python --version 确认是哪个Python
② pip install pyserial 重新安装
③ 如果装了多个Python，指定pip路径：
C:\Users\LYS\AppData\Local\Python\pythoncore-3.14-64\Scripts\pip install pyserial
🔍 错误5：Windows 终端中文乱码
解决： PowerShell中输入：
[System.Console]::OutputEncoding = [System.Text.Encoding]::UTF8
或在PowerShell Profile中加入：
$OutputEncoding = [Console]::OutputEncoding = [Text.Encoding]::UTF8
如果还不行，换用Windows Terminal——它对UTF-8支持比旧cmd好得多。

🚀 进阶话题：多版本工具链共存

多版本ARM GCC管理进阶
实际开发中你可能需要同时使用不同版本的ARM GCC——比如老项目用10.x，新项目用14.x。以下是在Windows上管理多版本的最佳实践：
# 方案1：在PATH中只保留一个默认版本，其他用绝对路径
# 默认版本（PATH中）：14.2
arm-none-eabi-gcc --version  # → 14.2

# 其他版本用完整路径：
"C:\Program Files (x86)\Arm GNU Toolchain arm-none-eabi\10.3 rel1\bin\arm-none-eabi-gcc" --version

# 方案2：创建版本切换函数（放入PowerShell Profile）
function use-arm-gcc-10 {
    $env:PATH = "C:\tools\arm-gcc-10\bin;$([Environment]::GetEnvironmentVariable('PATH','User'));$([Environment]::GetEnvironmentVariable('PATH','Machine'))"
    arm-none-eabi-gcc --version
}

# 方案3：在Makefile/CMake中显式指定编译器路径
# Makefile:
CC := C:/tools/arm-gcc-10/bin/arm-none-eabi-gcc

# CMake toolchain file:
set(CMAKE_C_COMPILER "C:/tools/arm-gcc-10/bin/arm-none-eabi-gcc.exe")
⚠️ 注意：不同版本的ARM GCC编译出来的库不兼容！如果项目依赖了预编译库（.a文件），必须使用同一版本编译。遇到链接报"undefined reference"时，优先怀疑版本不匹配。
WSL2嵌入式开发环境进阶
Windows Subsystem for Linux (WSL2) 让你在Windows里跑真正的Linux环境。有些开源嵌入式工具在Linux下更好用（比如一些脚本工具、DFU烧录工具等）。
# 安装WSL2（管理员PowerShell运行一次）
wsl --install -d Ubuntu-24.04

# WSL2里的嵌入式工具链安装
sudo apt update
sudo apt install gcc-arm-none-eabi   # ARM GCC编译器
sudo apt install cmake                # CMake
sudo apt install make                 # GNU Make
sudo apt install python3-pip          # Python包管理
pip3 install pyserial stlink-tools   # 串口和ST-Link工具

# USB设备直通（在WSL2里操作物理硬件）
# 需要安装 usbipd-win（Windows侧）
# winget install usbipd
# usbipd wsl list        # 列出USB设备
# usbipd wsl attach -b 1-2  # 把设备附到WSL2
💡 何时用WSL2：如果你主要在Windows工作，WSL2作补充就够了——大部分嵌入式工具Windows版都可用。只有在需要某些Linux专属工具链或脚本的时候再切过去。日常开发用Windows原生工具效率更高。
包管理器一键装环境进阶
厌倦了手动下载安装？用Windows包管理器一行命令搞定全部工具：
# 检查是否已安装 winget（Win11自带）
winget --version

# 一键安装嵌入式全套工具（管理员终端运行）
winget install -e --id Arm.GnuArmEmbeddedToolchain   # ARM GCC
winget install -e --id Kitware.CMake                   # CMake
winget install -e --id Ezwinports.Make                 # GNU Make
winget install -e --id GnuWin32.Make                   # 替代版
winget install -e --id Python.Python.3.14              # Python
winget install -e --id PuTTY.PuTTY                     # PuTTY
winget install -e --id Microsoft.VisualStudioCode      # VS Code
winget install -e --id Git.Git                         # Git

# 装完验证（所有packages列表）
winget list
⚠️ winget版本可能滞后，ARM GCC建议还是手动去官网下载最新版。但对于CMake/Python/Git等工具，winget的版本足够用。

settings.json：让 IntelliSense 认识 ARM GCC

新手最常遇到的 VS Code 问题：打开 STM32 项目后满屏红色波浪线 —— #include "stm32f1xx.h" 报"无法打开源文件"。这是因为 IntelliSense 的默认编译器是 x86 gcc，不知道 ARM 头文件在哪。

📝 .vscode/settings.json — 嵌入式项目工作区配置 
{
  // === IntelliSense 配置（解决红色波浪线） ===
  "C_Cpp.default.compilerPath": "C:/Program Files (x86)/Arm GNU Toolchain arm-none-eabi/14.2/bin/arm-none-eabi-gcc.exe",
  "C_Cpp.default.compilerArgs": [
    "-mcpu=cortex-m3",     // 你的MCU核心
    "-mthumb",             // Thumb指令集
    "-specs=nosys.specs"   // 无系统调用
  ],
  "C_Cpp.default.includePath": [
    "${workspaceFolder}/**",
    "${workspaceFolder}/bsp/**",
    "${workspaceFolder}/lib/CMSIS/Include"  // CMSIS头文件路径
  ],
  // 让IntelliSense知道ARM GCC预定义宏
  "C_Cpp.default.defines": [
    "STM32F103xB",  // 你的MCU型号宏
    "USE_HAL_DRIVER"
  ],
  "C_Cpp.default.intelliSenseMode": "gcc-arm",
  "C_Cpp.default.cStandard": "c11",
  "C_Cpp.default.cppStandard": "c++17",

  // === 文件关联 ===
  "files.associations": {
    "*.s": "arm",       // 启动文件用ARM汇编语法
    "*.ld": "linkerscript"  // 链接脚本有专用语法高亮
  },

  // === 排除编译产物（加快搜索） ===
  "search.exclude": {
    "build/**": true,
    "**/*.o": true,
    "**/*.elf": true,
    "**/*.bin": true
  },

  // === Cortex-Debug 使用arm-none-eabi-gdb ===
  "cortex-debug.armToolchainPath": "C:/Program Files (x86)/Arm GNU Toolchain arm-none-eabi/14.2/bin",
  "cortex-debug.openocdPath": "C:/Program Files/OpenOCD/bin/openocd.exe"
}
⚡ 踩坑：改完 settings.json 红色波浪线还在？
不要只按 Ctrl+S！做完以下操作：
① 确认路径里没有中文字符（ARM GCC装在Program Files没问题，但不要装在"我的文档"下）
② 按 Ctrl+Shift+P → 输入 C/C++: Reset IntelliSense Database → 回车
③ 按 Ctrl+Shift+P → 输入 Developer: Reload Window → 回车
④ 在 .c 文件里看一下状态栏右下角显示的是不是 "Win32" 而不是 "Linux" 或 "Mac"——ARM GCC 在 Windows 上 IntelliSense 模式应显示为 Win32
如果第四步显示了Linux，去 settings.json 把 "intelliSenseMode" 改为 "gcc-arm"

tasks.json：一键编译快捷键

虽然可以在终端手动打 cmake --build build，但配置 tasks.json 后可以按 Ctrl+Shift+B 直接编译，编译错误自动跳转到对应行。

📝 .vscode/tasks.json — 编译任务配置 
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "🔨 编译 (CMake + Make)",
      "type": "shell",
      "command": "cmake",
      "args": ["--build", "build", "-j8"],
      "options": { "cwd": "${workspaceFolder}" },
      "group": { "kind": "build", "isDefault": true },
      "problemMatcher": "$gcc",  // 自动解析编译错误
      "presentation": {
        "reveal": "always",       // 编译时自动显示终端
        "panel": "dedicated"
      }
    },
    {
      "label": "🧹 清理",
      "type": "shell",
      "command": "cmake",
      "args": ["--build", "build", "--target", "clean"],
      "options": { "cwd": "${workspaceFolder}" },
      "problemMatcher": []
    },
    {
      "label": "📦 重新配置CMake",
      "type": "shell",
      "command": "cmake",
      "args": [
        "-B", "build",
        "-G", "Ninja",
        "-DCMAKE_TOOLCHAIN_FILE=toolchain-arm.cmake",
        "-DCMAKE_BUILD_TYPE=Debug"
      ],
      "options": { "cwd": "${workspaceFolder}" },
      "problemMatcher": []
    }
  ]
}

关键配置说明：
• "problemMatcher": "$gcc" — VS Code内置的GCC错误解析器，编译输出中的 main.c:42:5: error: 会被自动识别为可点击的错误链接
• "isDefault": true — 设为默认构建任务，Ctrl+Shift+B 直接触发，不需要选任务
• "cwd" — 设为 ${workspaceFolder}，确保相对路径正确

launch.json：断点调试配置

Cortex-Debug 扩展配合 launch.json 实现 VS Code 内的断点调试：设断点、单步执行、查看变量、看寄存器，和 Keil/IAR 一样好用。

📝 .vscode/launch.json — ST-Link V2 调试配置 
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "🐞 STM32F103 (ST-Link + OpenOCD)",
      "cwd": "${workspaceFolder}",
      "type": "cortex-debug",
      "request": "launch",
      "servertype": "openocd",
      "device": "STM32F103C8",                // MCU型号
      "executable": "build/stm32-blink.elf",   // 要调试的ELF
      "configFiles": [
        "interface/stlink.cfg",                 // 调试器接口
        "target/stm32f1x.cfg"                   // 目标MCU
      ],
      "svdFile": "STM32F103.svd",               // 外设寄存器定义文件
      "runToEntryPoint": "main",                // 烧录后自动停在main
      "preLaunchTask": "🔨 编译 (CMake + Make)", // 调试前先编译
      "postDebugTask": ""
    },
    {
      "name": "🐞 STM32F103 (J-Link)",
      "type": "cortex-debug",
      "request": "launch",
      "servertype": "jlink",
      "device": "STM32F103C8",
      "executable": "build/stm32-blink.elf",
      "svdFile": "STM32F103.svd",
      "runToEntryPoint": "main",
      "interface": "swd"                       // J-Link用SWD接口
    }
  ]
}
💡 SVD文件是什么？SVD（System View Description）是ARM定义的XML格式文件，描述了MCU所有外设寄存器的地址、位域和含义。有了SVD文件，Cortex-Debug可以在调试时显示 GPIOA->ODR 每个位的实时值。STM32全系列的SVD文件在 github.com/posborne/cmsis-svd 可以下载。下载后放到项目根目录，在 launch.json 里指定路径即可。

🔌 调试探针选型与实战接线

调试探针（Debug Probe）是把你的PC和MCU连接起来的桥梁。选择不对，调试体验会非常痛苦。以下是主流选择：

探针价格支持MCU速度优点缺点
ST-Link V2 仿版¥10-15STM32全系列4MHz SWD便宜、够用、资料多只支持STM32；仿版偶尔掉固件
ST-Link V3 MINI¥70-90STM32全系列24MHz SWD速度快、有虚拟串口、正品质量好贵、淘宝不好买正品

OpenOCD 配置文件详解

OpenOCD 是连接调试探针和 GDB 的中间层，在 VS Code + Cortex-Debug 方案中必不可少。

📝 openocd-stlink.cfg — 为STM32F103C8T6定制 
# ===== 调试器接口配置 =====
source [find interface/stlink.cfg]    # 使用ST-Link V2
# 也可以换成:
# source [find interface/cmsis-dap.cfg]  # DAPLink
# source [find interface/jlink.cfg]      # J-Link

# ===== 传输协议 =====
transport select hla_swd     # ST-Link用HLA+SWD模式
# transport select swd        # DAPLink用这个

# ===== 目标芯片配置 =====
source [find target/stm32f1x.cfg]   # STM32F103

# ===== 连接参数 =====
adapter speed 4000      # SWD时钟 4MHz (稳定值，不要调更高)

# ===== 复位配置 =====
# reset_config srst_only      # 仅复位芯片
reset_config srst_nogate     # 推荐：连接时自动复位
# reset_config none           # 不自动复位(调试低功耗唤醒时用)

# ===== Flash保护配置 =====
# 如果不加这行，STM32F103的option bytes保护可能导致擦除失败
# 请确认 flash bank 在 stm32f1x.cfg 中已声明

SWD接线：4根线就够了

不管是ST-Link还是DAPLink，SWD只需要接4根线（标准20pin JTAG口里大部分脚都是空的）：

ST-Link引脚颜色（常规）接STM32F103引脚说明
GND黑色GND共地
SWCLK黄色PA14 (SWCLK)时钟线
SWDIO绿色PA13 (SWDIO)数据线
3.3V红色3.3V目标板供电（可选：可让ST-Link给板子供电，但最大100mA）
⚠️ 接线禁忌：
• 不要把ST-Link的5V接到3.3V引脚！ 虽然ST-Link上有5V引脚，但那是给5V目标板用的（如STM32F407 Discovery）。蓝色小板是3.3V的。
• 如果目标板已有独立供电（USB线），只接 GND + SWCLK + SWDIO 三根线，不要接3.3V！两个电源打架可能烧坏ST-Link。
• SWCLK 和 SWDIO 不需要上拉电阻，ST-Link内部已有。

🐛 环境搭建踩坑实录

以下每个坑都是真实踩过的，按发生频率排序：

🔍 坑1：arm-none-eabi-gcc 命令找不到，但确实装了

❌ 现象：安装完 ARM GCC 后，在 PowerShell 里输入 arm-none-eabi-gcc --version 提示 "无法识别为 cmdlet..."
✅ 根因：ARM GCC 安装器默认不勾选"Add to PATH"选项！（这个设计非常坑）
✅ 解决：
① 重新运行安装器，在最后一个界面必须手动勾选 "Add path to environment variable"
② 或者手动添加：
[Environment]::SetEnvironmentVariable('Path', 'C:\Program Files (x86)\Arm GNU Toolchain arm-none-eabi\14.2\bin;' + [Environment]::GetEnvironmentVariable('Path','User'), 'User')
③ 关键一步：注销或重启，或至少关闭并重新打开所有终端窗口 —— PATH 变更不会实时传播到已打开的进程
④ 终极排错：用 Get-Command arm-none-eabi-gcc 确认系统能找到它，如果找不到，用 echo $env:Path -split ';' 检查PATH内容
🔍 坑2：OpenOCD 提示 "Error: libusb_open() failed with LIBUSB_ERROR_NOT_SUPPORTED"

❌ 现象：VS Code 调试启动时，OpenOCD 输出一片红字，核心是 LIBUSB_ERROR_NOT_SUPPORTED
✅ 根因：Windows 没有给 ST-Link 装 WinUSB 驱动，默认装的是 ST 官方 ST-Link 驱动或 Windows 自动装的 HID 驱动，而 OpenOCD 需要 WinUSB（通过 Zadig 安装）
✅ 解决：
① 下载 Zadig（USB驱动替换工具）
② 插上 ST-Link，打开 Zadig → Options → List All Devices
③ 下拉列表里找到 "ST-Link Debug"（或 "STM32 STLink"）
④ 在 Driver 那一栏，选择 WinUSB（不是 libusb-win32，不是 libusbK）
⑤ 点击 Replace Driver，等10秒完成
⑥ 再次尝试 OpenOCD 连接
⚠️ 注意：装了 WinUSB 后，STM32CubeProgrammer 的 ST-Link 模式可能无法识别。需要时候再换回原驱动（设备管理器→ST-Link→更新驱动→从列表选取→ST官方驱动）
🔍 坑3：CMake配置时找不到 Python3

❌ 现象：CMake 配置阶段报错 Could NOT find Python3 (missing: Python3_EXECUTABLE)
✅ 根因：Python 安装时没有勾选 "Add Python to PATH"，或者安装后没有重启终端
✅ 解决：
① 在终端输入 python --version（或 python3 --version）—— 如果也报错，说明 Python 确实没在 PATH 里
② 重新安装 Python：从 python.org 下载，安装界面必须勾选底部的 "Add Python to PATH"
③ 或者在 CMake 命令中手动指定：
cmake -DPython3_EXECUTABLE="C:/Users/你的用户名/AppData/Local/Programs/Python/Python314/python.exe" ..
④ 如果 Python 在 PATH 里但 CMake 仍然找不到，检查是否是 Microsoft Store 版 Python（app execution alias）—— 在 "应用执行别名" 设置里关掉 Python 的别名，装官网版
🔍 坑4：VS Code 终端里 make 正常，Ctrl+Shift+B 编译失败

❌ 现象：在 VS Code 内置终端手动 cmake --build build 成功，但按 Ctrl+Shift+B 报 "make is not recognized"
✅ 根因：VS Code 的 tasks.json 使用了一个不同的环境——它不读取你在 PowerShell Profile 里修改的 PATH
✅ 解决：
① 在 tasks.json 里给每个 task 添加 "options": { "env": { "Path": "..." } } 显式设置 PATH
② 更彻底的方案：把 make 的路径加到系统环境变量（而非用户环境变量）：
Win+R → sysdm.cpl → 高级 → 环境变量 → 系统变量 → Path → 添加 C:\Program Files (x86)\GnuWin32\bin
③ 重启 VS Code（完全关闭再打开，不是 Reload Window）
🔍 坑5：STM32F103蓝色小板通过ST-Link能烧录，但复位后程序不运行

❌ 现象：烧录成功（Verify OK），但板子上的LED不亮，跟没烧一样
✅ 根因：BOOT0 跳线帽位置错误。蓝色小板上有两组跳线：BOOT0 和 BOOT1。如果 BOOT0=1，芯片从系统存储器（System Memory）启动，不会运行你烧进去的程序
✅ 解决：
① 把 BOOT0 的跳线帽拔下来，接到标记为 0 的一侧（通常是下方位置）
② 把 BOOT1 也接到 0 侧
③ 按一下板上的 RESET 按钮，或者重新上电
④ 核心原则：正常运行时 BOOT0=0，BOOT1=0；烧录时 BOOT0=1(串口烧录) 或者 0(ST-Link烧录)\n
⑤ 另一个可能：链接脚本写错了，程序入口地址不在 0x08000000。检查 .ld 文件里 FLASH (rx) : ORIGIN = 0x08000000

🐳 Docker 容器化开发环境（可选进阶）

如果你在多台电脑之间切换（公司台式机 + 家里笔记本），或者想让新人快速上手不用装一堆工具，Docker 容器化的开发环境是个好选择。

📝 Dockerfile — ARM GCC 交叉编译环境 
# Dockerfile-embedded
FROM ubuntu:24.04

# 避免安装时区选择交互
ENV DEBIAN_FRONTEND=noninteractive

# 安装构建工具链
RUN apt-get update && apt-get install -y --no-install-recommends \
    # ARM GCC 交叉编译器
    gcc-arm-none-eabi=15:14.2* \
    binutils-arm-none-eabi \
    # 构建系统
    cmake \
    ninja-build \
    make \
    # Python (构建脚本用)
    python3 \
    python3-pip \
    # 工具
    git \
    curl \
    # 清理APT缓存减小镜像体积
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*

# 验证工具链是否可用
RUN arm-none-eabi-gcc --version && cmake --version

# 设置工作目录
WORKDIR /workspace

# 默认命令：bash
CMD ["/bin/bash"]
# 使用方式（在项目根目录执行）
# 1. 构建镜像
docker build -t arm-gcc-dev -f Dockerfile-embedded .

# 2. 启动容器（挂载当前项目目录）
docker run -it --rm -v ${PWD}:/workspace arm-gcc-dev

# 3. 在容器内编译
mkdir build && cd build
cmake -DCMAKE_TOOLCHAIN_FILE=../toolchain-arm.cmake ..
cmake --build . -j$(nproc)

# 4. 烧录（需要 USB 直通到容器，需额外配置 --device 参数）
# docker run -it --rm -v ${PWD}:/workspace --device=/dev/ttyUSB0 arm-gcc-dev
💡 Docker 的局限性：容器内无法直接访问 Windows 的 USB 设备（如 ST-Link），所以烧录和调试需要在宿主机上做。但编译完全可以在容器里跑，CI/CD 流水线也推荐用这种方式保证环境一致性。如果确实需要容器内烧录，Windows 上需要用 WSL2 + usbipd 做USB转发，比较折腾。

🐧 WSL2 + 嵌入式开发（Windows 上的 Linux 体验）

WSL2 是 Windows 上运行 Linux 的最佳方式，对于习惯 Linux 命令行的开发者来说体验极佳。关键是可以直接用 Linux 版的 ARM GCC + OpenOCD。

# 1. 安装 WSL2（需要管理员 PowerShell）
wsl --install -d Ubuntu-24.04
# 首次启动设置用户名/密码

# 2. 在 WSL2 内安装嵌入式工具链（一条命令搞定）
sudo apt update && sudo apt install -y \
    gcc-arm-none-eabi \     # ARM GCC (Ubuntu 24.04自带14.2)
    binutils-arm-none-eabi \  # binutils
    cmake \                  # CMake
    ninja-build \            # Ninja
    make \                   # GNU Make
    gdb-multiarch \          # GDB支持多架构
    openocd \                # OpenOCD调试服务器
    python3 \                # Python
    python3-serial           # pyserial

# 3. 验证安装
arm-none-eabi-gcc --version
openocd --version

# 4. USB设备直通到WSL2（编译在WSL2，烧录也需要在WSL2）
# 安装 usbipd（在 Windows 管理员 PowerShell 中）：
winget install dorssel.usbipd-win

# 连接 ST-Link 到 WSL2：
usbipd list                     # 找到 ST-Link 的 BUSID
usbipd bind --busid      # 绑定
usbipd attach --wsl --busid   # 挂载到WSL2

# 在 WSL2 内验证设备：
lsusb                           # 应看到 "STMicroelectronics ST-LINK"
openocd -f interface/stlink.cfg -f target/stm32f1x.cfg  # 测试连接

# 5. VS Code 的 WSL2 集成（推荐工作流）
# 在 VS Code 中安装 "WSL" 扩展
# 在 WSL2 终端里 cd 到项目目录 → code .
# VS Code 会自动在 WSL2 中运行，IntelliSense 用 WSL2 里的 ARM GCC
💡 WSL2 vs 原生 Windows 开发对比：
• 优势：Linux原生工具链（apt一条命令全装）、文件系统性能好（ext4）、与树莓派/服务器开发体验一致、CI/CD直接复用Docker
• 劣势：USB直通配置稍繁琐（usbipd偶尔需要重新attach）、GUI调试不如原生Windows流畅
• 建议：如果你未来要开发Linux嵌入式（树莓派/Buildroot/Yocto），WSL2是必选项。纯MCU开发选原生Windows或WSL2都可以

🔌 USB转串口驱动深度排错

串口是嵌入式开发的命脉，但 Windows 的串口驱动问题是新手第一道坎。

芯片识别名驱动来源常见坑

🔍 CH340 在 Win11 上装不上驱动（驱动签名错误）

❌ 现象：设备管理器里 CH340 显示黄色感叹号，属性显示"驱动程序签名强制"错误
✅ 解决：
① 按住 Shift → 重启 → 疑难解答 → 高级选项 → 启动设置 → 重启 → 按 7（禁用驱动程序强制签名），这是临时方案
② 去 wch.cn 下载最新版驱动(2024年后的)，新版驱动有微软签名
③ 设备管理器 → 右键CH340 → 卸载设备（勾选"删除此设备的驱动程序软件"）→ 重新安装新版驱动
④ 终极方案：换 CP2102 模块（¥8），Silicon Labs 的驱动签名从未出过问题
🔍 串口能识别但 PuTTY/serial monitor 打不开（"Access Denied"）

❌ 现象：设备管理器里串口正常，但任何串口工具打开都报"Access Denied"或"端口被占用"
✅ 根因：另一个程序正在占用串口（前一个PuTTY没关、Arduino IDE打开了串口监视器、ST-Link Utility占用了虚拟串口）
✅ 解决：
① 关闭所有串口工具（包括系统托盘里的）
② 设备管理器 → 右键串口 → 禁用 → 重新启用
③ 查谁占用了串口（PowerShell管理员）：
Get-Process | Where-Object { $_.Modules.FileName -like '*Serial*' }
④ 如果是 Windows 10/11 的"快速启动"导致，重启电脑（不是关机再开机——Win10/11的关机是"混合休眠"！）

📦 Vcpkg：C/C++ 包管理器用于嵌入式工具

Vcpkg 是微软维护的 C/C++ 包管理器，虽然主要用于桌面开发，但可以方便地安装嵌入式辅助工具：

# 安装 vcpkg
git clone https://github.com/Microsoft/vcpkg.git C:\dev\vcpkg
cd C:\dev\vcpkg
.\bootstrap-vcpkg.bat

# 安装嵌入式相关工具
.\vcpkg install protobuf:x64-windows       # 序列化库（IoT通信用）
.\vcpkg install nlohmann-json:x64-windows   # JSON解析（解析设备配置）
.\vcpkg install spdlog:x64-windows          # 日志库（仿真/测试用）
.\vcpkg install benchmark:x64-windows       # 性能测试（算法验证）

# 注意：vcpkg 目前不直接支持 arm-none-eabi 的交叉编译包
# 嵌入式MCU库还是用 git submodule / FetchContent 管理

📋 知识点回顾

🎯 本章核心要点：

1. 工具链必装6件套：
• ARM GCC 14.2 — 交叉编译器（.c → .elf）
• CMake 4.3 — 现代构建系统
• Make 3.81 — 经典构建工具
• PuTTY — 串口终端
• VS Code + 插件 — 代码编辑与调试
• Python + pyserial — 脚本工具与串口

2. PATH环境变量：
• 把工具的bin目录加入PATH才能在任意目录调用
• 修改PATH后必须重新打开命令行窗口才生效
• 用 check_env.ps1 脚本一键验证所有工具

3. 入门硬件三件套（约¥25）：
• STM32F103C8T6 最小系统板（¥5-10）
• ST-Link V2 烧录器（¥10-15）
• CH340G USB转TTL 模块（¥5）

4. 版本管理：
• 装Git，配置 user.name 和 user.email
• 创建 .gitignore，只提交源码，不提交编译产物
• git status / git add / git commit 是每日三连

5. 排错思路：
• 命令找不到 → 检查PATH → 重新打开终端
• CMake找不到编译器 → 指定 toolchain file
• 串口不识别 → 装驱动（CH340/ST-Link驱动）
