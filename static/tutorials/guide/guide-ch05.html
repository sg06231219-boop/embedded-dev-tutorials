第5章：烧录与调试

📖 小白完全指南 · 5/14 | 阅读时间约3分钟

编译生成.bin/.hex文件后，需要用烧录器把它写进单片机的Flash里。调试则是让程序在芯片里暂停、单步执行、查看变量，像"慢放"一样找Bug。

💡 类比：烧录=把程序刻到芯片的DNA里（写死在Flash中），调试=给程序做体检（停住→检查→继续→再检查）。

调试接口：JTAG vs SWD

ARM Cortex-M芯片支持两种调试接口。了解区别有助于选对调试器：

特性JTAGSWD
引脚数5根（TMS/TCK/TDI/TDO/TRST）2根（SWDIO/SWCLK）
速度中等快（专用协议）
占用引脚多（5个GPIO）少（2个GPIO）
适用场景老芯片/FPGA/多核Cortex-M单片机（最常用）
推荐几乎不用✅ 默认选择
💡 实际经验：STM32最小系统板上的4针排针（3V3/GND/SWDIO/SWCLK）就是SWD接口。日常调试只需接这4根就够了——比JTAG省3根线，速度还更快。ST-Link/V2和DAP-Link都支持SWD。
调试器选购指南省钱建议
市场上调试器琳琅满目，选对能省不少钱和麻烦：
调试器价格支持芯片速度推荐度
ST-Link/V2 仿¥8-15STM32全系列4MHz SWD⭐⭐⭐⭐⭐ 入门首选
DAP-Link 仿¥15-30所有Cortex-M10MHz SWD⭐⭐⭐⭐ 更通用
J-Link EDU¥400+所有ARM核50MHz SWD⭐⭐⭐ 专业首选

⚠️ 注意：¥8的仿ST-Link品质参差不齐。有些不能给目标板供电（需要额外给板子供电），有些SWD速度不稳定。买之前看评论，优先选有3.3V供电输出的。

ST-Link 烧录（st-flash）

⚠️ 前提条件：需要安装stlink工具。下载地址：github.com/stlink-org/stlink/releases（同GitHub，可能被墙）。替代方案：使用STM32CubeIDE自带的st-link驱动。

# github.com/stlink-org/stlink/releases
# 下载后解压，把bin目录加PATH

# 烧录BIN文件
st-flash write blink.bin 0x08000000

# 烧录HEX文件
st-flash --format ihex write blink.hex

# 擦除整片Flash
st-flash erase

# 读取Flash内容
st-flash read flash_dump.bin 0x08000000 0x10000

串口ISP烧录（零硬件成本！）

💡 没有ST-Link？用USB转TTL也能烧！STM32内置Bootloader支持串口下载，只需要CH340模块（¥5）。速度比ST-Link慢，但零额外成本。
# 方式1：FlyMcu（最简单的Windows串口烧录工具）
# 下载：www.flymcu.com（免费，中文界面）
# 使用：
# 1. BOOT0跳线接1（或按住BOOT0键上电）
# 2. 打开FlyMcu → 选择COM口和hex文件
# 3. 点击"开始编程"
# 4. 烧录完成后BOOT0接回0，按复位键运行

# 方式2：stm32flash（命令行工具）
# 下载：github.com/ARMin stm32flash
# 使用：
stm32flash -w blink.hex -v -g 0x0 COM3
# -w 写入  -v 校验  -g 0x0 写完从0地址运行  COM3 你的串口
⚠️ 串口烧录要点：
• BOOT0必须接高电平(1)才能进入下载模式
• 烧录完成后BOOT0接回低电平(0)，按复位键才能运行程序
• 有些最小系统板有BOOT0跳线帽，方便切换
• 每次烧录都要切换BOOT0，不如ST-Link方便，但学习阶段完全够用

GDB调试

⚠️ 说明：GDB调试需要硬件调试器（ST-Link/DAP-Link/J-Link）。如果你暂时没有调试器，可以先跳过GDB部分，用printf调试（第11章详解）。下面提供两种调试方案：方案A用OpenOCD（免费开源，DAP-Link/ST-Link通用），方案B用st-util（ST-Link专用，更简单）。

方案A：OpenOCD + GDB（推荐，通用性更好）

# 看到 "Info : stm32f1x.cpu: hardware has 6 breakpoints" 就说明连接成功

# 终端2：启动GDB客户端

# GDB常用命令（两方案通用）
(gdb) target remote :3333         # 连接OpenOCD（注意端口3333）
(gdb) monitor reset halt          # 复位并停住
(gdb) load                         # 烧录程序到芯片
(gdb) break main                   # 在main函数设断点
(gdb) continue                     # 继续运行
(gdb) step                         # 单步（进入函数）
(gdb) next                         # 单步（跳过函数）
(gdb) print variable               # 打印变量值
(gdb) watch variable               # 监视变量变化
(gdb) backtrace                    # 查看调用栈
(gdb) info registers               # 查看CPU寄存器
(gdb) x/10x 0x20000000            # 查看内存（16进制）
(gdb) delete                       # 删除所有断点
(gdb) quit                         # 退出

GDB高阶技巧

断点类型详解进阶
GDB支持多种断点类型，不同场景用不同断点能极大提高调试效率：
# === 1. 普通断点（最基础）===
(gdb) break main           # 在main函数入口
(gdb) break main.c:42      # 在main.c的第42行
(gdb) break HAL_UART_Transmit  # 在函数入口

# === 2. 条件断点（只在条件满足时停下）===
(gdb) break main.c:100 if count > 50
(gdb) break blink.c:30 if error_code != 0
# 示例：只在温度>40°C时停在read_temp函数
(gdb) break read_temp if raw_value > 500

# === 3. 临时断点（命中一次后自动删除）===
(gdb) tbreak init_hardware
# 适合"我只想看初始化跑一次"的场景

# === 4. 硬件断点（不修改Flash内容）===
(gdb) hbreak *(0x08000400)
# 硬件断点不修改Flash，适合调试Flash中的代码
# 注意：Cortex-M3只有6个硬件断点！

# === 5. 观察点（变量值改变时停下）===
(gdb) watch global_error_count
(gdb) watch *0x20000100     # 监视内存地址
(gdb) rwatch *0x20000100    # 读监视（地址被读时停下）
(gdb) awatch *0x20000100    # 读写监视（读写都停下）
# 示例：找出谁改了我的变量
(gdb) watch sensor_data.raw_value
# 任何修改sensor_data.raw_value的代码都会触发断点

# === 6. 捕获点（捕获系统事件）===
(gdb) catch fork     # 捕获fork调用
(gdb) catch throw    # 捕获C++异常
# 嵌入式GDB通常不支持catch（Linux应用调试才常用）
⚠️ 硬件断点限制：Cortex-M系列只有4-8个硬件断点（STM32F103是6个）。当你设置第7个断点时GDB会报错。解决方法：① 删掉不需要的断点 ② 用软件断点（改Flash，但Flash有擦写寿命限制）③ 混合使用硬件断点和观察点。
GDB脚本化调试实用技巧
把常用调试命令写入.gdbinit文件，每次调试自动执行，省去重复输入：

Bootloader：让产品能远程升级

Bootloader概念进阶
Bootloader是芯片上电后第一个运行的程序，它决定是跳转到APP还是进入升级模式。有了Bootloader，产品出厂后可以通过串口/WiFi/蓝牙升级固件，不用拆机重烧：
// Bootloader的典型判断逻辑
int main(void) {
    HAL_Init();
    SystemClock_Config();
    
    // 判断是否进入升级模式
    if (check_upgrade_pin() || check_upgrade_flag()) {
        // 升级模式：等待接收新固件
        enter_upgrade_mode();  // 通过串口/YModem接收固件
    } else {
        // 正常模式：跳转到APP
        jump_to_app(APP_START_ADDR);  // 如 0x08004000
    }
}

// 跳转到APP的函数
void jump_to_app(uint32_t app_addr) {
    // 1. 检查栈顶地址是否合法（在RAM范围内）
    uint32_t app_stack = *(uint32_t*)app_addr;
    if ((app_stack & 0x2FFE0000) != 0x20000000) {
        return;  // 栈地址不在SRAM区，APP无效
    }
    
    // 2. 关闭所有中断
    __disable_irq();
    
    // 3. 重新设置向量表
    SCB->VTOR = app_addr;
    
    // 4. 设置栈指针并跳转（内联汇编）
    __set_MSP(app_stack);
    uint32_t app_entry = *(uint32_t*)(app_addr + 4);
    void (*app_reset)(void) = (void(*)())app_entry;
    app_reset();  // 永远不会返回
}

// APP端需要做的：
// 1. 链接脚本中Flash起始地址改为APP_START_ADDR
// 2. main开头痛设置VTOR: SCB->VTOR = APP_START_ADDR;
// 3. 中断向量偏移也要相应调整
⚠️ 关键概念：Flash分区示例——
• 0x08000000 - 0x08003FFF：Bootloader区（16KB）
• 0x08004000 - 0x0800FFFF：APP区（48KB）
• 0x08010000 - 0x0801FFFF：OTA下载区/配置区
注意：APP的链接脚本必须把FLASH起始地址改为0x08004000，否则向量表偏移错误会导致所有中断失效。

🔍 常见问题：烧录失败合集
1. "No ST-LINK detected" / "Cannot connect to target"
• 检查USB线——有些线只能充电不能传数据（换根线试试！）
• 检查SWD接线：SWDIO/SWCLK/GND至少3根
• 检查目标板是否已供电（仿ST-Link通常不给目标板供电！）
• Windows设备管理器里看ST-Link是否被识别（有黄色感叹号=驱动未装）
• 尝试拔插USB，更换USB口
2. "Flash download failed" / "Cannot program"
• 芯片是否被读保护了？用 st-flash --read-unprotect 或 openocd -c "stm32f1x unlock 0" 解锁
• Flash是否被写保护了？检查选项字节（Option Bytes）
• BOOT0引脚是否接错？（BOOT0=1会进入系统Bootloader，SWD无法连接）
3. "Target voltage detection failed"
• 目标板没电！检查目标板的3.3V供电是否正常
• 如果用仿ST-Link，确保它输出3.3V给目标板（或单独供电）
4. 烧录成功但程序不运行
• BOOT0是否已接回0？（串口ISP烧录后必须切回BOOT0=0）
• 按一下复位键试试
• 检查启动文件/链接脚本——向量表是否正确放置？
• 检查SystemInit()是否正确配置了时钟

SWO/SWV：串行线输出与实时跟踪

SWO（Serial Wire Output）概念进阶性能分析利器
SWD只用了2根线（SWDIO+SWCLK），但ARM Cortex-M还有第三根线：SWO（Serial Wire Output）。通过这根线，芯片可以实时输出调试信息、程序计数器采样、数据跟踪等，无需占用UART。配合SWV（Serial Wire Viewer），可以在不停止CPU的情况下分析程序行为。
SWO功能说明对CPU影响
ITM printf通过SWO输出printf信息（比串口快100倍）极低（硬件FIFO自动发送）
PC采样周期性采样程序计数器，生成代码耗时热力图零（纯硬件读取）
数据跟踪实时输出变量值（类似无线watch）低
异常跟踪记录所有中断/异常的进入/退出时间极低
计数器DWT周期计数器、CPI（每指令周期数）零
⚠️ 硬件要求：SWO需要调试器支持（J-Link全系列支持，ST-Link/V2不支持SWO，ST-Link/V3支持，DAP-Link部分支持）。还需要芯片的SWO引脚引出（STM32最小系统板上的SWO通常就是PB3，但需要检查原理图）。
ITM printf：比串口快100倍的调试输出实用技巧
传统串口printf在115200波特率下每字节约86μs，输出一行"Hello World"（11字节+换行）需要约1ms。如果在1KHz定时中断里用printf，CPU会被串口完全占满。ITM printf通过SWO输出，速度可达几MB/s，在中断里使用几乎没有性能损失。
// === ITM printf配置（STM32F103 + J-Link/ST-Link V3）===

// 步骤1：在CubeMX中启用SWO引脚
// Pinout → SYS → Debug → 选择 "Serial Wire" 或 "Trace Asynchronous Sw"
// SWO引脚通常是PB3

// 步骤2：重定向printf到ITM
// 在main.c中添加：
#include <stdio.h>

// ITM发送单个字符（硬件FIFO）
int _write(int fd, char *ptr, int len) {
    for (int i = 0; i < len; i++) {
        ITM_SendChar((*ptr++));  // CMSIS内置函数
    }
    return len;
}

// 步骤3：初始化SWO（在SystemClock_Config之后）
void SWO_Init(uint32_t portMask, uint32_t cpuFreq, uint32_t swoBaud) {
    // 使能ITM和DWT
    CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
    
    // 解锁ITM
    ITM->LAR = 0xC5ACCE55;
    
    // 使能ITM的port 0（printf输出通道）
    ITM->TER |= (1UL << 0);
    
    // 配置TPIU（Trace Port Interface Unit）
    TPI->SPPR = 2;  // 异步模式（SWO）
    TPI->ACPR = (cpuFreq / swoBaud) - 1;  // 波特率分频
    
    // 使能SWO输出
    TPI->FFCR = (1UL << 2);  // 启用格式化
    ITM->TCR |= ITM_TCR_ITMENA_Msk;  // 全局使能ITM
}

// 使用方法：
int main(void) {
    HAL_Init();
    SystemClock_Config();
    SWO_Init(1, 72000000, 2000000);  // port0, 72MHz CPU, 2MHz SWO
    
    while (1) {
        printf("System tick: %lu\\r\\n", HAL_GetTick());
        HAL_Delay(1000);
    }
}

// J-Link SWO Viewer Client → 选择CPU频率和SWO速度 → Start

💡 性能对比实测：在1KHz定时中断中每秒输出1000次printf：串口115200方式→CPU占用率~90%，程序几乎卡死；ITM printf @2MHz→CPU占用率~2%，程序流畅运行。差距达45倍。

RTT（Real-Time Transfer）：J-Link的高速调试通道

SEGGER RTT进阶高性能
RTT（Real-Time Transfer）是SEGGER开发的调试输出技术，不需要SWO引脚、不需要额外UART，仅通过普通的SWD/JTAG接口就能实现高速双向数据传输（速度可达1MB/s以上）。RTT在目标芯片的RAM中开辟一个环形缓冲区，调试器通过SWD直接读写这个缓冲区。

}

⚠️ RTT vs SWO vs 串口printf对比：
• RTT（推荐）：速度快（~1MB/s）、不占引脚、需要J-Link。RAM中开缓冲区，不影响实时性
• SWO/ITM：速度快（~2MB/s）、需要SWO引脚和专用工具。纯硬件输出，CPU开销最低
• 串口printf：速度慢（~11KB/s@115200）、需要TX引脚。实现最简单，但实时性最差
• 选择策略：有J-Link→首选RTT；有SWO→首选ITM；啥都没有→用串口printf

HardFault深度解析：单片机最恐怖的异常

HardFault是什么重要进阶
HardFault是ARM Cortex-M处理器在遇到无法恢复的错误时触发的异常（异常号3）。一旦发生HardFault，程序会跳转到HardFault_Handler，通常的表现是：LED不闪了、串口没输出了、调试器显示"HardFault_Handler"。对于新手来说，遇到HardFault往往意味着"卡住了，不知道怎么办"。
// === HardFault常见原因（排名分先后）===
//
// 1. 空指针/野指针访问（最常见，占60%+）
//    uint8_t *ptr = NULL;  *ptr = 0x55;  // 访问地址0，必HardFault
//    解决办法：解引用前检查指针非NULL
//
// 2. 访问未初始化的外设时钟
//    GPIOA->ODR = 0xFF;  // 忘了__HAL_RCC_GPIOA_CLK_ENABLE()
//    解决办法：操作外设前确保RCC时钟已使能
//
// 3. 栈溢出（Stack Overflow）
//    递归过深 / 局部变量太大 / 中断嵌套太深
//    部分芯片有MPU可以检测，大部分没有
//    解决办法：增大栈空间 / 大数组用static或堆
//
// 4. 非对齐访问（Cortex-M0/M0+最常见）
//    uint32_t *p = (uint32_t*)0x20000001;  // 非4字节对齐
//    uint32_t val = *p;  // Cortex-M0上HardFault!
//    解决办法：memcpy到对齐的变量
//
// 5. 除以零（如果CCR_DIV_0_TRP使能）
//    int result = 5 / 0;  // 触发UsageFault或HardFault
//
// 6. 执行非法指令 / 跳转到非代码区
//    函数指针指向了数据区 / 栈被破坏导致返回地址错误
//
// 7. 中断中调用FreeRTOS API但优先级不正确
//    在优先级高于configMAX_SYSCALL_INTERRUPT_PRIORITY的中断中调了API
//
// 8. Flash等待周期配置错误
//    系统时钟提高了但Flash等待周期没跟上
HardFault现场分析：保存异常帧调试技巧必学
发生HardFault时，Cortex-M会自动压栈8个寄存器（R0-R3, R12, LR, PC, xPSR）。通过分析这些寄存器的值，可以精确定位出错位置。以下是完整的HardFault Handler实现：
// === HardFault Handler（放在stm32f1xx_it.c或main.c）===
// 发生HardFault时会自动调用此函数

// 定义异常栈帧结构（CM3自动压栈的8个寄存器）
typedef struct {
    uint32_t r0;
    uint32_t r1;
    uint32_t r2;
    uint32_t r3;
    uint32_t r12;
    uint32_t lr;      // 异常返回地址
    uint32_t pc;      // 程序计数器（出错位置！）
    uint32_t xpsr;    // 程序状态寄存器
} ExceptionStackFrame;

void HardFault_Handler(void) {
    // 获取栈帧指针
    // MSP（主栈指针）在进入HardFault时会被自动调整
    __asm volatile (
        "TST LR, #4          \\n"   // 检查EXC_RETURN的bit2
        "ITE EQ              \\n"   // 判断使用的是MSP还是PSP
        "MRSEQ R0, MSP       \\n"   // 如果bit2=0，用MSP
        "MRSNE R0, PSP       \\n"   // 如果bit2=1，用PSP
        "B hard_fault_handler_c \\n" // 跳转到C函数
    );
}

// C语言处理函数
void hard_fault_handler_c(ExceptionStackFrame *frame, uint32_t stacked_lr) {
    // 此时调试器可以在这些变量上设断点来查看
    volatile uint32_t fault_pc  = frame->pc;    // ← 出错时的PC地址
    volatile uint32_t fault_lr  = frame->lr;     // 返回地址
    volatile uint32_t fault_r0  = frame->r0;
    volatile uint32_t fault_r1  = frame->r1;
    volatile uint32_t fault_r2  = frame->r2;
    volatile uint32_t fault_r3  = frame->r3;
    volatile uint32_t stacked_pc = (uint32_t)fault_pc;
    
    // 读取CFSR（Configurable Fault Status Register）
    // 这个寄存器告诉你HardFault的具体原因
    volatile uint32_t cfsr = SCB->CFSR;
    volatile uint32_t hfsr = SCB->HFSR;
    volatile uint32_t mmfar = SCB->MMFAR;  // MemManage Fault Address
    volatile uint32_t bfar  = SCB->BFAR;   // BusFault Address
    
    // 在GDB中查看：
    // (gdb) print fault_pc        → 错误发生的地址
    // (gdb) print /x cfsr         → 查看具体错误码
    // (gdb) info line *fault_pc   → 对应源代码哪一行
    
    // 用LED闪烁指示HardFault（最简单的通知方式）
    while (1) {
        // LED快闪3次，停顿1秒，重复
        for (int i = 0; i < 3; i++) {
            HAL_GPIO_TogglePin(LED_GPIO_Port, LED_Pin);
            HAL_Delay(200);
        }
        HAL_Delay(1000);
    }
}

// === 使用GDB定位HardFault ===
// (gdb) break HardFault_Handler
// (gdb) continue
// ...程序停在HardFault_Handler...
// (gdb) print fault_pc
// $1 = (volatile uint32_t) 0x0800034a
// (gdb) info line *0x0800034a
// Line 42 of "main.c" starts at address 0x0800034a
//     → 就是main.c第42行出了问题！
// (gdb) print /x cfsr
// $2 = 0x00008200  → PRECISERR(bit1=0)+IBUSERR(bit8=1)
//    → 指令总线错误：试图从非法地址取指令
// (gdb) x/4i fault_pc - 8
// → 查看出错位置附近的汇编指令
💡 实战技巧：在HardFault_Handler中设一个断点（GDB: break HardFault_Handler），一旦程序进入，立即查看PC寄存器（info registers pc）指向的地址，然后用info line *地址找到源码行。80%的HardFault问题可以在5分钟内定位。

Cortex-M故障寄存器详解（CFSR/HFSR）

// SCB->CFSR 分为三个8位字段：
// [15:8] = UFSR (UsageFault Status Register)
//  [7:0] = BFSR (BusFault Status Register)
// [31:16]= MFSR (MemManage Status Register)

// === 常见CFSR故障码速查 ===
//
// BFSR (BusFault, CFSR[7:0]):
//   bit 7 = BFARVALID: BFAR寄存器包含有效地址
//   bit 4 = STKERR: 压栈时总线错误（栈溢出！）
//   bit 3 = UNSTKERR: 出栈时总线错误
//   bit 2 = IMPRECISERR: 不精确错误（最难查...）
//   bit 1 = PRECISERR: 精确错误（地址在BFAR中）
//   bit 0 = IBUSERR: 指令总线错误
//
// MFSR (MemManage, CFSR[31:16]):
//   bit 7 = MMARVALID: MMFAR包含有效地址
//   bit 5 = MLSPERR: 浮点懒压栈错误
//   bit 4 = MSTKERR: 压栈时MPU访问违规
//   bit 3 = MUNSTKERR: 出栈时MPU访问违规
//   bit 1 = DACCVIOL: 数据访问违规（访问了MPU禁用的区域）
//   bit 0 = IACCVIOL: 指令访问违规
//
// UFSR (UsageFault, CFSR[15:8]):
//   bit 9 = DIVBYZERO: 除以零（需先使能CCR_DIV_0_TRP）
//   bit 8 = UNALIGNED: 非对齐访问
//   bit 3 = NOCP: 无协处理器
//   bit 2 = INVPC: 无效PC加载（跳转到非代码区）
//   bit 1 = INVSTATE: 无效状态
//   bit 0 = UNDEFINSTR: 未定义指令
//
// HFSR (HardFault Status Register):
//   bit 31 = DEBUGEVT: 调试事件触发
//   bit 30 = FORCED: 被强制提升为HardFault
//     → 说明真正的故障是MemManage/BusFault/UsageFault
//     → 应查CFSR找到根因
//   bit 1 = VECTTBL: 向量表读取错误

芯片读保护与选项字节操作

读保护（RDP）与选项字节安全进阶
STM32芯片有读保护机制，可以防止Flash内容被非法读取。但开发过程中不小心启用读保护后，调试器无法连接，很多人以为"芯片烧了"。了解选项字节操作可以救急。
// === STM32选项字节（Option Bytes）结构 ===
// 地址: 0x1FFFF800 (每个芯片可能不同，查参考手册)
//
// RDP (Read Protection Level):
//   0xAA = Level 0, 无保护（默认，正常调试）
//   0xBB = Level 1, 读保护开启（Flash内容不能读，但可调试）
//   0xCC = Level 2, 不可逆保护（连ST官方都解不了！绝对不要用！）
//
// 注意：从Level 1改回Level 0会自动擦除整片Flash！

// === 命令行操作选项字节（OpenOCD）===

# 读取选项字节
openocd -f interface/stlink.cfg -f target/stm32f1x.cfg \\
        -c "init; reset halt; stm32f1x options_read 0; shutdown"

# 解除读保护（会擦除Flash！）
openocd -f interface/stlink.cfg -f target/stm32f1x.cfg \\
        -c "init; reset halt; stm32f1x unlock 0; shutdown"

# 但OpenOCD解保护有时不稳定，用STM32CubeProgrammer更可靠：
# STM32CubeProgrammer CLI (STM32_Programmer_CLI):
STM32_Programmer_CLI -c port=SWD -ob RDP=0xAA

# === ST-Link命令行操作（st-flash）===

# 解除读保护
st-flash --read-unprotect

# 查看选项字节
st-flash --option

# === 代码中修改选项字节（危险操作！）===
// 软件修改选项字节的代码（仅生产环境使用）
void SetReadProtection(void) {
    HAL_FLASH_Unlock();
    HAL_FLASH_OB_Unlock();
    
    FLASH_OBProgramInitTypeDef ob_config = {0};
    HAL_FLASHEx_OBGetConfig(&ob_config);
    
    // ❗警告：设置Level 2后永远无法恢复！
    ob_config.RDPLevel = OB_RDP_LEVEL_1;  // 仅Level 1
    ob_config.OptionType = OPTIONBYTE_WRP | OPTIONBYTE_RDP;
    
    // WRP（写保护）：保护Bootloader区不被意外擦除
    ob_config.WRPState = OB_WRPSTATE_ENABLE;
    ob_config.WRPPage = OB_WRP_PAGES0TO3;  // 保护前4页（每页1KB=4KB）
    
    HAL_FLASHEx_OBProgram(&ob_config);
    HAL_FLASH_OB_Launch();  // 复位后生效
    
    HAL_FLASH_OB_Lock();
    HAL_FLASH_Lock();
}
🚫 绝对不要设Level 2！Level 2是不可逆的——设了之后连ST官方都无法恢复。STM32秒变"一次性芯片"。量产产品用Level 1就够了（阻止Flash读出，但允许通过擦除恢复）。

双Bank OTA升级：安全固件更新方案

双Bank OTA升级原理进阶工业级方案
前面介绍了单Bank的Bootloader（Bootloader+APP）。这种方案有一个致命缺陷：升级过程中断电=变砖。双Bank方案通过在Flash中保存两份APP副本来解决这个问题：
// === 单Bank方案 ===（升级断电→变砖！）
// Flash布局:
//   0x08000000: Bootloader  (16KB)
//   0x08004000: APP         (48KB)  ← 升级时被覆盖
//   
// 升级流程: 接收新固件 → 擦除APP区 → 写入新固件
//             ↑ 这期间断电 → APP区空了 → 变砖

// === 双Bank方案 ===（升级断电→回滚旧版本）
// Flash布局:
//   0x08000000: Bootloader  (16KB)
//   0x08004000: APP_BankA   (48KB)  ← 当前运行版本
//   0x08010000: APP_BankB   (48KB)  ← 下载新固件到这里
//   0x0801C000: Flag区域    (16KB)  ← 存储当前激活哪个Bank
//
// 升级流程:
//   1. Bootloader运行 → 检查Flag → 启动BankA
//   2. APP运行期间收到升级指令
//   3. 下载新固件到BankB（BankA仍在正常运行）
//   4. 下载完成、CRC校验通过
//   5. 写入Flag：下次启动用BankB
//   6. 软件复位 → Bootloader启动BankB
//   ↑ 任何时候断电，Flag没变，旧版本照常运行

// === 升级状态机 ===
typedef enum {
    OTA_IDLE,           // 空闲
    OTA_DOWNLOADING,    // 正在下载新固件
    OTA_VERIFYING,      // 校验中（CRC32）
    OTA_READY,          // 下载完成，等待切换
    OTA_SWITCHING,      // 正在切换Bank
    OTA_FAILED          // 升级失败
} OTA_State_t;

// === 双Bank切换逻辑（Bootloader中）===
typedef struct {
    uint32_t magic;         // 魔数：0xAA55F00D
    uint32_t active_bank;   // 0=BankA, 1=BankB
    uint32_t bank_a_crc;    // BankA固件的CRC32
    uint32_t bank_b_crc;    // BankB固件的CRC32
    uint32_t bank_a_size;   // BankA固件大小
    uint32_t bank_b_size;   // BankB固件大小
} BootFlag_t;

#define BANK_A_ADDR  0x08004000
#define BANK_B_ADDR  0x08010000
#define FLAG_ADDR    0x0801F000  // 最后一页放Flag

void Bootloader_Main(void) {
    BootFlag_t *flag = (BootFlag_t*)FLAG_ADDR;
    
    // 检查Flag有效性
    if (flag->magic != 0xAA55F00D) {
        // Flag无效，使用默认BankA
        goto_boot_bank_a();
        return;
    }
    
    uint32_t app_addr;
    if (flag->active_bank == 0) {
        app_addr = BANK_A_ADDR;
    } else {
        app_addr = BANK_B_ADDR;
    }
    
    // 检查栈顶地址有效性
    uint32_t sp = *(uint32_t*)app_addr;
    if ((sp & 0x2FFE0000) != 0x20000000) {
        // 栈地址无效，尝试另一个Bank
        flag->active_bank ^= 1;  // 切换
        app_addr = (flag->active_bank == 0) ? BANK_A_ADDR : BANK_B_ADDR;
    }
    
    jump_to_app(app_addr);
}
⚠️ 双Bank前提：需要芯片有足够的Flash空间。STM32F103C8T6只有64KB（实际128KB），双Bank每个Bank只有约56KB（减去Bootloader和Flag），对代码量大的项目不够用。考虑用STM32F103RCT6（256KB Flash）或F4系列（512KB+）。

调试器固件更新与常见故障修复

🔧 调试器自身故障排查

1. 仿ST-Link突然不识别（"Unknown USB Device"）
• 现象：插上电脑后设备管理器显示"Unknown USB Device"或"设备描述符请求失败"
• 原因：仿ST-Link的固件损坏（可能是意外断电/静电导致）
• 解决：
- 下载ST-Link固件升级工具：STM32 ST-LINK Utility → ST-LINK菜单 → Firmware Update
- 如果工具检测不到 → 短接ST-Link上的BOOT0跳线（进入DFU模式）再试
- 极少数仿品需要换一颗STM32F103C8T6主控芯片（¥5）

2. J-Link OB提示"The firmware is defective"
• 现象：J-Link Commander显示固件损坏
• 解决：下载J-Link软件包 → 运行JLinkSTM32.exe → 选择正确的固件.bin文件 → 重新烧录
• 注意：J-Link OB的固件是加密的，只能用SEGGER官方工具恢复

3. ST-Link提示"Target voltage detection failed"
• 检查JTAG/SWD接口的第1脚（VCC/VTref）：
- 这个脚不是给目标板供电的，是检测目标板电压用的
- 必须连接到目标板的VCC（3.3V），否则ST-Link拒绝工作
• 仿ST-Link的3.3V引脚可能没有供电能力 → 目标板必须另外供电

4. DAP-Link烧录失败"SWD Communication Failure"
• 尝试降低SWD时钟频率
- OpenOCD: adapter speed 100（降到100KHz）
- pyOCD: --frequency 100000
• 检查杜邦线质量——SWD对接触电阻敏感

• 串口ISP（省钱）：用CH340就能烧，但需要切换BOOT0，不支持调试
• J-Link（专业）：速度快功能全，但¥400+，初学者不需要
• DAP-Link（通用）：开源方案，支持所有Cortex-M芯片

✅ 调试接口：
• SWD（2线）是Cortex-M单片机的标准调试接口
• JTAG（5线）几乎不用，SWD更快更省引脚
• 标准SWD连接：3V3 + GND + SWDIO + SWCLK（4根线）

✅ GDB调试核心技能：
• 断点：break(普通)/tbreak(临时)/hbreak(硬件)/条件断点
• 观察点：watch(写)/rwatch(读)/awatch(读写)，找出谁改了变量
• 控制执行：continue(继续)/step(进入)/next(跳过)/finish(跳出)
• 查看状态：print(变量)/info registers(寄存器)/backtrace(调用栈)/x(内存)
• 脚本化：.gdbinit自动执行常用命令

✅ Bootloader核心概念：
• Bootloader是芯片上电后第一个运行的程序
• 判断条件后决定：跳转APP 还是 进入升级模式
• APP需要重新设置VTOR（向量表偏移）
• 链接脚本中APP的Flash起始地址必须修改

✅ 高级调试技术（新增）：
• SWO/SWV：通过SWO引脚实时输出调试信息，ITM printf比串口快100倍
• RTT：SEGGER的高速调试通道，仅用SWD就能实现~1MB/s双向传输，不占额外引脚
• HardFault诊断：通过保存异常栈帧+读取CFSR/HFSR寄存器精确定位错误地址
• 选项字节：读保护(RDP)的配置与解除（Level 1解除会擦除Flash）

✅ 固件升级方案：
• 单Bank：Bootloader + APP，简单但有断电变砖风险
• 双Bank：同时保存新旧版本，升级断电自动回滚，工业级可靠性
• Level 2读保护是不可逆的——绝对不要设置

⚠️ 常见陷阱（扩展）：
• 仿ST-Link不能给目标板供电——需要额外接3.3V
• 硬件断点只有6个，用完就不能加
• 烧录成功但不运行→检查BOOT0、按复位键
• USB线只能充电不能传数据→换根线！
• HAL_Delay在中断里调用且优先级>SysTick→死锁！
• ST-Link的VTref脚是检测不是供电——目标板必须另外供电
• HardFault 80%由空指针/未初始化外设/栈溢出引起——先查这三样
