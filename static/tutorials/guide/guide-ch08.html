裸机程序（super loop）就像一个人干所有事——烧水、切菜、炒菜、摆盘，只能一件一件来。RTOS让多个任务同时运行（实际是快速切换），就像雇了多个厨师各司其职。

💡 类比：RTOS就像餐厅后厨——任务(Task)=厨师（各干各的活），队列(Queue)=传菜窗口（任务间传递数据），信号量(Semaphore)=叫号器（通知"饭好了"），互斥量(Mutex)=独占灶台（同一时间只有一个厨师用），事件组(Event Group)=多个条件都满足才行动。

🔄 调度器原理

FreeRTOS的核心是调度器(Scheduler)——它负责决定哪个任务现在运行。理解调度器，就理解了RTOS的灵魂。

抢占式调度 vs 合作式调度

模式切换时机实时性适用场景
抢占式高优先级就绪→立即抢占低优先级高绝大多数应用（默认）
合作式任务主动放弃CPU（调vTaskDelay/阻塞API/主动Yield）低极简单场景/资源极度受限
// FreeRTOSConfig.h 中切换调度模式
#define configUSE_PREEMPTION    1   // 1=抢占式  0=合作式
#define configUSE_TIME_SLICING  1   // 1=同优先级时间片轮转
抢占式示例：低优先级TaskA正在运行，高优先级TaskB被中断唤醒→调度器立即暂停TaskA保存现场→运行TaskB→TaskB阻塞后恢复TaskA。TaskA完全不知道被打断过。

时间片轮转

当多个任务同优先级时，FreeRTOS采用时间片轮转：每个任务运行一个tick(默认1ms)，然后自动切换到下一个同优先级任务。

// 三个同优先级任务的时间片轮转
// Task1(优先级1) → 运行1ms → Task2(优先级1) → 运行1ms → Task3(优先级1) → 运行1ms → Task1...
// 如果高优先级Task0(优先级2)就绪，立即打断这个循环

void vTimeSliceDemo(void) {
    // 任务A、B、C都设为优先级1
    xTaskCreate(vTaskA, "A", 256, NULL, 1, NULL);
    xTaskCreate(vTaskB, "B", 256, NULL, 1, NULL);
    xTaskCreate(vTaskC, "C", 256, NULL, 1, NULL);
    // 三个任务会自动轮流执行，每个1ms
}

// 如果不想自动切换（任务独占CPU），vTaskDelay(1)主动让出

📋 任务优先级策略

优先级是RTOS系统的灵魂。分配得当，系统行云流水；分配不当，轻则响应迟钝，重则死锁崩溃。本节带你建立优先级分配的直觉。

优先级分配的黄金法则

在嵌入式系统中，并不是"所有任务生来平等"。有一条经过无数项目验证的优先级分配法则：

优先级层级任务类型典型优先级(0最低)实时性要求
最高关键控制（电机控制、安全保护、紧急停机）configMAX_PRIORITIES-1 起微秒级响应，错过=灾难
高传感器采集、通讯协议栈（CAN/UART中断驱动）中间偏高毫秒级响应，丢包不可接受
中数据处理、滤波算法、状态机逻辑中间10~100ms级别，可容忍抖动
低UI更新、日志输出、OLED刷新0~1百毫秒级别，肉眼感觉不到延迟即可
最低空闲任务(Idle Task)0(由系统自动创建)无实时性要求，清理删除的任务
STM32电机控制实战优先级分配：
• 优先级5(最高)：FOC电流环控制任务(每50μs执行一次，由ADC中断触发)
• 优先级4：CAN通讯接收任务(接收上位机指令，100μs级别)
• 优先级3：编码器数据采集任务(1ms周期)
• 优先级2：位置环PID计算任务(10ms周期)
• 优先级1：OLED显示更新任务(100ms周期)
• 优先级0(最低)：IDLE任务(系统自动)
核心原则：离硬件越近、延迟越致命的任务，优先级越高。

优先级反转问题

这是RTOS领域最臭名昭著的bug之一——1997年火星探路者号(Pathfinder)就因为它差点报废，Spirit号火星车也受其困扰。

经典反转场景危险
低优先级TaskL 获取了互斥量(持锁中)→ 被 中优先级TaskM 抢占 → TaskL挂起无法释放锁 → 高优先级TaskH 等锁进入阻塞 → TaskH被TaskM"间接"阻塞 → 等于TaskM优先级反转到了TaskH上面！
⚠️ 不使用互斥量的后果：
如果TaskH等待的锁被TaskL持有，而TaskL被TaskM打断——TaskH可能永久阻塞。中优先级的TaskM"卡住了"高优先级TaskH，这就是优先级反转。真实的火星车就是因为总线管理器的优先级反转导致系统复位。

优先级继承（Priority Inheritance）

FreeRTOS的互斥量(Mutex)自动支持优先级继承——这是解决优先级反转的标准方案：

// 互斥量自动执行优先级继承——你什么都不用做！
SemaphoreHandle_t xMutex;

void vTaskH(void *pv) {  // 优先级3(高)
    while (1) {
        // TaskH取锁时，如果TaskL(优先级1)正持有锁
        // FreeRTOS自动将TaskL临时提升到优先级3
        xSemaphoreTake(xMutex, portMAX_DELAY);
        // ... 临界操作（TaskL已被提升，不会被TaskM打断）
        xSemaphoreGive(xMutex);
        // TaskL优先级恢复为1
    }
}
✅ 防止优先级反转的三种方案：
1. 优先级继承(FreeRTOS互斥量自动)：持锁任务临时提升到等待者中最高优先级（最常用）
2. 优先级天花板：所有用这个锁的任务都设为同一高优先级（简单粗暴，浪费CPU）
3. 避免共享：用队列传递数据替代共享变量（从根源消灭反转，推荐首选）

饥饿问题与解决方案

饥饿(Starvation)：低优先级任务永远得不到CPU，因为总有更高优先级的任务在运行。

🔍 饥饿诊断信号：
• 某个任务上报的心跳/看门狗一直超时
• 日志任务在系统繁忙时完全停止输出
• 低优先级的传感器采集数据长期不更新
• CPU利用率报告里某些任务执行时间为零
// 饥饿修复方案1：给低优先级任务留执行窗口
void vHighPriorityTask(void *pv) {
    while (1) {
        // 高优先级工作...
        ProcessCriticalData();
        
        // 关键：主动延时，给低优先级任务让路
        vTaskDelay(pdMS_TO_TICKS(10));  // 让出10ms
    }
}

// 饥饿修复方案2：用软件定时器代替低优先级任务
// 将非紧急的定期工作放进定时器回调(在Daemon任务中执行)
TimerHandle_t xLogTimer;
xLogTimer = xTimerCreate("Log", pdMS_TO_TICKS(1000), 
                          pdTRUE, NULL, vLogCallback);
xTimerStart(xLogTimer, 0);

// 饥饿修复方案3：重新评估——低优先级任务是不是该提高优先级？
// 如果需要响应1秒以内的事件，那就不是"低优先级"
优先级分配速查表实用
如果你有...建议优先级分配理由

🎯 一个简单的自测问题："如果这个任务晚执行100ms，会发生什么？" → 如果答案是"灾难"就高优先级，"没感觉"就低优先级，"有点不舒服"就中优先级。这个问法比任何优先级计算公式都好用。

📝 任务创建与管理

#include "FreeRTOS.h"
#include "task.h"

// 任务1：LED闪烁
void vLEDTask(void *pvParameters) {
    while (1) {
        HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
        vTaskDelay(pdMS_TO_TICKS(500));  // 延时500ms
    }
}

// 任务2：串口打印
void vPrintTask(void *pvParameters) {
    while (1) {
        printf("System running, tick: %lu\n", xTaskGetTickCount());
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}

int main(void) {
    HAL_Init();
    SystemClock_Config();

    // 创建任务（参数：函数、名称、栈大小、参数、优先级、句柄）
    xTaskCreate(vLEDTask,    "LED",   128, NULL, 1, NULL);
    xTaskCreate(vPrintTask,  "Print", 256, NULL, 2, NULL);

    // 启动调度器
    vTaskStartScheduler();

}

任务状态机

每个任务在FreeRTOS中有四种状态：

状态含义举例
Running正在使用CPU（只有一个任务）当前执行代码的任务
Ready就绪等待CPU被高优先级抢走的任务
Blocked等待事件（超时/信号量/队列）vTaskDelay中、等信号量
Suspended挂起（不参与调度）vTaskSuspend后
// 任务挂起与恢复
TaskHandle_t xTaskHandle;

void vControlTask(void *pv) {
    while (1) {
        // 挂起LED任务（停止闪烁）
        vTaskSuspend(xTaskHandle);
        vTaskDelay(pdMS_TO_TICKS(3000));  // 3秒不闪
        
        // 恢复LED任务
        vTaskResume(xTaskHandle);
        vTaskDelay(pdMS_TO_TICKS(3000));  // 3秒继续闪
    }
}

// 主函数中保存句柄
xTaskCreate(vLEDTask, "LED", 128, NULL, 1, &xTaskHandle);

📊 队列（Queue）详解

队列是FreeRTOS任务间通信的主力工具。它是一个FIFO缓冲区，一个任务往里放，另一个从里取，天然线程安全。

💡 类比：队列像传送带——生产者把零件放在传送带上，消费者从另一端取走。传送带有固定容量，放满了就停，取空了就等。
// 创建队列：存放10个float值
QueueHandle_t sensor_queue;
sensor_queue = xQueueCreate(10, sizeof(float));

// 任务A：发送数据（生产者）
void vSensorTask(void* pv) {
    float temp;
    for (;;) {
        temp = read_temperature();
        // 发送到队列，等最多100ms
        if (xQueueSend(sensor_queue, &temp, pdMS_TO_TICKS(100)) != pdTRUE) {
            // 队列满了！消费者太慢
            printf("Queue full! Dropping data.\n");
        }
        vTaskDelay(pdMS_TO_TICKS(1000));  // 1秒采样一次
    }
}

// 任务B：接收数据（消费者）
void vDisplayTask(void* pv) {
    float temp;
    for (;;) {
        // 阻塞等待数据（无限等）
        if (xQueueReceive(sensor_queue, &temp, portMAX_DELAY) == pdTRUE) {
            printf("温度: %.1f°C\n", temp);
            // 处理数据...
        }
    }
}

队列API变体与选择指南

API行为适用场景
xQueueSend队列满则等待超时一般发送（别名xQueueSendToBack）
xQueueSendToBack发送到队尾常规FIFO发送
xQueueSendToFront发送到队首（插队）紧急消息/高优先级数据
xQueueOverwrite队列满则覆盖最旧数据（队列长度只能为1）只关心最新值（如最新传感器读数）
xQueuePeek读取但不移除预览下一个数据，多个消费者
xQueueReceive读取并移除一般接收
xQueueSendFromISR中断中发送中断服务程序中推数据
xQueueReceiveFromISR中断中接收（几乎不用）一般不推荐

队列集 (Queue Set)

当一个任务需要同时等待多个队列时，可以用队列集——一个Select/Poll模式。

// 队列集：同时等待多个数据源
QueueSetHandle_t xQueueSet;
QueueHandle_t xTempQueue, xHumQueue, xPressQueue;

void setup_queue_set(void) {
    xTempQueue = xQueueCreate(5, sizeof(float));
    xHumQueue  = xQueueCreate(5, sizeof(float));
    xPressQueue = xQueueCreate(3, sizeof(float));
    
    // 创建队列集（能容纳所有队列的句柄）
    xQueueSet = xQueueCreateSet(3);
    
    // 把队列添加到集合
    xQueueAddToSet(xTempQueue, xQueueSet);
    xQueueAddToSet(xHumQueue, xQueueSet);
    xQueueAddToSet(xPressQueue, xQueueSet);
}

void vAggregatorTask(void *pv) {
    for (;;) {
        // 等待任意一个队列有数据
        QueueHandle_t xActiveQueue = xQueueSelectFromSet(
            xQueueSet, portMAX_DELAY);
        
        float data;
        if (xQueueReceive(xActiveQueue, &data, 0) == pdTRUE) {
            if (xActiveQueue == xTempQueue) {
                printf("温度: %.1f°C\n", data);
            } else if (xActiveQueue == xHumQueue) {
                printf("湿度: %.1f%%\n", data);
            } else if (xActiveQueue == xPressQueue) {
                printf("气压: %.1f hPa\n", data);
            }
        }
    }
}

🔐 信号量（Semaphore）详解

💡 类比：二值信号量像一个令牌——拿到令牌的才能进入临界区，其他人排队。计数信号量像停车场的车位——有几个空位就能进几辆车。

二值信号量：任务同步

// 中断通知任务：按键中断 → 信号量 → 处理任务
SemaphoreHandle_t xButtonSem;

// ISR中释放信号量（只能Give，不能Take！）
void HAL_GPIO_EXTI_Callback(uint16_t pin) {
    if (pin == BUTTON_PIN) {
        BaseType_t xHigherPriorityTaskWoken = pdFALSE;
        xSemaphoreGiveFromISR(xButtonSem, &xHigherPriorityTaskWoken);
        portYIELD_FROM_ISR(xHigherPriorityTaskWoken);  // 唤醒高优先级
    }
}

// 任务中等待信号量
void vButtonTask(void *p) {
    while (1) {
        if (xSemaphoreTake(xButtonSem, portMAX_DELAY)) {
            handle_button_press();  // 处理按键
        }
    }
}

// 初始化
xButtonSem = xSemaphoreCreateBinary();  // 初始状态=0（未释放）

计数信号量：管理多个资源

// 管理5个DMA通道的计数信号量
SemaphoreHandle_t xDmaSem;
xDmaSem = xSemaphoreCreateCounting(5, 5);  // 最大5，初始5

// 使用DMA前获取
if (xSemaphoreTake(xDmaSem, pdMS_TO_TICKS(100)) == pdTRUE) {
    start_dma_transfer(buffer, size);
    // 传输完成中断中归还：xSemaphoreGiveFromISR(xDmaSem, ...)
}
// 超时=所有DMA都在忙，等待或报错

互斥量（Mutex）：保护共享资源

// 保护I2C总线（多任务共享）
SemaphoreHandle_t xI2cMutex;

void vTempSensorTask(void *pv) {
    for (;;) {
        if (xSemaphoreTake(xI2cMutex, pdMS_TO_TICKS(50)) == pdTRUE) {
            read_temp_via_i2c();       // 使用I2C
            xSemaphoreGive(xI2cMutex); // 释放
        }
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}

void vOledDisplayTask(void *pv) {
    for (;;) {
        if (xSemaphoreTake(xI2cMutex, pdMS_TO_TICKS(50)) == pdTRUE) {
            oled_refresh_via_i2c();    // 使用I2C
            xSemaphoreGive(xI2cMutex); // 释放
        }
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}

// 创建互斥量
xI2cMutex = xSemaphoreCreateMutex();

信号量 vs 互斥量（Mutex）

⚠ 关键区别：用互斥量(Mutex)而不是二值信号量做资源互斥！Mutex有优先级继承机制（防止优先级反转），二值信号量没有。
特性二值信号量互斥量(Mutex)
用途任务同步（通知信号）资源互斥（保护临界区）
优先级继承❌ 无✅ 有（自动提升持有者优先级）
递归锁❌ 不支持✅ 支持(xSemaphoreCreateRecursiveMutex)
ISR中使用✅ 可以Give❌ 不能（只能任务间使用）
典型场景ISR通知任务"数据就绪"保护I2C/SPI/串口等共享外设
优先级反转问题详解进阶
场景：三个任务 A(优先级低)、B(优先级中)、C(优先级高)
① A获取锁 → ② C就绪想拿锁(C被阻塞等A释放锁) → ③ B就绪抢占A → ④ A放不了锁 → ⑤ C被B无限期阻塞！

用二值信号量的结果：高优先级C被低优先级B饿死 = 优先级反转
用Mutex的结果：Mutex检测到C在等锁 → 临时将A优先级提升到C的级别 → A快速完成→释放锁→C获取→A降回原优先级
这就是优先级继承(Priority Inheritance)——Mutex自动解决优先级反转。

🏴 事件组（Event Group）

事件组是FreeRTOS中最强大的同步机制之一：一个任务可以等待多个事件中的任意一个或全部。不需要多个信号量。

💡 类比：事件组像机场登机——需要身份证(bit0)+登机牌(bit1)+安检通过(bit2)全满足才能登机。少任何一个都不行。也可以设置成"任一条件满足"就出发（VIP通道）。
// 事件组：等待多个条件
EventGroupHandle_t xSystemEvent;

#define EVENT_WIFI_READY   (1 << 0)
#define EVENT_SD_READY     (1 << 1)
#define EVENT_SENSOR_OK    (1 << 2)
#define EVENT_TIME_SYNC    (1 << 3)

void vInitTask(void *pv) {
    // 等待所有初始化完成（AND所有位）
    EventBits_t uxBits = xEventGroupWaitBits(
        xSystemEvent,
        EVENT_WIFI_READY | EVENT_SD_READY | EVENT_SENSOR_OK | EVENT_TIME_SYNC,
        pdTRUE,           // 等待后清除这些位
        pdTRUE,           // 等待所有位(AND)，pdFALSE=任一(OR)
        pdMS_TO_TICKS(10000)  // 超时10秒
    );
    
    if ((uxBits & (EVENT_WIFI_READY | EVENT_SD_READY | EVENT_SENSOR_OK | EVENT_TIME_SYNC))
        == (EVENT_WIFI_READY | EVENT_SD_READY | EVENT_SENSOR_OK | EVENT_TIME_SYNC)) {
        printf("All systems ready!\n");
    } else {
        printf("Init timeout! Missing: WiFi=%d SD=%d Sensor=%d Time=%d\n",
            !!(uxBits & EVENT_WIFI_READY),
            !!(uxBits & EVENT_SD_READY),
            !!(uxBits & EVENT_SENSOR_OK),
            !!(uxBits & EVENT_TIME_SYNC));
    }
}

// 各个模块初始化完成后设置对应位
void wifi_ready(void) {
    xEventGroupSetBits(xSystemEvent, EVENT_WIFI_READY);
}
void sd_ready(void) {
    xEventGroupSetBits(xSystemEvent, EVENT_SD_READY);
}

// 中断中设置事件
void some_isr(void) {

📬 任务通知（Task Notification）

任务通知是FreeRTOS最高效的同步方式——无需创建队列/信号量/事件组对象，每个任务自带一个通知值。

💡 性能对比：任务通知比二值信号量快45%，比队列快60%，且零RAM开销（不需要创建内核对象）。但只适用于一对一的通信。
// 任务通知：最轻量的任务间通信
TaskHandle_t xSensorTaskHandle;

// 中断/任务A：发送通知给任务B
void vNotifySender(void *pv) {
    for (;;) {
        vTaskDelay(pdMS_TO_TICKS(500));

        xTaskNotifyGive(xSensorTaskHandle);
        // 或者带值发送：
        // xTaskNotify(xSensorTaskHandle, (1<<0), eSetBits);
    }
}
void HAL_GPIO_EXTI_Callback(uint16_t pin) {

⚠️ ISR安全指南

中断服务程序(ISR)是嵌入式系统的"特权公民"——它们可以打断任何任务，但它们也必须遵守特殊的规则。本节说明如何在FreeRTOS中写出安全、高效的ISR。

FromISR API vs 普通API

FreeRTOS中大多数API都有两个版本：任务版本和ISR版本。规则很简单——在ISR中只能用FromISR版本：

场景使用的API不能在ISR用的普通版
发送队列xQueueSendFromISR() / xQueueSendToBackFromISR()xQueueSend()
接收队列xQueueReceiveFromISR()xQueueReceive()
释放信号量xSemaphoreGiveFromISR()xSemaphoreGive()
取信号量xSemaphoreTakeFromISR()xSemaphoreTake()
任务通知xTaskNotifyFromISR() / xTaskNotifyGiveFromISR()xTaskNotify() / xTaskNotifyGive()
恢复任务xTaskResumeFromISR()vTaskResume()
设置事件组xEventGroupSetBitsFromISR()xEventGroupSetBits()
🚫 致命错误：在ISR中调用普通版API（如xQueueSend()）不会编译报错，但运行时会导致未定义行为——轻则数据错乱，重则HardFault。因为普通版API会尝试阻塞等待（调vTaskSuspendAll等），而ISR根本没有任务上下文。

pxHigherPriorityTaskWoken 参数详解

每个FromISR函数都有这个参数——它是整个FreeRTOS ISR机制的核心，也是最容易被初学者忽略的地方。

// pxHigherPriorityTaskWoken 的工作机制
BaseType_t xHigherPriorityTaskWoken = pdFALSE;  // 初始化

void ADC_IRQHandler(void) {  // ADC中断（优先级高于SysTick）
    uint16_t adc_val = ADC1->DR;
    
    // 向任务发送ADC数据——注意最后一个参数
    xQueueSendFromISR(xADCQueue, &adc_val, &xHigherPriorityTaskWoken);
    
    // ★★★ 关键：告诉调度器"有高优先级任务被唤醒了" ★★★
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
    //                    ↑ 如果为pdTRUE，强制在ISR返回后立即切换任务
}

// 为什么这是必要的？
// 场景：低优先级TaskL正在运行，ADC中断发生
// 1. ISR中xQueueSendFromISR把数据放进队列
// 2. 等待队列的高优先级TaskH被唤醒→变为Ready
// 3. xHigherPriorityTaskWoken 被设为 pdTRUE
// 4. portYIELD_FROM_ISR(pdTRUE)→调度器立即让TaskH运行
// 5. 如果不调用portYIELD_FROM_ISR：ISR返回后继续运行TaskL，
//    TaskH要等到下一个SysTick才可能运行——延迟可达1ms!
参数机制拆解原理
pxHigherPriorityTaskWoken 是一个"出参"标志位，每个FromISR函数内部会：
1. 检查解锁的任务/信号量/队列是否让更高优先级任务变为Ready
2. 如果是 → 将 *pxHigherPriorityTaskWoken 设为 pdTRUE
3. 任务代码在ISR末尾调用 portYIELD_FROM_ISR(xHigherPriorityTaskWoken)
4. 如果有多个FromISR调用→共享同一个变量，每个调用都可能把标志置为TRUE
5. 最后只需一次portYIELD_FROM_ISR()即可
注意：一定在ISR开头初始化为pdFALSE！很多bug就源于忘记初始化。 
// ✅ 正确：多个FromISR调用共享同一标志
void TIM2_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;  // 初始化！
    
    // 先发信号量通知采集任务
    xSemaphoreGiveFromISR(xSampleSemaphore, &xHigherPriorityTaskWoken);
    // 再发队列数据给处理任务
    xQueueSendFromISR(xDataQueue, &sensor_data, &xHigherPriorityTaskWoken);
    
    // 一次portYIELD即可处理所有被唤醒的任务
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}

// ❌ 常见错误：忘记初始化
void WRONG_TIM_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken;  // 未初始化！可能是垃圾值pdTRUE
    xSemaphoreGiveFromISR(xSem, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);  // 可能强制不必要的任务切换
}

中断服务程序太长的危害

ISR应该像闪电一样快——进来、做事、出去。过长的ISR会引发一系列连锁反应：

问题后果触发阈值(Cortex-M3/4)
嵌套中断延迟同优先级/低优先级中断得不到服务ISR > 2-3μs
任务调度延迟高优先级任务得不到CPUISR > 10μs(跳过多个tick)
中断丢失如果新中断在ISR处理期间重复产生取决于外设FIFO深度
看门狗复位ISR阻塞了喂狗任务ISR > 看门狗超时/2
优先级反转ISR在临界区禁了中断，阻塞所有ISR临界区 > 1μs
✅ ISR设计黄金原则——"中断瘦身"三招式：
1. ISR只做数据搬运：例如ADC ISR只读DR寄存器→xQueueSendFromISR，复杂处理留给任务
2. 用DMA代替ISR轮询：让硬件做重复的数据搬运，ISR只在传输完成/半完成时触发
3. 任务通知轻量级唤醒：如果只需要"通知"而不需要传数据，vTaskNotifyGiveFromISR()比队列更快(~2μs vs ~5μs)
// ❌ 坏代码：ISR里做复杂处理
void BAD_ADC_IRQHandler(void) {
    uint16_t raw = ADC1->DR;
    float voltage = raw * 3.3f / 4096.0f;  // 浮点运算在ISR中很重
    float temperature = (voltage - 0.76f) / 0.0025f + 25.0f;  // 更重
    if (temperature > 80.0f) {
        EmergencyShutdown();  // 函数调用在ISR中不可控
    }
    printf("Temp=%.1f\r\n", temperature);  // 阻塞式UART在ISR中=灾难
}

// ✅ 好代码：ISR只搬运数据
void GOOD_ADC_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    uint16_t raw = ADC1->DR;  // 只读寄存器
    xQueueSendFromISR(xADCRawQueue, &raw, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
    // 所有浮点计算、判断、日志都交给任务做
}

ARM Cortex-M的中断优先级数值和直觉相反：数字越小优先级越高。而且FreeRTOS对中断优先级有严格约束。

// 优先级寄存器：8位宽，但芯片厂商可能只实现高4位(16级)或高3位(8级)
// STM32F1: 4位→16级(0~15)，数字越小优先级越高
// STM32F4: 4位→16级(0~15)，0=最高，15=最低

📨 流缓冲区与消息缓冲区

队列适合传递固定大小的数据块，但有时你需要像串口一样流式传输字节。FreeRTOS从v10.0.0开始提供Stream Buffer和Message Buffer。

💡 类比：队列像固定规格的快递箱——只能放统一尺寸的包裹。Stream Buffer像水管——任意字节流都行，Message Buffer是带换行的水管（每次发一条完整消息）。
// Stream Buffer：流式传输任意长度字节
// 适合：串口数据桥接、传感器原始字节流、音频流

StreamBufferHandle_t xStreamBuf;

// 创建1024字节的流缓冲区，触发级别为1（有1字节就唤醒接收任务）
xStreamBuf = xStreamBufferCreate(1024, 1);

// 生产者：写入字节流
void vUartRxTask(void *pv) {
    uint8_t buf[64];
    for (;;) {
        size_t len = HAL_UART_Receive(&huart2, buf, sizeof(buf), portMAX_DELAY);
        // 写入Stream Buffer（可能阻塞如果空间不够）
        size_t sent = xStreamBufferSend(xStreamBuf, buf, len, pdMS_TO_TICKS(100));
        if (sent < len) {
            printf("Stream buffer full! Dropped %d bytes\n", len - sent);
        }
    }
}

// 消费者：读取字节流
void vProcessTask(void *pv) {
    uint8_t buf[128];
    for (;;) {
        // 阻塞等待任意数量字节
        size_t received = xStreamBufferReceive(xStreamBuf, buf, sizeof(buf), portMAX_DELAY);
        // 处理received个字节...
        process_bytes(buf, received);
    }
}
// Message Buffer：以消息为单位的流缓冲
// 适合：变长日志、JSON字符串、AT指令响应

    char buf[512];
    for (;;) {

💡 Stream vs Message Buffer选择：Stream适合连续字节流（UART桥接、音频数据），Message适合离散消息（JSON日志、AT指令响应）。Message Buffer保证每次Receive返回完整一条消息，不会截断。

🔒 临界区与挂起调度器

有时候你需要保护一段代码不被中断或任务切换打断。FreeRTOS提供了两种机制。

// 临界区（Critical Section）：屏蔽中断，最严格保护
// ⚠ 只能在极短代码段使用（<1μs）！长时间的临界区破坏系统实时性

void safe_counter_increment(void) {
    taskENTER_CRITICAL();  // 关中断（或关优先级低于configMAX_SYSCALL_INTERRUPT_PRIORITY的中断）
    global_counter++;       // 原子操作
    taskEXIT_CRITICAL();   // 恢复中断
}

// 从ISR中使用
void TIM1_UP_IRQHandler(void) {
    UBaseType_t uxSavedInterruptStatus;
    uxSavedInterruptStatus = taskENTER_CRITICAL_FROM_ISR();
    critical_data++;
    taskEXIT_CRITICAL_FROM_ISR(uxSavedInterruptStatus);
}
// 挂起调度器：禁止任务切换但允许中断
// 比临界区更轻量，适合较长代码段

void safe_multi_step_update(void) {
    vTaskSuspendAll();  // 挂起调度器（当前任务不会被抢占）
    
    // 这段代码不会被其他任务打断（但中断仍可发生）
    sensor_data[0] = read_sensor_1();
    sensor_data[1] = read_sensor_2();
    sensor_data[2] = read_sensor_3();
    data_ready = pdTRUE;
    
    xTaskResumeAll();   // 恢复调度器（可能立即触发上下文切换）
    // ⚠ 如果xTaskResumeAll返回pdTRUE，说明有高优先级任务就绪
}

// ⚠ 挂起调度器期间不能调用会阻塞的API（vTaskDelay/xQueueReceive等）
//    否则可能导致调度器永久挂起！
⚠ 临界区vs挂起调度器选型：
• 临界区(taskENTER_CRITICAL)：保护被ISR和任务同时访问的变量（如ISR中更新的计数器），代码<1μs
• 挂起调度器(vTaskSuspendAll)：保护仅任务间共享的数据，代码可稍长但绝不能调阻塞API
• 互斥量(xSemaphoreTake)：保护任务间共享的耗时操作（如I2C通信），最安全
• 能用互斥量解决就不要用临界区/挂起调度器！

🔧 Stack溢出检测与configASSERT

栈溢出是嵌入式FreeRTOS的头号隐性杀手——它不报编译错、不触发断言、甚至可能"看起来一切正常"，直到某天系统莫名其妙HardFault。本节教你如何防患于未然。

为什么会栈溢出？

每个FreeRTOS任务都有自己的栈空间（在xTaskCreate时分配），如果任务实际使用的栈超过了分配的栈大小，就会踩踏相邻内存区域（通常是下一个任务的TCB或栈），导致不可预测的行为。

🔍 常见栈溢出原因：
• 函数调用深度太大（递归或深层嵌套）→ 局部变量层层压栈
• 局部变量太大（在栈上声明了大数组，如 char buf[1024]）
• sprintf/printf 系列函数调用（内部缓冲区通常很大）
• 中断嵌套：ISR使用任务栈（Cortex-M使用MSP或PSP，通常ISR用MSP）
• 栈空间估算不足：以为128字够了，实际上函数调用链很长
• 编译器优化等级变化：低优化可能产生更多栈使用

FreeRTOSConfig.h 中的检测选项

FreeRTOS提供了两级栈溢出检测，在FreeRTOSConfig.h中配置：

// FreeRTOSConfig.h 中的关键配置

// 选项0：不检测（正式发布时可用，开发阶段必须开）
#define configCHECK_FOR_STACK_OVERFLOW    0

// 选项1：快速检测（Tick中断中检查当前任务栈顶）
// 工作原理：每次Tick中断发生时，检查当前任务栈指针是否
//          越过栈底——只能检测"已经溢出"的情况
// 优点：开销极小（几条指令）
// 缺点：可能检测不到瞬时溢出（溢出后栈指针又恢复正常的情况）

configASSERT 实战用法

configASSERT是FreeRTOS的运行时断言机制——它能捕获API的非法调用和内部状态异常。这是比栈溢出检测更基础的保护手段。

// FreeRTOSConfig.h 中定义configASSERT
// 开发阶段：详细记录后死循环
#define configASSERT(x)  \
    if ((x) == 0) {      \
        taskDISABLE_INTERRUPTS();  \
        printf("ASSERT FAILED: %s line %d\r\n", __FILE__, __LINE__); \
        for (;;);          \
    }

// 进阶版本：打印文件名+行号（__FILE__和__LINE__需要编译器支持）
// 更进阶：把断言信息写入RTC备份域或Flash特定扇区
#define configASSERT(x)  \
    if ((x) == 0) {      \
        portDISABLE_INTERRUPTS();  \
        /* 记录到备份寄存器 */          \
        RTC->BKP0R = (uint32_t)__LINE__; \
        RTC->BKP1R = 0xASSERT00;         \
        NVIC_SystemReset();              \
    }

// configASSERT 能捕获的常见错误：
// - 在调度器启动前调用阻塞API（如xQueueReceive）
// - 在临界区内调用阻塞API
// - 在ISR中调用非FromISR API
// - 无效的任务优先级（>= configMAX_PRIORITIES）
// - 传递NULL句柄给API
configASSERT 保护实战案例：
// 经典的"调度前取队列"bug会被configASSERT捕获
void vBugDemo(void *pv) {
    QueueHandle_t xQ = xQueueCreate(10, sizeof(int));
    int val;
    // ❌ 调度器还没启动就在等队列——configASSERT会在这里触发
    xQueueReceive(xQ, &val, portMAX_DELAY);
    // 输出: ASSERT FAILED: queue.c line 1532

提示：启用configASSERT后，所有FreeRTOS内部函数都会在关键路径上插入断言检查。这会让系统稍慢（约5~10%），但能捕获90%的API误用。正式发布时可以定义为空宏来关闭。

uxTaskGetStackHighWaterMark() 调试技巧

即使栈检测没有触发溢出，你的任务也可能在危险的边缘反复试探。uxTaskGetStackHighWaterMark()返回任务历史上栈空间剩余最少时的"水位"——值越小说明越接近溢出。

// 栈水位监测任务——每个任务都该有一个监控任务
void vStackMonitorTask(void *pvParameters) {
    TaskHandle_t tasks_to_monitor[] = {
        xLEDTaskHandle, xSensorTaskHandle, xCommTaskHandle
    };
    const char *task_names[] = {"LED", "Sensor", "Comm"};
    
    while (1) {
        printf("===== Stack WaterMark Report =====\r\n");
        for (int i = 0; i < 3; i++) {
            // 返回剩余栈空间（单位：字，不是字节！）
            UBaseType_t uxHighWaterMark = 
                uxTaskGetStackHighWaterMark(tasks_to_monitor[i]);
            
            printf("  %s: %lu words (%lu bytes) remaining\r\n",
                   task_names[i],
                   uxHighWaterMark,
                   uxHighWaterMark * sizeof(StackType_t));
            
            // 如果剩余不到20%，发出警告
            if (uxHighWaterMark < 50) {  // 假设栈大小为256字
                printf("  ⚠️ WARNING: %s stack critically low!\r\n", 
                       task_names[i]);
            }
            
            // 如果水位为0或1，栈可能已经溢出但还没触发检测
            if (uxHighWaterMark <= 1) {
                printf("  🚨 CRITICAL: %s stack overflow likely!\r\n",
                       task_names[i]);
            }
        }
        vTaskDelay(pdMS_TO_TICKS(5000));  // 每5秒检查一次
    }
}

// 创建监控任务
xTaskCreate(vStackMonitorTask, "StackMon", 256, NULL, 1, NULL);
栈大小快速估算指南实用
保守估算公式（开发阶段用）：
栈大小 = (最大函数调用深度 × 每层开销) + 最大局部变量总和 + 安全余量(50%)

实测步进法（推荐）：
1. 先分配"大方"的栈（如512字/2KB）
2. 运行5分钟后调用 uxTaskGetStackHighWaterMark()
3. 如果剩余 > 60% → 可适当减小栈
4. 如果剩余 < 20% → 必须增大栈
5. 最终栈大小 = 实际最大使用量 × 1.3（30%安全余量）

💡 终极调试技巧——栈内容染色法：在xTaskCreate之前，用memset把整个栈空间填满0xA5(或其他已知值)。运行一段时间后暂停系统，用调试器查看栈内存——找到第一个不是0xA5的位置，就能精确算出最大栈用量。FreeRTOS的选项2本质就是自动做了染色检查（在任务创建时填充魔数）。

🛠 常见问题排查

🔍 问题1：任务创建后不运行
❌ 症状：创建了3个任务,只有1个在运行。
✅ 排查：
1. vTaskStartScheduler() 之前没有创建任何任务？调度器不会创建任务
2. 检查优先级是否合理——高优先级任务一直"Running"→低优先级永远得不到CPU
3. 高优先级任务是否有vTaskDelay()或阻塞调用？没有=死循环=饿死其他任务
4. 用vTaskList()查看所有任务的运行状态
✅ 修复：确保每个高优先级任务都有阻塞调用，让出CPU给低优先级。
🔍 问题2：HardFault / 程序跑飞
❌ 症状：程序不定期崩溃，HardFault_Handler。
✅ 排查：
1. 任务栈溢出（最常见！）：uxTaskGetStackHighWaterMark(NULL)接近0
2. 中断中使用阻塞API：ISR中调用了xQueueReceive等
3. 互斥量未释放：某个任务拿了锁但忘了Give
4. 动态内存耗尽：heap太小，xTaskCreate返回NULL
✅ 修复：
1. 增大任务栈（printf密集的任务至少512字）
2. 启用configCHECK_FOR_STACK_OVERFLOW = 2
3. 检查所有Take都有对应的Give
4. 增大configTOTAL_HEAP_SIZE或用heap_4.c
🔍 问题3：队列数据丢失
❌ 症状：传感器读数偶尔丢失。
✅ 排查：
1. 生产者使用xQueueSend但队列满了——检查返回值不是pdTRUE
2. 消费者处理太慢，队列积压
3. 中断中以不安全速率发送
✅ 修复：
1. 增加队列长度
2. 如果只关心最新值，用xQueueOverwrite代替
3. 消费者任务提高优先级
🔍 问题4：vTaskDelay时间不准
❌ 症状：vTaskDelay(1000)实际延时不是1000ms。
✅ 排查：
1. configTICK_RATE_HZ不是1000？手动改过SysTick配置？
2. 更高优先级中断打断了SysTick？
3. 时钟配置错误导致系统时钟不准
✅ 修复：
1. 确认configCPU_CLOCK_HZ和configTICK_RATE_HZ匹配实际硬件
2. 用示波器测量SysTick频率
🔍 问题5：HAL_Delay与FreeRTOS冲突
❌ 症状：在FreeRTOS任务中调用HAL_Delay后系统卡死。
✅ 原因：HAL_Delay基于轮询等待SysTick计数——它不会阻塞任务让出CPU，反而空耗CPU导致同优先级和低优先级任务饿死。
✅ 修复：任务中用vTaskDelay(pdMS_TO_TICKS(ms))替代所有HAL_Delay(ms)。初始化代码中（调度器启动前）可以继续用HAL_Delay。

项目目标实战
搭建一个完整的多任务数据采集系统：
• Task1 (优先级3): 温度传感器读取(1Hz)，通过队列发送
• Task2 (优先级2): OLED显示刷新(10Hz)
• Task3 (优先级2): 串口数据上报(1Hz)
• Task4 (优先级1): LED心跳指示(2Hz)
• 用互斥量保护I2C总线(Task1和Task2共享)
• 用队列传递传感器数据
• 用事件组协调系统启动 
// 完整项目：main.c
#include "FreeRTOS.h"
#include "task.h"
#include "queue.h"
#include "semphr.h"
#include "event_groups.h"

// === 全局对象 ===
QueueHandle_t xSensorQueue;
SemaphoreHandle_t xI2cMutex;
EventGroupHandle_t xSystemEvent;

#define EVT_SENSOR_OK   (1 << 0)
#define EVT_DISPLAY_OK  (1 << 1)
#define EVT_UART_OK     (1 << 2)

// === 数据结构 ===
typedef struct {
    float temperature;
    float humidity;
    uint32_t timestamp;
} SensorData_t;

// === Task1: 传感器采集 (优先级3, 最高) ===
void vSensorTask(void *pv) {
    SensorData_t data;
    
    // 初始化传感器
    if (bme280_init()) {
        xEventGroupSetBits(xSystemEvent, EVT_SENSOR_OK);
    }
    
    for (;;) {
        // 获取I2C使用权
        if (xSemaphoreTake(xI2cMutex, pdMS_TO_TICKS(100)) == pdTRUE) {
            data.temperature = bme280_read_temp();
            data.humidity    = bme280_read_humidity();
            data.timestamp   = xTaskGetTickCount();
            xSemaphoreGive(xI2cMutex);
            
            // 发送数据到显示任务和上报任务
            xQueueOverwrite(xSensorQueue, &data);  // 只保留最新数据
        }
        
        vTaskDelay(pdMS_TO_TICKS(1000));  // 1Hz
    }
}

// === Task2: OLED显示 (优先级2) ===
void vDisplayTask(void *pv) {
    SensorData_t data;
    oled_init();
    xEventGroupSetBits(xSystemEvent, EVT_DISPLAY_OK);
    
    for (;;) {
        // 非阻塞获取最新数据
        if (xQueuePeek(xSensorQueue, &data, 0) == pdTRUE) {
            if (xSemaphoreTake(xI2cMutex, pdMS_TO_TICKS(50)) == pdTRUE) {
                oled_show_data(data.temperature, data.humidity);
                xSemaphoreGive(xI2cMutex);
            }
        }
        vTaskDelay(pdMS_TO_TICKS(100));  // 10Hz刷新
    }
}

// === Task3: 串口上报 (优先级2) ===
void vReportTask(void *pv) {
    SensorData_t data;
    xEventGroupSetBits(xSystemEvent, EVT_UART_OK);
    
    for (;;) {
        if (xQueueReceive(xSensorQueue, &data, pdMS_TO_TICKS(5000)) == pdTRUE) {
            printf("[%lu] T=%.1f°C H=%.1f%%\r\n",
                   data.timestamp, data.temperature, data.humidity);
        }
        // 不需要vTaskDelay——xQueueReceive已经阻塞等待
    }
}

// === Task4: 心跳LED (优先级1, 最低) ===
void vHeartbeatTask(void *pv) {
    for (;;) {
        HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
        vTaskDelay(pdMS_TO_TICKS(500));  // 2Hz
    }
}

// === 主函数 ===
int main(void) {
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_I2C1_Init();
    MX_USART1_UART_Init();
    
    // 创建IPC对象
    xSensorQueue = xQueueCreate(1, sizeof(SensorData_t));  // 队列长度1+覆盖
    xI2cMutex    = xSemaphoreCreateMutex();
    xSystemEvent = xEventGroupCreate();
    
    if (xSensorQueue && xI2cMutex && xSystemEvent) {
        // 创建任务（优先级从高到低）
        xTaskCreate(vSensorTask,    "Sensor",  512, NULL, 3, NULL);
        xTaskCreate(vDisplayTask,   "Display", 384, NULL, 2, NULL);
        xTaskCreate(vReportTask,    "Report",  512, NULL, 2, NULL);
        xTaskCreate(vHeartbeatTask, "Beat",    128, NULL, 1, NULL);
        
        // 等待系统初始化完成
        EventBits_t uxBits = xEventGroupWaitBits(
            xSystemEvent,
            EVT_SENSOR_OK | EVT_DISPLAY_OK | EVT_UART_OK,
            pdTRUE, pdTRUE, pdMS_TO_TICKS(5000));
        
        if (uxBits & EVT_SENSOR_OK) printf("Sensor OK\n");
        if (uxBits & EVT_DISPLAY_OK) printf("Display OK\n");
        if (uxBits & EVT_UART_OK) printf("UART OK\n");
        
        // 启动调度器
        vTaskStartScheduler();
    }
    
    // 初始化失败
    printf("Failed to create RTOS objects!\n");
    while (1);
}

方案特点适用

📋 知识点回顾

✅ FreeRTOS核心IPC全览：
• 队列：任务间数据传递，FIFO，线程安全——传数据用这个
• 二值信号量：任务/ISR同步（通知信号）——发通知用这个
• 互斥量(Mutex)：资源互斥，有优先级继承——保护共享资源用这个
• 计数信号量：管理多个同类资源——资源池用这个
• 事件组：多条件同步(AND/OR)——等待多个事件用这个
• 任务通知：最轻量一对一通信——追求性能用这个
• 软件定时器：不需要单独任务，Daemon任务执行回调
⚠ 关键原则：
• 资源互斥用Mutex，不要用二值信号量
• ISR中只能Give信号量，不能Take
• 永远不要在高优先级ISR里调用阻塞API
• 队列满时不丢数据用xQueueSend，只关心最新值用xQueueOverwrite
• 每个任务必须有阻塞点（vTaskDelay/队列等待/信号量等待）
• 任务栈用uxTaskGetStackHighWaterMark()测量，不要猜
📏 优先级设计准则：
• 硬实时(电机控制) → 最高
• 数据采集 → 中高
• UI/显示 → 中
• 后台/日志 → 低
• 永不阻塞的一般任务 → 最低
