第1章：ARM GCC编译工具链详解

📖 小白完全指南 · 1/14 | 阅读时间约3分钟

ARM GCC不是"一个"程序，而是一整套工具。每个工具负责编译流程的一个环节，就像一条流水线——钢材进来，汽车出去。

💡 类比：编译就像翻译一本外文小说——预处理=整理草稿（去掉注释、展开宏）；编译=逐句翻译成目标语言；汇编=查字典把句子变成字母编码；链接=把散页装订成册、补上目录和索引。

编译四阶段

阶段工具输入输出做了什么
1. 预处理cpp (C Preprocessor).c + .h.i展开宏、条件编译、去注释
2. 编译cc1 / cc1plus.i.sC/C++ → ARM汇编
3. 汇编as (Assembler).s.o汇编 → 机器码(目标文件)
4. 链接ld (Linker).o + 库.elf合并目标文件、分配地址
# 一步到位（最常用）
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -O2 -Wall main.c -o main.elf

# 分步执行（学习用）
arm-none-eabi-gcc -E main.c -o main.i          # 只预处理
arm-none-eabi-gcc -S main.i -o main.s           # 只编译到汇编
arm-none-eabi-as -mcpu=cortex-m3 main.s -o main.o  # 只汇编
arm-none-eabi-ld main.o -T stm32.ld -o main.elf    # 只链接

核心工具详解

编译参数速查必须理解
每次编译都会用到这些参数，搞懂它们是读懂所有构建命令的前提：
参数含义为什么需要
-mcpu=cortex-m3目标CPU型号STM32F103=M3, F407=M4, H743=M7
-mthumb使用Thumb指令集代码更小（16位指令），ARM单片机必选
-O2优化等级O0=不优化(调试用), O2=速度优化, Os=体积优化
-Wall开启所有警告帮你提前发现Bug，必须开
-ffunction-sections每个函数单独section配合--gc-sections删除未用函数
-fdata-sections每个数据单独section同上，删除未用变量
-g3调试信息等级3GDB调试需要，等级3=宏信息也包含
--specs=nano.specs使用精简C库代码体积减半！嵌入式必选
--specs=nosys.specs无系统调用裸机程序不需要OS接口
-Wl,--gc-sections链接时删除未用代码与-ffunction-sections配合，省Flash
-Wl,-Map=xx.map生成map文件查看函数地址和大小，排错利器
arm-none-eabi-gcc编译器驱动
主入口，自动调用预处理→编译→汇编→链接四个阶段。平时99%的情况用这个就够了。
# 常用编译选项
-mcpu=cortex-m3    # 指定CPU架构（M3/M4/M7等）
-mthumb            # 使用Thumb指令集（代码更小）
-O2                # 优化等级（O0/O1/O2/Os）
-Wall              # 开启所有常见警告
-ffunction-sections # 每个函数单独一个section（配合链接器优化）
-ggdb3             # 生成GDB调试信息
-I./inc            # 头文件搜索路径
-DSTM32F103        # 定义预处理宏
arm-none-eabi-objcopy格式转换
把ELF格式转成BIN/HEX，烧录器需要的是这些"纯数据"格式。
# ELF → BIN（纯二进制，最常用）
arm-none-eabi-objcopy -O binary main.elf main.bin

# ELF → HEX（Intel HEX格式，带地址信息）
arm-none-eabi-objcopy -O ihex main.elf main.hex

# 查看各段信息
arm-none-eabi-objcopy -O binary -j .isr_vector main.elf vector.bin
arm-none-eabi-objdump反汇编器
查看目标文件的汇编代码、段信息。调试时看CPU到底在执行什么。
# 反汇编ELF文件（带C源码对照）
arm-none-eabi-objdump -d -S main.elf > main.lst

# 查看段头部信息
arm-none-eabi-objdump -h main.elf

# 只反汇编某个函数
arm-none-eabi-objdump -d main.elf | Select-String -Pattern "main>:" -Context 0,20
arm-none-eabi-size空间分析
查看编译后的代码占多少Flash和RAM，嵌入式开发必须关注！
arm-none-eabi-size main.elf
#    text    data     bss     dec     hex filename
#   12340     120    2048   14508    38ac main.elf
# Flash占用 = text + data = 12460 字节
# RAM占用  = data + bss  = 2168 字节
arm-none-eabi-nm符号表
列出所有符号（函数、全局变量）及其地址。
# 按地址排序查看所有符号
arm-none-eabi-nm -n main.elf

# 只看未定义符号
arm-none-eabi-nm -u main.elf

# 查看某个函数地址
arm-none-eabi-nm main.elf | findstr "main"

.map 文件

链接器输出的"地图"——每个函数、变量被放在了哪个地址，各段大小是多少。排错时非常有用。

# 生成map文件
arm-none-eabi-gcc ... -Wl,-Map=main.map main.elf

# map文件关键信息：
# .text    0x08000000    12340  ← 代码段
# .data    0x20000000      120  ← 数据段
# .bss     0x20000078     2048  ← BSS段
# main     0x08000120          ← main函数地址
🔍 常见问题：编译成功但程序跑飞
90%是链接脚本(.ld)里Flash/RAM起始地址写错了。STM32F103C8T6的Flash从0x08000000开始，RAM从0x20000000开始。检查.map文件确认地址。
💡 小技巧：编译时永远加 -Wl,-Map=output.map 和 -g，这两个选项不要省。
📝 完整示例：编译并分析一个STM32项目
输入命令：
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -O2 -Wall -g3 \
    -ffunction-sections -fdata-sections \
    -I./inc -DSTM32F103 \
    src/main.c src/stm32f1xx_it.c -T STM32F103C8Tx.ld \
    -Wl,--gc-sections -Wl,-Map=app.map --specs=nano.specs \
    -o app.elf

arm-none-eabi-objcopy -O binary app.elf app.bin
arm-none-eabi-size app.elf
输出结果：
   text    data     bss     dec     hex filename
  12340     120    2048   14508    38ac app.elf

Flash占用 = text + data = 12460 字节 (12.2KB)
RAM占用  = data + bss  = 2168 字节 (2.1KB)
STM32F103C8 有 64KB Flash, 20KB RAM → 充足!
