1 烧录方式对比

烧录（Flash Programming）就是把编译好的程序文件写入芯片内部 Flash 存储器的过程。调试（Debug）则是通过仿真器实时控制芯片运行，观察寄存器、内存、单步执行。不同的工具链适合不同的场景，下面逐一解析。

把芯片想成一本手写笔记本。烧录就是在空白页上"写入内容"，调试就是"翻页检查+修改内容"。不同的烧录器就像不同的笔——有的快但贵，有的慢但免费，有的只能写字，有的还能擦除重写。
🔌
ST-Link
+ 便宜（正品约80元）
+ 官方支持好
+ 兼容STM32全系列
− 只支持ST芯片
− 部分是克隆版
⚡
J-Link
+ 速度最快
+ 支持所有ARM芯片
+ 调试体验最佳
− 价格贵（500元+）
− 商业用途要license
🔄
DAP-Link
+ 开源免费
+ CMSIS-DAP协议
+ 可自己DIY
− 速度中等
− 需要自己组装
📡
串口ISP
+ 零成本（只需USB转串口）
+ 所有STM32都支持
− 只能烧录不能调试
− 需要切换BOOT引脚

1.1 ST-Link 系列

📦 ST-Link/V2

  • 接口：SWD（2线：SWDIO + SWCLK）
  • 速度：烧录约 4-8 KB/s，调试流畅
  • 驱动：Windows 需要安装 ST-Link Driver（stm32/ST-Link\USBDriver）
  • 芯片支持：STM32 全系列、STM8
  • Clone 识别：淘宝大量低价 Clone 版，功能基本一致但固件可能需更新
  • 连接方式：SWD 接口（4根线：GND / SWCLK / SWDIO / 3.3V）

📦 ST-Link/V3

  • 带宽：是 V2 的 6 倍（USB 3.0），烧录速度可达 50+ KB/s
  • 新功能：Bridge SPI / I2C / GPIO 调试扩展
  • 多探针：支持多芯片并行调试
  • 注意：cfg 文件使用 stlink 接口类型（非 stlink_dap）
接线图 — ST-Link V2 → STM32
// ST-Link V2 —— STM32 (SWD接口) // ─────────────────────────────── // Pin 1 (3.3V) ──── 3.3V (芯片供电) // Pin 7 (SWDIO) ─── SWDIO (PA13) // Pin 9 (SWCLK) ─── SWCLK (PA14) // Pin 12 (TDI) ─── (可选) // Pin 15 (TDO) ─── (可选) // Pin 20 (GND) ─── GND // ⚠️ 重要：芯片和ST-Link必须共地！ // ⚠️ 如果使用独立供电，只接 GND/SWDIO/SWCLK 三根线即可

1.2 J-Link 系列

J-Link 是 SEGGER 公司出品的高端调试器，在业内被认为是"黄金标准"。支持的芯片超过 5000 种，调试速度快，功能丰富。

⚡ J-Link vs 兼容版

  • 原装 J-Link：速度最快，支持所有功能，包含丰富的调试插件
  • J-Link OB：STM32 芯片内置的简化版 J-Link（STM32F072 等），功能受限但免费
  • 兼容 J-Link：第三方克隆，价格低但固件更新可能有问题
  • 限制：非 SEGGER 官方硬件在商业项目需购买 license
J-Link 固件升级后可能会检测到克隆并限制功能。如果使用的是兼容版，谨慎升级固件！

1.3 DAP-Link

DAP-Link 是 ARM 官方推出的开源调试方案，基于 CMSIS-DAP 协议。任何带有 MCU（推荐 LPC11U / RP2040 / STM32F1）的开发板都可以刷成 DAP-Link。

强烈推荐 DIY DAP-Link！树莓派 Pico 只要 10 元钱，刷上 DAP-Link 固件就是一个专业调试器。性价比无敌。

🛠️ 推荐 DAP-Link DIY 方案

  • 方案A：树莓派 Pico（10元）× 1块，参考 Pico DAP-Link 固件烧录
  • 方案B：STM32F103 蓝色小板（约8元），刷 DAPLink 固件
  • 方案C：NUCLEO 开发板自带 DAP-Link（直接用板载调试器）

1.4 串口 ISP

串口 ISP（In-System Programming）是最古老的 STM32 烧录方式，通过芯片内置的 Bootloader 经 UART 下载程序。零成本，只需要一个 USB 转 TTL 模块。

接线图 — USB转TTL → STM32
// USB转TTL模块 —— STM32 // ───────────────────────────── // GND ─── GND // RX ─── TX (PA9) // TX ─── RX (PA10) // ⚠️ 芯片必须进入 Bootloader 模式（详见第7节）

2 OpenOCD 安装与配置

OpenOCD（Open On-Chip Debugger）是一个开源的片上调试工具，通过调试器（ST-Link / J-Link / DAP-Link 等）与芯片通信，充当 GDB 和芯片之间的"桥梁"。

OpenOCD 就像一个"翻译官"：GDB（调试器前端）说的是高级语言，芯片（Target）说的是电路语言，OpenOCD 在中间做翻译和传递。GDB ↔ OpenOCD ↔ 调试器 ↔ 芯片。
工作原理图
┌──────────────┐ TCP:3333 ┌──────────────┐ USB ┌────────┐ │ GDB │ ◄──────────────► │ OpenOCD │ ◄────────►│ ST-Link│ │ (调试前端) │ │ (桥梁) │ │ /JLink │ └──────────────┘ └──────────────┘ └────────┘ │ │ JTAG/SWD ▼ ┌────────┐ │ STM32 │ │ Target │ └────────┘

2.1 Windows 安装

方式一：从 GitHub 下载（推荐）

① 下载最新 Release
打开 xpack-dev-tools/openocd/releases，下载 xpack-openocd-{version}-win32-x64.zip（约 20MB）。
② 解压到合适位置
建议放在 C:\Program Files\OpenOCD\ 或 D:\Tools\OpenOCD\，路径不要有中文和空格。
③ 配置环境变量（可选）
将 bin 目录添加到 PATH：
openocd -v # 验证安装

方式二：MSYS2 / MSYS2 编译安装

BASH
pacman -S mingw-w64-x86_64-openocd
④ 安装 USB 驱动（Windows）
  • ST-Link：安装 STSW-LINK009 驱动（官网下载或驱动人生自动识别）
  • J-Link：安装 SEGGER J-Link Software
  • DAP-Link：Windows 10+ 自动识别，无需手动装驱动

2.2 OpenOCD 目录结构

OpenOCD 目录结构

2.3 cfg 配置文件说明

OpenOCD 使用 TCL 脚本语言编写配置文件。三个关键组件：Interface（调试器） + Target（芯片） + Transport（传输协议）。

OpenOCD 配置文件结构
# 配置文件由三部分组成： # # ① interface：调试器硬件（interface/*.cfg） # ② transport：传输协议（swd 或 jtag） # ③ target：目标芯片（target/*.cfg 或 board/*.cfg） # 示例：ST-Link + STM32F103 source [find interface/stlink.cfg] # ① 加载调试器配置 transport select hla_swd # ② 使用 SWD 协议（STM32推荐） source [find target/stm32f1x.cfg] # ③ 加载芯片配置

2.4 ST-Link 配置详解

🔌 ST-Link 接口配置文件

  • interface/stlink.cfg — ST-Link V1/V2（最常用）
  • interface/stlink-dap.cfg — ST-Link V3
  • interface/stlink-v2.cfg — ST-Link V2 专用
  • interface/stlink-v3.cfg — ST-Link V3 专用
ST-Link + STM32F1 系列
# stm32f103.cfg — 通用配置文件 # 使用 board/ 下的配置更方便（包含了 target + often reset config） source [find interface/stlink.cfg] transport select hla_swd # 可选：调整 SWD 速率（kHz） # adapter speed 4000 # 4 MHz，高速（可能不稳定） # adapter speed 1000 # 1 MHz，中速（推荐） # adapter speed 500 # 500 kHz，低速（最稳定） source [find target/stm32f1x.cfg]
ST-Link + STM32F4 系列
# stm32f4x.cfg — STM32F4 专用 source [find interface/stlink.cfg] transport select hla_swd source [find target/stm32f4x.cfg] # 如果使用 board/ 下的配置（推荐） # source [find board/stm32f4discovery.cfg]

3 OpenOCD 命令行操作

3.1 启动连接

BASH / CMD
# 基础连接命令 openocd -f interface/stlink.cfg -f target/stm32f1x.cfg # 指定配置文件（最推荐写法） openocd -f interface/stlink.cfg -f target/stm32f1x.cfg # 指定端口（不指定则用默认：3333 GDB / 4444 Telnet / 6666 TCL） openocd -f interface/stlink.cfg -f target/stm32f1x.cfg \ -c "adapter speed 1000" # 显示详细调试信息 openocd -d3 -f interface/stlink.cfg -f target/stm32f1x.cfg # 成功后 OpenOCD 会输出： # Info : ST-Link V2 JTAG/SWD interface # Info : using swd # Info : Found flash device ... (flash auto-program enabled) # Info : stm32f1x.cpu: hardware has 6 breakpoints, 4 watchpoints # # 等待 GDB 连接后，OpenOCD 阻塞运行（不要关掉这个窗口）
OpenOCD 启动后会一直运行，直到按 Ctrl+C 退出。保持这个窗口开着，用另一个终端/窗口连接 GDB 进行调试。

3.2 Telnet 连接与命令

除了 GDB，我们还可以通过 Telnet 直接向 OpenOCD 发送命令。这是快速执行烧录、擦除操作的利器。

BASH
# 新开一个终端，连接到 OpenOCD Telnet telnet localhost 4444 # 连接成功后可以看到： # Trying ::1... # Connected to localhost. # Escape character is '^]'. # Open On-Chip Debugger # > # 常用命令（直接输入并回车） help # 查看所有命令 halt # 暂停 CPU reset # 复位芯片 shutdown # 关闭 OpenOCD

3.3 烧录文件

OpenOCD 支持三种常见格式：.elf（含调试信息）、.hex（Intel格式）、.bin（纯二进制）。

Telnet 命令
# 先连接到 OpenOCD telnet # 然后执行烧录命令： # 烧录 .elf 文件（推荐！包含调试信息） program build/myproject.elf verify reset exit # 烧录 .bin 文件（需要指定起始地址） program build/myproject.bin 0x08000000 verify reset exit # 烧录 .hex 文件（自动识别格式和地址） program build/myproject.hex verify reset exit # 参数解释： # program <file> [addr] — 烧录文件 # verify — 烧录后验证（强烈推荐开启） # reset — 烧录后复位芯片 # exit — 烧录后退出（Telnet 会断开连接）
💡 烧录流程详解
  1. halt — 先暂停 CPU（确保 Flash 不被读取）
  2. flash erase_address — 擦除对应地址的 Flash（可选，program 会自动擦除）
  3. flash write_image — 写入数据到 Flash
  4. verify_image — 读取 Flash 与原文件对比（验证正确性）
  5. reset run — 复位芯片并运行

3.4 擦除操作

Telnet 命令
# 擦除整个芯片 Flash stm32f1x mass_erase 0 # 或使用通用命令 flash erase_address 0x08000000 0x10000 # 擦除指定扇区（部分芯片支持） flash erase_sector 0 first 0 last # 擦除后验证 Flash 为空（应该全是 0xFF） mdw 0x08000000 16

3.5 读取 Flash / 内存

Telnet 命令
# 读取内存（word，32位） mdw 0x08000000 16 # 从 Flash 起始地址读取16个32位字 # 读取内存（byte，8位） mdb 0x20000000 32 # 从 RAM 起始地址读取32字节 # 读取内存（halfword，16位） mdh 0x08000000 16 # 读取外设寄存器 mdw 0x40021000 4 # RCC APB2ENR 寄存器 # 将 Flash 读出保存为 .bin 文件 dump_image backup.bin 0x08000000 0x10000 # 读取64KB

3.6 验证与比较

Telnet 命令
# 验证烧录是否正确（与源文件比对） verify_image build/myproject.elf # 比较两个内存区域 cmp 0x08000000 0x08010000 backup.bin 0x0 # 读出并保存 dump_image output.bin 0x08000000 0x10000
一键烧录脚本（Windows .bat）：把以下内容保存为 flash.bat，双击即可完成烧录。
flash.bat
@echo off cd /d "%~dp0" # 先杀掉可能占用端口的旧OpenOCD进程 taskkill /F /IM openocd.exe 2>nul # 启动 OpenOCD 并后台运行 start "OpenOCD" openocd.exe -f interface/stlink.cfg -f target/stm32f1x.cfg # 等待 OpenOCD 启动 timeout /t 3 /nobreak >nul # 通过 Telnet 自动烧录 echo program build/firmware.elf verify reset exit ^& exit | telnet localhost 4444 timeout /t 5 /nobreak >nul taskkill /F /IM openocd.exe echo Flash complete!

4 GDB 基础

GDB（GNU Debugger）是 Linux 下最强大的程序调试器，ARM 嵌入式使用专门的 arm-none-eabi-gdb。配合 OpenOCD 可以实现源码级调试、寄存器查看、内存修改等功能。

GDB 就像芯片的"监控摄像头控制台"。你可以让芯片"暂停"（暂停运行）、"单步走"（执行一条指令）、"查看变量"（看内存里的值）、"设断点"（让程序停在你关心的位置）。OpenOCD 就是连接摄像头和监控台的"网线"。

4.1 安装 arm-none-eabi-gdb

① Windows 安装
下载 GNU Arm Embedded Toolchain，安装后 bin\arm-none-eabi-gdb.exe 在 PATH 中。
验证安装
arm-none-eabi-gdb -v # 输出示例： # arm-none-eabi-gdb.exe (Arm GNU Toolchain 11.3.Rel1) 12.1 # Copyright (C) 2022 Free Software Foundation, Inc.

4.2 启动 GDB 并连接 OpenOCD

完整的调试流程分为两步：先启动 OpenOCD，再启动 GDB 并连接。

第一步：启动 OpenOCD
openocd -f interface/stlink.cfg -f target/stm32f1x.cfg # 保持运行，切换到另一个终端
第二步：启动 GDB
# 在另一个终端，进入项目目录（.elf文件所在位置） cd myproject/ # 启动 GDB，加载 .elf 文件（重要！含调试信息） arm-none-eabi-gdb build/myproject.elf # GDB 交互界面出现： # (gdb) _
第三步：在 GDB 中连接 OpenOCD
# 在 (gdb) 提示符下输入： (gdb) target remote localhost:3333 # 成功后显示： # Remote debugging using localhost:3333 # 0x08000xxx in ...
完整启动脚本（保存为 gdbinit）
# 在项目根目录创建 .gdbinit 文件 # GDB 启动时会自动加载 target remote localhost:3333 # 加载 .elf 中的符号表（可省略，openocd会自动加载） # load # 设置芯片向量表地址 set mem inaccessible-by-default off # GDB 终端设置（回车不退出，连续输出） set pagination off # 监控 Reset 引脚（复位时自动 halt） monitor reset halt

GDB 命令

5 GDB 高级技巧

5.1 条件断点

普通断点每次都停，条件断点只有满足特定条件才停。这在循环调试中非常有用。

GDB 命令
# 普通断点：循环第1000次时才停 (gdb) break delay_ms (gdb) condition 1 i == 1000 # 创建时直接带条件 (gdb) break main.c:30 if counter > 50 # 跳过前N次（忽略计数） (gdb) break my_func (gdb) ignore 1 99 # 忽略1号断点前99次触发 # 删除条件（恢复为普通断点） (gdb) condition 1 # 动态修改变量触发断点（适合状态机调试） (gdb) break handle_uart if (state == 0xAA)

5.2 观察点 (Watchpoint)

观察点不是停在某一行代码，而是当某个变量/内存的值发生变化时才停。相当于给数据装上"监控摄像头"。

GDB 命令
# 观察点：当 counter 的值被写入时暂停 (gdb) watch counter # 硬件观察点：当 0x20000000 地址的内存被写入时暂停 (gdb) watch *(int*)0x20000000 # 读取观察点：当变量被读取时暂停（消耗一个硬件断点） (gdb) rwatch counter # 访问观察点：读或写都触发 (gdb) awatch counter # 查看当前所有观察点 (gdb) info watchpoints # 删除观察点 (gdb) delete 3
ARM Cortex-M 最多只有 4 个硬件观察点（watchpoint）。别设太多，否则会报错。watch variable 会消耗一个硬件断点资源。

5.3 多线程调试

STM32 使用 RTX / FreeRTOS 时，程序可能有多个任务同时运行。GDB 支持在线程间切换和单独控制。

GDB 命令
# 查看所有线程 (gdb) info threads # 输出示例： # Id Target Id Frame # * 1 Thread 1 (Name) delay_ms () at main.c:25 # 2 Thread 2 vTaskDelay () at freertos/tasks.c:1000 # 3 Thread 3 UART_IRQHandler () at usart.c:50 # 切换到指定线程（* 表示当前活跃线程） (gdb) thread 2 # 在所有线程的指定函数设断点 (gdb) break UART_IRQHandler thread 3 # 只让某个线程继续（冻结其他线程） (gdb) thread apply 2 next (gdb) thread apply all bt # 查看所有线程的调用栈 # 冻结/解冻线程 (gdb) thread 2 (gdb) disable scheduler-locking on # 只让当前线程运行 (gdb) disable scheduler-locking off # 恢复多线程调度

5.4 调试 FreeRTOS

FreeRTOS 使用软件定时器/任务调度，普通的 next 可能跳进系统调度导致调试体验很差。以下技巧可以改善：

FreeRTOS 调试配置

7 串口 ISP 烧录

串口 ISP（In-System Programming）是 STM32 内置的 Bootloader 提供的功能。所有 STM32 芯片都内置了 UART Bootloader，只要把 BOOT0 引脚拉高，就能从内部 ROM 区（0x1FFF0000）启动这个 bootloader，通过串口下载程序。

这是零硬件成本的烧录方案！只要你有一块 STM32 芯片 + 一个 USB 转 TTL 模块（某宝 5 元），就能烧录。不需要任何调试器！
把 STM32 想象成一台有"两个启动盘"的电脑。正常运行时从 Flash（硬盘）启动（BOOT0=0），刷固件时从 ROM Bootloader（内置恢复系统）启动（BOOT0=1）。串口 ISP 就是通过内置恢复系统往硬盘写数据。

7.1 BOOT0 引脚操作

这是串口 ISP 最关键的地方！STM32 有三种启动模式，由 BOOT0 和 BOOT1 两个引脚控制：

BOOT0BOOT1启动位置用途
0xFlash (0x08000000)正常运行
10System Memory (0x1FFF0000)串口 ISP / USB DFU
11SRAM (0x20000000) SRAM 调试（掉电丢失）
烧录操作流程
  1. 断电，将 BOOT0 跳线帽插到 3.3V（高电平），BOOT1 保持接地
  2. 通电，芯片进入 System Memory 模式（ROM Bootloader）
  3. 在电脑上打开 FlyMcu / stm32flash，选择 COM 口，波特率 57600 或 115200
  4. 点击"开始编程"，等待下载完成
  5. 断电，将 BOOT0 跳线帽恢复到 GND（正常运行模式）
  6. 重新通电，程序从 Flash 启动
BOOT0 跳线帽必须在断电状态下切换！带电插拔可能导致芯片损坏或进入未知状态。

8 常见问题与解决

8.1 OpenOCD 连不上

❌ 症状：OpenOCD 启动后报错 "Error: open failed"

  • 驱动未安装：设备管理器中 ST-Link 有黄色感叹号 → 安装 STSW-LINK009 驱动
  • 权限问题（Linux）：sudo openocd ... 或将用户加入 dialout 组
  • 端口被占用：netstat -ano | findstr 3333 查看是否已有 OpenOCD 占用
  • USB 线太长/质量差：USB 线超过 1.5m 或只接了电源线没接数据线
  • 调试器固件损坏：Clone 版 ST-Link 固件损坏，需要重新刷固件（用 ST-Link Utility 修复）
排查命令
# Windows：查看 ST-Link 是否被识别 lsusb # Git Bash 下，或在设备管理器查看 # 检查端口占用 netstat -ano | findstr "3333 4444"

❌ 症状：OpenOCD 连接成功但 GDB 连接时报错 "Remote connection refused"

  • OpenOCD 未完全启动完成 → 等 2-3 秒再连
  • 防火墙阻止了 3333 端口 → 关闭防火墙或允许 openocd.exe
  • 端口写错了 → 检查 target remote localhost:3333

❌ 症状："stm32f1x.cpu: hardware has 0 breakpoints, 0 watchpoints"

  • 这是正常的（部分 Clone 调试器不会报告断点数量），不影响使用
  • 真实 ARM Cortex-M 有 6 个硬件断点 + 4 个观察点
  • 如果实际无法设置断点，可能是芯片已被读保护（见下文）

8.2 GDB 超时 / 连接失败

❌ 症状：GDB 报 "Timeout waiting for target"

  • SWD 速率太高：adapter speed 1000 → 改成 adapter speed 500
  • 芯片掉电：目标板供电不足或断电，OpenOCD 连不上 halted 状态的芯片
  • 接线问题：SWDIO / SWCLK 线路太长（超过 20cm）或接反
  • SWDIO 未接上拉电阻：部分板子需要 10kΩ 上拉（STM32 内部已有，但部分开发板去掉了）

❌ 症状："undefined debug reason 7 - target needs reset"

  • GDB 先于 OpenOCD 完成复位 → 在 .gdbinit 中加 monitor reset halt
  • 或 (gdb) monitor reset halt 手动复位

8.3 芯片锁死 / 读保护

如果芯片的 Option Bytes 被错误配置（例如把 SWD 引脚配置为普通 GPIO，或开启了读保护 RDP），芯片可能会"锁死"无法连接。此时需要用串口 ISP 或 ST-Link Utility 强制擦除。

🔓 解除读保护（STM32F1）

  • 方法A：FlyMcu → 芯片解锁 → 擦除 → 重新烧录
  • 方法B：ST-Link Utility → Target → Option Bytes → 关闭 RDP
  • 方法C：串口 ISP 进入 Bootloader → 全片擦除（会清除 Flash）

🔓 解除 SWD 禁用（引脚配置错误）

  • 有些用户不小心把 PA13/PA14（SWDIO/SWCLK）配置成了普通 GPIO 并编程
  • 解决：拉高 BOOT0 → 进入串口 ISP → 重新烧录正确固件
  • 或者：按住复位键 → 连接 OpenOCD → 在复位释放前执行 monitor reset halt

8.5 调试断点失效

⚠️ 断点无法生效的常见原因

  • 优化等级太高：-Og 或 -O2 优化后部分变量被优化掉，断点设在被优化代码上会失效 → 改为 -O0
  • Flash 烧录后立即运行：用 monitor reset halt 确保芯片停在断点处
  • 中断里设断点：中断处理函数中设断点可能触发其他问题，小心使用
  • 硬件断点耗尽：Cortex-M 只有 6 个硬件断点，设太多会失败 → 用软件断点 break 替代（需芯片支持）
确保调试信息的 CMakeLists.txt / Makefile
# 调试编译选项（不优化，保留所有调试信息） CFLAGS += -Og -g3 -ggdb3 # -Og : 最高优化等级（但保留调试信息） # -g3 : 生成最详细的 DWARF 调试信息 # -ggdb3: 生成 GDB 友好格式 # 正式发布编译选项 CFLAGS += -O2 -g0 # 不生成调试信息
