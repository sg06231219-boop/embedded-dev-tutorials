🔍 环境一键验证脚本

复制下面的脚本到PowerShell运行，自动检测所有工具是否配置正确：

# 保存为 check_env.ps1 或直接复制运行
# 自动检测Python命令（python/python3/python3.14等）
$pyCmd = $null
foreach ($cmd in @('python3.14','python3','python')) {
    try { $null = & $cmd --version 2>&1; $pyCmd = $cmd; break } catch {}
}
if (-not $pyCmd) { Write-Host "❌ Python未找到，请确认已安装并加入PATH" -ForegroundColor Red }

$tools = @(
    @{name='ARM GCC'; cmd='arm-none-eabi-gcc --version'},
    @{name='CMake'; cmd='cmake --version'},
    @{name='Make'; cmd='make --version'},
    @{name='Python'; cmd="$pyCmd --version"},
    @{name='pyserial'; cmd="$pyCmd -m serial.tools.list_ports -h"},
    @{name='esptool'; cmd="$pyCmd -m esptool version"}
)

Write-Host "`n=== 嵌入式环境检测 ===`n" -ForegroundColor Cyan
foreach ($t in $tools) {
    Write-Host "检测 $($t.name)... " -NoNewline
    try {
        $result = Invoke-Expression $t.cmd 2>&1
        if ($LASTEXITCODE -eq 0 -or $result -match '\d') {
            Write-Host "✅ OK" -ForegroundColor Green
        } else {
            Write-Host "❌ 失败" -ForegroundColor Red
        }
    } catch {
        Write-Host "❌ 未安装" -ForegroundColor Red
    }
}
Write-Host "`n检测完成！" -ForegroundColor Cyan

• 如果Make检测失败，说明GnuWin32的bin目录未加入PATH，手动添加：C:\Program Files (x86)\GnuWin32\bin
• Python Scripts目录也需检查：C:\Users\你的用户名\AppData\Local\Programs\Python\Python3xx\Scripts
• pyserial和esptool装在哪个Python下，那个Scripts目录就要加PATH

第1章：ARM GCC编译工具链详解

ARM GCC不是"一个"程序，而是一整套工具。每个工具负责编译流程的一个环节，就像一条流水线——钢材进来，汽车出去。

💡 类比：编译就像翻译一本外文小说——预处理=整理草稿（去掉注释、展开宏）；编译=逐句翻译成目标语言；汇编=查字典把句子变成字母编码；链接=把散页装订成册、补上目录和索引。

编译四阶段

阶段工具输入输出做了什么
1. 预处理cpp (C Preprocessor).c + .h.i展开宏、条件编译、去注释
2. 编译cc1 / cc1plus.i.sC/C++ → ARM汇编
3. 汇编as (Assembler).s.o汇编 → 机器码(目标文件)
4. 链接ld (Linker).o + 库.elf合并目标文件、分配地址
# 一步到位（最常用）
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -O2 -Wall main.c -o main.elf

# 分步执行（学习用）
arm-none-eabi-gcc -E main.c -o main.i          # 只预处理
arm-none-eabi-gcc -S main.i -o main.s           # 只编译到汇编
arm-none-eabi-as -mcpu=cortex-m3 main.s -o main.o  # 只汇编
arm-none-eabi-ld main.o -T stm32.ld -o main.elf    # 只链接

核心工具详解

编译参数速查必须理解
每次编译都会用到这些参数，搞懂它们是读懂所有构建命令的前提：
参数含义为什么需要
-mcpu=cortex-m3目标CPU型号STM32F103=M3, F407=M4, H743=M7
-mthumb使用Thumb指令集代码更小（16位指令），ARM单片机必选
-O2优化等级O0=不优化(调试用), O2=速度优化, Os=体积优化
-Wall开启所有警告帮你提前发现Bug，必须开
-ffunction-sections每个函数单独section配合--gc-sections删除未用函数
-fdata-sections每个数据单独section同上，删除未用变量
-g3调试信息等级3GDB调试需要，等级3=宏信息也包含
--specs=nano.specs使用精简C库代码体积减半！嵌入式必选
--specs=nosys.specs无系统调用裸机程序不需要OS接口
-Wl,--gc-sections链接时删除未用代码与-ffunction-sections配合，省Flash
-Wl,-Map=xx.map生成map文件查看函数地址和大小，排错利器
arm-none-eabi-gcc编译器驱动
主入口，自动调用预处理→编译→汇编→链接四个阶段。平时99%的情况用这个就够了。
# 常用编译选项
-mcpu=cortex-m3    # 指定CPU架构（M3/M4/M7等）
-mthumb            # 使用Thumb指令集（代码更小）
-O2                # 优化等级（O0/O1/O2/Os）
-Wall              # 开启所有常见警告
-ffunction-sections # 每个函数单独一个section（配合链接器优化）
-ggdb3             # 生成GDB调试信息
-I./inc            # 头文件搜索路径
-DSTM32F103        # 定义预处理宏
arm-none-eabi-objcopy格式转换
把ELF格式转成BIN/HEX，烧录器需要的是这些"纯数据"格式。
# ELF → BIN（纯二进制，最常用）
arm-none-eabi-objcopy -O binary main.elf main.bin

# ELF → HEX（Intel HEX格式，带地址信息）
arm-none-eabi-objcopy -O ihex main.elf main.hex

# 查看各段信息
arm-none-eabi-objcopy -O binary -j .isr_vector main.elf vector.bin
arm-none-eabi-objdump反汇编器
查看目标文件的汇编代码、段信息。调试时看CPU到底在执行什么。
# 反汇编ELF文件（带C源码对照）
arm-none-eabi-objdump -d -S main.elf > main.lst

# 查看段头部信息
arm-none-eabi-objdump -h main.elf

# 只反汇编某个函数
arm-none-eabi-objdump -d main.elf | Select-String -Pattern "main>:" -Context 0,20
arm-none-eabi-size空间分析
查看编译后的代码占多少Flash和RAM，嵌入式开发必须关注！
arm-none-eabi-size main.elf
#    text    data     bss     dec     hex filename
#   12340     120    2048   14508    38ac main.elf
# Flash占用 = text + data = 12460 字节
# RAM占用  = data + bss  = 2168 字节
arm-none-eabi-nm符号表
列出所有符号（函数、全局变量）及其地址。
# 按地址排序查看所有符号
arm-none-eabi-nm -n main.elf

# 只看未定义符号
arm-none-eabi-nm -u main.elf

# 查看某个函数地址
arm-none-eabi-nm main.elf | findstr "main"

.map 文件

链接器输出的"地图"——每个函数、变量被放在了哪个地址，各段大小是多少。排错时非常有用。

# 生成map文件
arm-none-eabi-gcc ... -Wl,-Map=main.map main.elf

# map文件关键信息：
# .text    0x08000000    12340  ← 代码段
# .data    0x20000000      120  ← 数据段
# .bss     0x20000078     2048  ← BSS段
# main     0x08000120          ← main函数地址
🔍 常见问题：编译成功但程序跑飞
90%是链接脚本(.ld)里Flash/RAM起始地址写错了。STM32F103C8T6的Flash从0x08000000开始，RAM从0x20000000开始。检查.map文件确认地址。
💡 小技巧：编译时永远加 -Wl,-Map=output.map 和 -g，这两个选项不要省。
📝 完整示例：编译并分析一个STM32项目
输入命令：
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -O2 -Wall -g3 \
    -ffunction-sections -fdata-sections \
    -I./inc -DSTM32F103 \
    src/main.c src/stm32f1xx_it.c -T STM32F103C8Tx.ld \
    -Wl,--gc-sections -Wl,-Map=app.map --specs=nano.specs \

arm-none-eabi-objcopy -O binary app.elf app.bin

第2章：CMake构建系统

CMake不直接编译代码——它是"构建文件生成器"。你写CMakeLists.txt描述项目结构，CMake帮你生成Makefile，再用make编译。

💡 类比：CMake就像建筑设计师，你告诉他"我要3层楼、2个卧室"（CMakeLists.txt），他画出施工图纸（Makefile），施工队（make）按图纸盖房子（编译）。

CMakeLists.txt 最小模板

cmake_minimum_required(VERSION 3.20)
project(stm32_blink C ASM)

set(SOURCES src/main.c src/stm32f1xx_it.c src/system_stm32f1xx.c)

include_directories(inc CMSIS)

add_compile_options(-mcpu=cortex-m3 -mthumb -Wall -ffunction-sections -fdata-sections)

add_link_options(-mcpu=cortex-m3 -mthumb -Wl,--gc-sections
    -T${CMAKE_SOURCE_DIR}/STM32F103C8Tx.ld
    -Wl,-Map=${PROJECT_NAME}.map)

add_executable(${PROJECT_NAME}.elf ${SOURCES})

add_custom_command(TARGET ${PROJECT_NAME}.elf POST_BUILD
    COMMAND ${CMAKE_OBJCOPY} -O binary ${PROJECT_NAME}.elf ${PROJECT_NAME}.bin
    COMMAND ${CMAKE_OBJCOPY} -O ihex ${PROJECT_NAME}.elf ${PROJECT_NAME}.hex
    COMMAND ${CMAKE_SIZE} ${PROJECT_NAME}.elf)

交叉编译工具链文件

# toolchain-arm.cmake
set(CMAKE_SYSTEM_NAME Generic)
set(CMAKE_SYSTEM_PROCESSOR ARM)

set(TOOLCHAIN_PREFIX arm-none-eabi-)
set(CMAKE_C_COMPILER ${TOOLCHAIN_PREFIX}gcc)
set(CMAKE_CXX_COMPILER ${TOOLCHAIN_PREFIX}g++)
set(CMAKE_ASM_COMPILER ${TOOLCHAIN_PREFIX}gcc)
set(CMAKE_AR ${TOOLCHAIN_PREFIX}ar)
set(CMAKE_OBJCOPY ${TOOLCHAIN_PREFIX}objcopy)
set(CMAKE_OBJDUMP ${TOOLCHAIN_PREFIX}objdump)
set(CMAKE_SIZE ${TOOLCHAIN_PREFIX}size)

set(CMAKE_TRY_COMPILE_TARGET_TYPE STATIC_LIBRARY)
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)

set(CMAKE_C_COMPILER "C:/Program Files (x86)/Arm GNU Toolchain arm-none-eabi/14.2 rel1/bin/arm-none-eabi-gcc.exe")

第3章：Makefile实战

Makefile是嵌入式项目最经典的构建方式——虽然CMake更现代，但大量开源项目仍在用Makefile，必须会读会写。

💡 类比：Makefile就像菜谱——target=菜名（"红烧肉"），prerequisites=食材（五花肉、酱油），recipe=做法（切肉→焯水→炖煮）。make就是你雇的厨师，按菜谱一步步做。

Makefile基本规则

# 格式（recipe必须用TAB缩进！）
target: prerequisites
	recipe

# 例子
main.o: main.c main.h
	arm-none-eabi-gcc -c -mcpu=cortex-m3 -mthumb main.c -o main.o

变量与自动变量

CC = arm-none-eabi-gcc
CFLAGS = -mcpu=cortex-m3 -mthumb -Wall -O2 -g
TARGET = myproject

# 自动变量
# $@ = 目标名    $< = 第一个依赖    $^ = 所有依赖
main.o: main.c main.h
	$(CC) $(CFLAGS) -c $< -o $@

模式规则

# 通用规则：所有.c → .o
%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@

# 链接：所有.o → .elf
$(TARGET).elf: $(OBJS)
	$(CC) $(LDFLAGS) $^ -o $@

# ====== STM32F103 Makefile ======
CC = arm-none-eabi-gcc
OBJCOPY = arm-none-eabi-objcopy
SIZE = arm-none-eabi-size
TARGET = blink
MCU = cortex-m3

C_SOURCES = $(wildcard src/*.c)
INCLUDES = -Iinc -Idrivers/CMSIS

CFLAGS  = -mcpu=$(MCU) -mthumb -Wall -O2 -g3
CFLAGS += -ffunction-sections -fdata-sections $(INCLUDES)
CFLAGS += -DSTM32F103C8T6

LDFLAGS  = -mcpu=$(MCU) -mthumb -Wl,--gc-sections
LDFLAGS += -TSTM32F103C8Tx.ld -Wl,-Map=$(TARGET).map
LDFLAGS += --specs=nano.specs --specs=nosys.specs

OBJS = $(C_SOURCES:.c=.o)

.PHONY: all clean flash

all: $(TARGET).bin $(TARGET).hex
	@echo "Build complete!"
	$(SIZE) $(TARGET).elf

$(TARGET).elf: $(OBJS)
	$(CC) $(LDFLAGS) $^ -o $@

$(TARGET).bin: $(TARGET).elf
	$(OBJCOPY) -O binary $< $@

$(TARGET).hex: $(TARGET).elf
	$(OBJCOPY) -O ihex $< $@

%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@

clean:
	powershell -Command "Remove-Item src/*.o $(TARGET).elf $(TARGET).bin $(TARGET).hex $(TARGET).map -ErrorAction SilentlyContinue"
# 注：嵌入式Makefile通常在Windows PowerShell环境运行
# 如果你在Linux/WSL下，改用：
# 	rm -f $(OBJS) $(TARGET).elf $(TARGET).bin $(TARGET).hex $(TARGET).map

flash: $(TARGET).bin
	st-flash write $(TARGET).bin 0x08000000

第4章：串口通信 — 单片机与电脑对话

串口(UART)是嵌入式开发最基础、最常用的通信方式。单片机的printf输出、调试信息、AT指令控制，全靠串口。可以说：不会串口，嵌入式寸步难行。

💡 类比：串口就像两个人用对讲机通话——波特率=说话速度（必须一致才能听懂），数据位=每个词的长度，停止位=说完了喘口气，校验位=确认对方听清了。速度不一样就全是乱码，就像一个人说中文、一个说英文。

串口参数

参数常见值说明
波特率(Baud Rate)9600 / 115200通信速度，双方必须一致
数据位(Data Bits)8每个字节的有效位数
停止位(Stop Bits)1帧结束标志
校验位(Parity)None简单错误检测，通常不用
流控(Flow Control)None硬件/软件流控，通常不用

#define GPIOC_CRH    (*(volatile uint32_t*)0x40011004)
#define GPIOC_ODR    (*(volatile uint32_t*)0x4001100C)

void delay(volatile uint32_t count) {
    while(count--);

int main(void) {
    // 步骤1：使能GPIOC时钟（必须先开时钟，否则引脚不工作！）
    RCC_APB2ENR |= (1 << 4);  // bit4 = IOPCEN

    // CRH控制PC8-PC15，PC13对应bits [23:20]

    // 步骤3：循环翻转LED
    while (1) {
        GPIOC_ODR ^= (1 << 13);  // 翻转PC13（XOR操作）
        delay(500000);            // 约500ms延时（72MHz下）
    }

第8章：FreeRTOS入门

裸机程序（super loop）就像一个人干所有事——烧水、切菜、炒菜、摆盘，只能一件一件来。RTOS让多个任务同时运行（实际是快速切换），就像雇了多个厨师各司其职。

💡 类比：RTOS就像餐厅后厨——任务(Task)=厨师（各干各的活），队列(Queue)=传菜窗口（任务间传递数据），信号量(Semaphore)=叫号器（通知"饭好了"），互斥量(Mutex)=独占灶台（同一时间只有一个厨师用）。

void HardFault_Handler(void) {
    __asm volatile (
        "TST LR, #4
"          // 检查使用哪个栈
        "ITE EQ
"
        "MRSEQ R0, MSP
"       // 主栈
        "MRSNE R0, PSP
"       // 进程栈
        "B Hard_Fault_Handler
" // 跳到C函数
    );
}

    uint8_t temp_dec;        // 温度小数部分

// dht11.c
#include "dht11.h"

static void delay_us(uint32_t us) {
    uint32_t ticks = us * (SystemCoreClock / 1000000) / 5;
    while (ticks--);
}

uint8_t DHT11_Read(DHT11_Data_t *data, GPIO_TypeDef *port, uint16_t pin) {
    uint8_t buf[5] = {0};
    GPIO_InitTypeDef GPIO_InitStruct = {0};

    // 主机发送起始信号（拉低18ms）
    GPIO_InitStruct.Pin = pin;
    GPIO_InitStruct.Mode = GPIO_MODE_OUTPUT_PP;
    GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_HIGH;
    HAL_GPIO_Init(port, &GPIO_InitStruct);
    HAL_GPIO_WritePin(port, pin, GPIO_PIN_RESET);
    HAL_Delay(18);
    HAL_GPIO_WritePin(port, pin, GPIO_PIN_SET);
    delay_us(30);

    // 切换为输入
    GPIO_InitStruct.Mode = GPIO_MODE_INPUT;
    GPIO_InitStruct.Pull = GPIO_PULLUP;
    HAL_GPIO_Init(port, &GPIO_InitStruct);

    // 等待DHT11响应
    uint32_t timeout = 100;
    while (HAL_GPIO_ReadPin(port, pin) && timeout--) delay_us(1);
    if (timeout == 0) return 1;  // 无响应

    // 接收40位数据
    for (int i = 0; i < 5; i++) {
        for (int j = 0; j < 8; j++) {
            timeout = 100;
            while (!HAL_GPIO_ReadPin(port, pin) && timeout--) delay_us(1);
            delay_us(40);
            if (HAL_GPIO_ReadPin(port, pin)) {
                buf[i] |= (1 << (7 - j));
                timeout = 100;
                while (HAL_GPIO_ReadPin(port, pin) && timeout--) delay_us(1);
            }
        }
    }

    // 校验
    if (buf[4] != (uint8_t)(buf[0] + buf[1] + buf[2] + buf[3]))
        return 2;  // 校验失败

    data->humidity_int = buf[0];
    data->humidity_dec = buf[1];
    data->temp_int     = buf[2];
    data->temp_dec     = buf[3];
    return 0;  // 成功
}

            printf("DHT11 error: %d\n", ret);
        }
        HAL_Delay(2000);  // DHT11最小间隔2秒
    }
}

启动文件(startup)说明

启动文件用汇编编写，做三件事：①初始化堆栈指针 ②初始化中断向量表 ③调用SystemInit()和main()

// startup_stm32f103xx.s 关键部分
  .section .isr_vector,"a",%progbits
  .word _estack              // 初始栈指针（RAM末尾）
  .word Reset_Handler         // 复位向量
  .word NMI_Handler
  .word HardFault_Handler
  .word MemManage_Handler
  .word BusFault_Handler
  // ... 更多中断向量 ...

Reset_Handler:
  ldr sp, =_estack           // 设置栈指针
  bl SystemInit              // 初始化时钟
  bl __libc_init_array       // C库初始化
  bl main                    // 跳转到main()
  bx lr                      // main不应返回

/* STM32F103C8Tx.ld - 完整版 */
ENTRY(Reset_Handler)

_estack = ORIGIN(RAM) + LENGTH(RAM);  /* 栈顶 = RAM末尾 */

MEMORY {
  FLASH (rx) : ORIGIN = 0x08000000, LENGTH = 64K
  RAM (rwx)  : ORIGIN = 0x20000000, LENGTH = 20K

  .isr_vector : {
    . = ALIGN(4);
    KEEP(*(.isr_vector))
    . = ALIGN(4);

    *(.glue_7)
    *(.glue_7t)
    *(.eh_frame)
    KEEP(*(.init))
    KEEP(*(.fini))
    . = ALIGN(4);
    _etext = .;
  } > FLASH

  .rodata : {
    . = ALIGN(4);
    *(.rodata)
    *(.rodata*)
    . = ALIGN(4);
  } > FLASH

  .ARM.extab : { *(.ARM.extab* .gnu.linkonce.armextab.*) } > FLASH
  .ARM : {
    __exidx_start = .;
    *(.ARM.exidx*)
    __exidx_end = .;
  } > FLASH

  .data : {
    . = ALIGN(4);
    _sdata = .;
    *(.data)
    *(.data*)
    . = ALIGN(4);

  } > RAM

  /DISCARD/ : { *(.ARM.attributes) }

常用位操作宏

// 位操作 — 嵌入式开发天天用
#define BIT(x)          (1U << (x))           // 第x位置1
#define SET_BIT(reg, x)  ((reg) |= BIT(x))     // 置位
#define CLR_BIT(reg, x)  ((reg) &= ~BIT(x))   // 清零
#define TOG_BIT(reg, x)  ((reg) ^= BIT(x))     // 翻转
#define GET_BIT(reg, x)  (((reg) >> (x)) & 1)  // 读取

// 多位操作
#define SET_MASK(reg, mask)   ((reg) |= (mask))
#define CLR_MASK(reg, mask)   ((reg) &= ~(mask))

// 寄存器写入（防编译器优化）
#define REG32(addr)  (*(volatile uint32_t *)(addr))
#define REG16(addr)  (*(volatile uint16_t *)(addr))
#define REG8(addr)   (*(volatile uint8_t *)(addr))

// 示例：直接操作STM32寄存器
#define RCC_APB2ENR  REG32(0x40021018)
#define GPIOC_CRH    REG32(0x40011004)
#define GPIOC_ODR    REG32(0x4001100C)

// 使能GPIOC时钟
SET_BIT(RCC_APB2ENR, 4);    // RCC_APB2ENR bit4 = IOPCEN
// PC13推挽输出
GPIOC_CRH = (GPIOC_CRH & ~0x00F00000) | 0x00300000;
// PC13输出高
SET_BIT(GPIOC_ODR, 13);

❓ 常见问题 FAQ

Q: STM32和Arduino哪个先学？

A: 如果你目标是做产品，先学STM32（寄存器级理解）；如果想快速出原型，先学Arduino。建议：先STM32打基础，Arduino当工具。

Q: 需要学51单片机吗？

A: 不必须。51架构古老，但教学资源丰富。如果学校课程用51，跟着学；否则直接STM32更高效。

Q: HAL库和寄存器操作哪个好？

A: 初学用HAL库快速上手，但要理解底层寄存器原理。工作后两种都要会。

Q: C语言要学到什么程度？

A: 至少掌握：指针、结构体、位操作、volatile关键字、回调函数。不需要学C++面向对象。
