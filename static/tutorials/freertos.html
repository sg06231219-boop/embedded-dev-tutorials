1. RTOS概念

1.1 为什么需要RTOS？

在嵌入式系统开发中，当你的程序变得越来越复杂时，传统的"裸机"编程方式会遇到很多问题。RTOS（Real-Time Operating System，实时操作系统）就是为了解决这些问题而诞生的。

想象一个餐厅厨房：

• 裸机编程就像一个厨师必须按顺序做完所有事：先洗菜→切菜→炒菜→上菜→洗碗→擦桌子...如果突然有VIP客人来了，也得等当前的事情做完才能响应。

• RTOS编程就像有多个厨师（任务）同时工作，还有个厨师长（调度器）在协调：洗菜工专心洗菜，炒菜师傅专心炒菜，VIP来了厨师长立马安排专人服务，其他厨师继续干活。

裸机编程的痛点

  • 阻塞等待问题：延时函数（如 HAL_Delay()）会完全阻塞CPU，期间无法响应任何事件
  • 功能耦合严重：所有功能写在一个 while(1) 里，代码难以维护和扩展
  • 实时性差：紧急事件无法立即响应，必须等当前代码执行完毕
  • 资源管理混乱：多个模块共享变量时，需要手动管理访问顺序，容易出错

RTOS带来的优势

  • 多任务并发：多个任务"同时"运行，CPU利用率大幅提升
  • 实时响应：高优先级任务可以抢占CPU，保证关键任务的时效性
  • 模块解耦：每个功能独立成任务，代码结构清晰，易于维护
  • 丰富的同步机制：信号量、互斥量、队列等，让任务间通信变得简单可靠

1.2 裸机 vs RTOS 对比

特性 裸机编程 RTOS编程
执行模式 顺序执行，死循环 多任务并行，抢占调度
延时处理 CPU空转等待 任务阻塞，CPU执行其他任务
事件响应 轮询检测，延迟不确定 中断+信号量，实时响应
代码结构 单一大循环，耦合严重 任务独立，模块化清晰
资源开销 极低，无额外开销 需要RAM存储任务栈和内核数据
开发难度 简单程序容易，复杂程序困难 需要学习RTOS概念，但复杂项目更易管理

什么时候该用RTOS？

• 系统需要同时处理多个独立功能（如：按键扫描+显示刷新+传感器采集+通信）

• 对实时性有要求（必须在指定时间内响应事件）

• 代码复杂度较高，裸机难以维护

什么时候不需要RTOS？

• 简单的单一功能程序（如：一个LED闪烁、简单的传感器读取）

• RAM资源极度有限（FreeRTOS最小需要几KB RAM）

• 对成本极其敏感的低端MCU

1.3 任务调度原理

RTOS的核心是任务调度器（Scheduler）。它负责决定哪个任务获得CPU的使用权。

基本概念

  • 任务（Task）：一个独立的执行单元，拥有自己的栈空间和优先级
  • 调度器（Scheduler）：决定何时切换任务、切换到哪个任务的内核组件
  • 上下文切换（Context Switch）：保存当前任务状态、恢复下一个任务状态的过程
  • 时间片（Time Slice）：每个任务获得CPU的时间配额，通常由SysTick中断触发
任务A运行
→
时间片到期/阻塞
→
上下文切换
→
任务B运行

调度器工作流程

SysTick中断触发

每隔固定时间（如1ms），系统定时器产生中断，通知调度器检查是否需要切换任务。

检查就绪任务

调度器在就绪列表中查找优先级最高的任务。如果当前运行任务不是最高优先级，或者时间片用完，则触发切换。

保存当前任务上下文

将当前任务的寄存器值（R0-R15、xPSR等）保存到它的栈中，记录栈指针位置。

恢复下一个任务上下文

从目标任务的栈中恢复寄存器值，加载PC指针，任务从中断点继续执行。

上下文切换就像看书时被打断：

你在看一本书（任务A），突然有人叫你（中断），你需要在当前页折角、记住看到哪里（保存上下文），然后去处理别的事（切换到任务B）。处理完后，你回来翻到折角的那页继续看（恢复上下文）。

1.4 抢占式 vs 协作式调度

🔴 抢占式调度（Preemptive）

特点：高优先级任务可以随时打断低优先级任务

  • 由硬件中断+内核定时触发
  • 响应时间可预测
  • 适合实时性要求高的场景
  • 需要注意资源共享问题
FreeRTOS默认
实时性好

🟡 协作式调度（Cooperative）

特点：任务主动让出CPU，调度器才会切换

  • 任务必须调用 taskYIELD() 或阻塞函数
  • 响应时间取决于任务协作
  • 无竞争条件，代码简单
  • 一个任务卡死，整个系统卡死
配置选择
代码简单

抢占式调度时序图示例

时间线 →
┌────────────────────────────────────────────────────────────┐
│ 任务A (低优先级)  │██████│      │████████│      │████      │
├────────────────────────────────────────────────────────────┤
│ 任务B (高优先级)  │      │██████│        │██████│          │
├────────────────────────────────────────────────────────────┤
│ 说明              │ A运行 │ B抢占 │ A继续  │ B再次 │ A继续   │
│                   │       │       │        │ 抢占   │         │
└────────────────────────────────────────────────────────────┘
关键事件：B任务就绪 → 立即抢占A → B完成后A恢复

抢占式调度的注意事项：

• 多个任务访问共享资源时，必须使用互斥量或信号量保护

• 中断服务函数中调用FreeRTOS API必须使用 FromISR 结尾的版本

• 任务临界区（关闭调度的时间）要尽可能短

2. FreeRTOS移植

FreeRTOS是一个开源的实时操作系统，需要移植到你的目标MCU上才能使用。本节介绍三种主流的移植方式。

2.1 方式一：STM32CubeMX添加（推荐）

最简单
官方支持

对于STM32用户，CubeMX提供了一键添加FreeRTOS的功能，自动完成配置和代码生成。

打开CubeMX工程

启动STM32CubeMX，打开你的现有工程或创建新工程。

添加FreeRTOS中间件

在左侧导航栏找到 Middleware → FREERTOS，勾选启用。

选择接口模式：CMSIS_V1（经典）或 CMSIS_V2（新版，推荐）

配置FreeRTOS参数

在Configuration选项卡中配置关键参数：

  • Tick Rate (Hz)：系统心跳频率，默认1000Hz（1ms）
  • MAX_PRIORITIES：最大优先级数，默认7
  • MINIMAL_STACK_SIZE：最小任务栈大小，默认128字（512字节）
  • TOTAL_HEAP_SIZE：堆内存大小，根据MCU RAM配置

创建任务

在 Tasks and Queues 选项卡点击 Add 添加任务：

  • Task Name：任务名称
  • Stack Size：栈大小（单位：字）
  • Priority：优先级（数字越大越高）
  • Entry Function：任务函数名

生成代码

点击 GENERATE CODE，CubeMX会自动生成FreeRTOS相关代码。

生成的文件位于 Middlewares/Third_Party/FreeRTOS/ 和 Core/Src/freertos.c

CubeMX生成的核心文件：

• freertos.c - 任务定义和初始化，你的主要编辑文件

• FreeRTOSConfig.h - 内核配置参数

• main.c - 自动调用 MX_FREERTOS_Init() 启动调度器

#ifndef FREERTOS_CONFIG_H

// MCU相关配置
#define configCPU_CLOCK_HZ          72000000UL   // CPU主频
#define configTICK_RATE_HZ          1000        // 心跳频率

// 内存配置
#define configTOTAL_HEAP_SIZE       40960       // 堆大小(字节)
#define configMINIMAL_STACK_SIZE    128         // 最小栈(字)

// 调度配置
#define configMAX_PRIORITIES        7           // 优先级数量
#define configUSE_PREEMPTION        1           // 启用抢占
#define configUSE_TIME_SLICING      1           // 启用时间片

// 功能开关
#define configUSE_MUTEXES           1
#define configUSE_COUNTING_SEMAPHORES 1
#define configUSE_RECURSIVE_MUTEXES  1
#define configUSE_TIMERS            1

// 中断优先级配置（Cortex-M）
#define configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY 5
#define configKERNEL_INTERRUPT_PRIORITY         255

// 钩子函数
#define configCHECK_FOR_STACK_OVERFLOW 2
#define configUSE_MALLOC_FAILED_HOOK    1

// 断言
#define configASSERT( x ) if( ( x ) == 0 ) { taskDISABLE_INTERRUPTS(); for( ;; ); }

#endif

修改启动文件

在 main.c 中添加启动代码：

#include "FreeRTOS.h"
#include "task.h"

// 任务函数声明
void vTask1(void *pvParameters);

int main(void) {
    // 硬件初始化
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    // ... 其他初始化
    
    // 创建任务
    xTaskCreate(vTask1, "Task1", 128, NULL, 1, NULL);
    
    // 启动调度器
    vTaskStartScheduler();
    
    // 正常情况不会执行到这里
    while(1);
}

void vTask1(void *pvParameters) {
    while(1) {
        // 任务代码
        vTaskDelay(100);  // 延时100个tick
    }
}

3. 任务管理

任务是FreeRTOS中最基本的执行单元。理解任务的创建、状态转换和删除是使用FreeRTOS的基础。

3.1 xTaskCreate 参数详解

函数原型

BaseType_t xTaskCreate(
    TaskFunction_t       pxTaskCode,    // 任务函数指针
    const char * const  pcName,         // 任务名称
    const uint16_t       usStackDepth,  // 栈大小（字）
    void * const          pvParameters,  // 传递参数
    UBaseType_t           uxPriority,    // 优先级
    TaskHandle_t * const pxCreatedTask  // 任务句柄
);
参数 类型 说明
pxTaskCode TaskFunction_t 任务函数名，函数签名为 void func(void *pvParameters)
pcName const char* 任务名称，用于调试识别，不影响功能
usStackDepth uint16_t 栈大小，单位是"字"（Word），实际字节 = 值 × 4
例如：128 = 512字节
pvParameters void* 传递给任务函数的参数，可为NULL
uxPriority UBaseType_t 任务优先级（0~configMAX_PRIORITIES-1）
数字越大优先级越高
pxCreatedTask TaskHandle_t* 用于接收任务句柄，可用于后续删除/操作任务，可为NULL

创建任务完整示例

#include "FreeRTOS.h"
#include "task.h"

// 任务句柄，用于后续操作
TaskHandle_t xLedTaskHandle;
TaskHandle_t xSensorTaskHandle;

// 任务函数声明
void vLedTask(void *pvParameters);
void vSensorTask(void *pvParameters);

int main(void) {
    // 硬件初始化...
    
    // 创建LED闪烁任务
    xTaskCreate(
        vLedTask,           // 任务函数
        "LED_Task",          // 任务名
        128,                 // 栈大小：128×4=512字节
        NULL,                // 无参数
        1,                   // 优先级1（低）
        &xLedTaskHandle      // 保存句柄
    );
    
    // 创建传感器读取任务
    uint8_t sensorId = 0x01;  // 传递参数
    xTaskCreate(
        vSensorTask,
        "Sensor_Task",
        256,                 // 传感器任务需要更大栈
        &sensorId,           // 传递传感器ID
        2,                   // 优先级2（较高）
        &xSensorTaskHandle
    );
    
    // 启动调度器
    vTaskStartScheduler();
    
    while(1);
}

// LED闪烁任务
void vLedTask(void *pvParameters) {
    while(1) {
        HAL_GPIO_TogglePin(LED_GPIO_Port, LED_Pin);
        vTaskDelay(500);  // 延时500个tick（500ms @ 1000Hz）
    }
}

// 传感器读取任务
void vSensorTask(void *pvParameters) {
    uint8_t id = *(uint8_t*)pvParameters;  // 获取参数
    
    while(1) {
        float temp = ReadTemperature(id);
        vTaskDelay(1000);
    }
}

3.2 任务优先级

任务优先级决定了调度器如何选择下一个运行的任务。

优先级规则

  • 优先级范围：0 到 configMAX_PRIORITIES - 1
  • 数字越大，优先级越高
  • 调度器总是选择就绪状态中优先级最高的任务运行
  • 相同优先级的任务按时间片轮转调度
  • 0 是最低优先级，通常用于空闲任务

优先级就像医院的急诊分诊：

• 危重病人（高优先级）立即处理

• 普通病人（低优先级）需要等待

• 同样紧急的病人按先来后到排队

优先级设计原则：

• 实时性要求高的任务分配高优先级（如：安全检测、通信接收）

• 计算密集型或延时较大的任务分配低优先级（如：数据显示、日志记录）

• 避免所有任务都设为最高优先级，否则失去意义

• 优先级数量不宜过多，7个左右足够大多数应用

3.3 任务状态机

FreeRTOS任务有四种基本状态，调度器负责管理状态转换。

运行态
Running
↔
就绪态
Ready
阻塞态
Blocked
挂起态
Suspended
状态 描述 进入条件 退出条件
运行态 任务正在执行 被调度器选中 时间片用完、被抢占、阻塞、挂起
就绪态 任务可运行，等待CPU 创建任务、阻塞结束、恢复 被调度器选中执行
阻塞态 任务等待事件 vTaskDelay、等待信号量/队列 超时到期、事件发生
挂起态 任务被临时禁用 vTaskSuspend() vTaskResume()

状态转换示例

void vTaskExample(void *pvParameters) {
    while(1) {
        // [运行态] 执行任务代码
        DoWork();
        
        // [运行态] → [阻塞态] 延时1秒
        vTaskDelay(1000);
        
        // 1秒后自动回到 [就绪态]
        // 调度器选中后回到 [运行态]
    }
}

3.4 删除任务

任务可以使用 vTaskDelete() 删除自己或其他任务。

删除任务API

void vTaskDelete(TaskHandle_t xTask);

参数：NULL 表示删除自己，传入句柄删除指定任务。

任务自删除示例

void vOneShotTask(void *pvParameters) {
    // 执行一次性任务
    InitializeHardware();
    SendStartupMessage();
    
    // 任务完成，删除自己
    vTaskDelete(NULL);
    
    // 不会执行到这里
}

删除其他任务示例

TaskHandle_t xTempTaskHandle;

void vMainTask(void *pvParameters) {
    // 创建临时任务
    xTaskCreate(vTempTask, "Temp", 128, NULL, 2, &xTempTaskHandle);
    
    vTaskDelay(5000);  // 等待5秒
    
    // 删除临时任务
    vTaskDelete(xTempTaskHandle);
    xTempTaskHandle = NULL;
}

删除任务注意事项：

• 删除任务会释放其栈内存（如果使用动态分配）

• 如果任务持有互斥量或信号量，删除前必须释放，否则造成死锁

• 被删除任务的内存不会立即回收，而是在空闲任务中清理

• 频繁创建/删除任务会导致内存碎片，建议使用任务挂起代替删除

3.5 任务函数模板

一个标准的任务函数通常包含以下结构：

标准任务模板

void vStandardTask(void *pvParameters) {
    /* ========== 1. 参数获取（可选）========== */
    TaskParams_t *params = (TaskParams_t*)pvParameters;
    
    /* ========== 2. 任务初始化 ========== */
    // 初始化外设、创建通信对象等
    InitPeripherals();
    QueueHandle_t xQueue = xQueueCreate(10, sizeof(Data_t));
    
    /* ========== 3. 主循环 ========== */
    while(1) {
        // 3.1 执行核心功能
        Data_t data = ReadSensor();
        
        // 3.2 与其他任务通信
        xQueueSend(xQueue, &data, 100);
        
        // 3.3 阻塞等待（让出CPU）
        vTaskDelay(100);  // 100ms周期
    }
    
    /* ========== 4. 清理（通常不会执行）========== */
    vQueueDelete(xQueue);
    vTaskDelete(NULL);
}

4. 队列通信

队列是FreeRTOS中任务间传递数据的主要机制。它实现了线程安全的数据缓冲区，让不同任务可以安全地交换信息。

4.1 队列基本概念

队列就像传送带：

• 生产者（发送任务）把物品放到传送带上

• 传送带（队列）按顺序存放物品

• 消费者（接收任务）从另一端取走物品

• 传送带满了，生产者必须等待

• 传送带空了，消费者必须等待

队列的特点

  • FIFO机制：先进先出，保证数据顺序
  • 线程安全：多任务同时访问不会冲突
  • 阻塞机制：队列满/空时任务自动阻塞
  • 拷贝传递：数据通过拷贝传递，不是指针

4.5 完整多任务通信示例

生产者-消费者模式完整代码

#include "FreeRTOS.h"
#include "task.h"
#include "queue.h"

// 数据结构定义
typedef struct {
    uint8_t sensorId;
    float temperature;
    uint32_t timestamp;
} SensorMessage_t;

// 全局队列句柄
QueueHandle_t xSensorQueue;

// 传感器采集任务（生产者）
void vSensorTask(void *pvParameters) {
    SensorMessage_t msg;
    msg.sensorId = 0x01;
    
    while(1) {
        // 读取传感器
        msg.temperature = ReadTemperatureSensor();
        msg.timestamp = HAL_GetTick();
        
        // 发送到队列（等待最多10ms）
        if (xQueueSend(xSensorQueue, &msg, 
                          pdMS_TO_TICKS(10)) != pdPASS) {
            // 队列满，丢弃数据
            Error_Handler();
        }
        
        // 1秒采集一次
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}

// 数据处理任务（消费者）
void vProcessTask(void *pvParameters) {
    SensorMessage_t rxMsg;
    float avgTemp = 0.0f;
    int count = 0;
    
    while(1) {
        // 阻塞等待数据
        if (xQueueReceive(xSensorQueue, &rxMsg, portMAX_DELAY) == pdPASS) {
            // 计算移动平均
            avgTemp = (avgTemp * count + rxMsg.temperature) / (count + 1);
            count++;
            
            // 每10个样本打印一次
            if (count >= 10) {
                printf("Avg Temp: %.2f°C\n", avgTemp);
                count = 0;
            }
        }
    }
}

// 主函数
int main(void) {
    HAL_Init();
    SystemClock_Config();
    
    // 创建队列：最多存放10条消息
    xSensorQueue = xQueueCreate(10, sizeof(SensorMessage_t));
    
    if (xSensorQueue == NULL) {
        Error_Handler();
    }
    
    // 创建任务
    xTaskCreate(vSensorTask, "Sensor", 128, NULL, 2, NULL);
    xTaskCreate(vProcessTask, "Process", 256, NULL, 1, NULL);
    
    // 启动调度器
    vTaskStartScheduler();
    
    while(1);
}

4.6 队列长度选择

队列长度的选择需要平衡内存消耗和数据缓冲需求。

场景 建议长度 原因
高速采样+低速处理 较大（50-100） 缓冲突发数据，防止丢数据
低速采集+高速处理 较小（5-10） 处理速度够快，无需大缓冲
事件通知 1 只关心最新事件，用队列传递指针
高速通信 中等到大 应对网络突发流量

队列内存计算：

队列占用内存 = 队列长度 × 数据大小 + 队列管理开销（约96字节）

例如：长度10，每条数据20字节 → 10×20+96 = 296字节

5. 信号量

信号量（Semaphore）是FreeRTOS中用于任务同步和资源管理的核心机制。

5.1 信号量基本概念

信号量就像停车场：

• 计数信号量：停车场有N个车位，进来一辆车计数减1，出去一辆计数加1

• 二值信号量：只有一个车位，要么空要么满

• 车位满时，新车必须等待（任务阻塞）

• 有车离开时，通知等待的车进入（唤醒阻塞任务）

5.2 二值信号量

二值信号量只有两个状态：有效（1）和无效（0），常用于任务同步。

创建二值信号量

SemaphoreHandle_t xSemaphoreCreateBinary(void);

注意：创建后信号量初始值为0（无效），需要先 xSemaphoreGive() 才能使用。

中断-任务同步示例

SemaphoreHandle_t xBinarySem;

// ISR中释放信号量
void EXTI0_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    
    // 清除中断标志
    __HAL_GPIO_EXTI_CLEAR_IT(GPIO_PIN_0);
    
    // 释放信号量（FromISR版本）
    xSemaphoreGiveFromISR(xBinarySem, &xHigherPriorityTaskWoken);
    
    // 如果唤醒了更高优先级任务，请求上下文切换
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}

// 任务中等待信号量
void vButtonTask(void *pvParameters) {
    while(1) {
        // 阻塞等待信号量
        if (xSemaphoreTake(xBinarySem, portMAX_DELAY) == pdPASS) {
            // 按键按下，执行操作
            HandleButtonPress();
        }
    }
}

int main(void) {
    HAL_Init();
    // ... 外设初始化
    
    // 创建二值信号量
    xBinarySem = xSemaphoreCreateBinary();
    
    // 创建任务
    xTaskCreate(vButtonTask, "Button", 128, NULL, 2, NULL);
    
    vTaskStartScheduler();
    while(1);
}

5.3 计数信号量

计数信号量可以记录多个事件的发生。

创建计数信号量

SemaphoreHandle_t xSemaphoreCreateCounting(
    UBaseType_t uxMaxCount,   // 最大计数值
    UBaseType_t uxInitialCount // 初始计数值
);

资源池管理示例

#define BUFFER_COUNT 5

uint8_t buffers[BUFFER_COUNT][256];
SemaphoreHandle_t xBufferSem;

void InitBufferPool(void) {
    // 创建计数信号量：5个缓冲区，初始全部可用
    xBufferSem = xSemaphoreCreateCounting(BUFFER_COUNT, BUFFER_COUNT);
}

uint8_t* GetBuffer(void) {
    if (xSemaphoreTake(xBufferSem, pdMS_TO_TICKS(100)) == pdPASS) {
        // 找到空闲缓冲区并返回
        for (int i = 0; i < BUFFER_COUNT; i++) {
            if (!bufferInUse[i]) {
                bufferInUse[i] = true;
                return buffers[i];
            }
        }
    }
    return NULL;  // 无可用缓冲区
}

void ReleaseBuffer(uint8_t* buf) {
    // 标记缓冲区为空闲
    int idx = (buf - buffers[0]) / 256;
    bufferInUse[idx] = false;
    
    // 释放信号量
    xSemaphoreGive(xBufferSem);
}

5.4 生产者-消费者模式

使用计数信号量实现经典的生产者-消费者模型。

完整示例

#define ITEM_COUNT 10

typedef struct {
    int data;
} Item_t;

Item_t buffer[ITEM_COUNT];
int in = 0, out = 0;

SemaphoreHandle_t xEmptySlots;   // 空槽位信号量
SemaphoreHandle_t xFilledSlots;  // 已填充信号量
SemaphoreHandle_t xMutex;        // 互斥访问缓冲区

void vProducer(void *pvParameters) {
    Item_t item;
    int produced = 0;
    
    while(1) {
        // 生产数据
        item.data = produced++;
        
        // 等待空槽位
        xSemaphoreTake(xEmptySlots, portMAX_DELAY);
        
        // 互斥访问缓冲区
        xSemaphoreTake(xMutex, portMAX_DELAY);
        buffer[in] = item;
        in = (in + 1) % ITEM_COUNT;
        xSemaphoreGive(xMutex);
        
        // 通知有新数据
        xSemaphoreGive(xFilledSlots);
        
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}

void vConsumer(void *pvParameters) {
    Item_t item;
    
    while(1) {
        // 等待有数据
        xSemaphoreTake(xFilledSlots, portMAX_DELAY);
        
        // 互斥访问缓冲区
        xSemaphoreTake(xMutex, portMAX_DELAY);
        item = buffer[out];
        out = (out + 1) % ITEM_COUNT;
        xSemaphoreGive(xMutex);
        
        // 释放空槽位
        xSemaphoreGive(xEmptySlots);
        
        // 消费数据
        printf("Consumed: %d\n", item.data);
    }
}

void Setup(void) {
    // 创建信号量
    xEmptySlots = xSemaphoreCreateCounting(ITEM_COUNT, ITEM_COUNT);
    xFilledSlots = xSemaphoreCreateCounting(ITEM_COUNT, 0);
    xMutex = xSemaphoreCreateMutex();
    
    // 创建任务
    xTaskCreate(vProducer, "Producer", 128, NULL, 1, NULL);
    xTaskCreate(vConsumer, "Consumer", 128, NULL, 1, NULL);
}

6. 互斥量

互斥量（Mutex）是一种特殊的二值信号量，专门用于保护共享资源。

6.1 互斥量 vs 信号量

特性 互斥量 (Mutex) 二值信号量
主要用途 保护共享资源 任务同步
所有权 有归属，只有持有者能释放 无归属，任何任务都可释放
优先级继承 支持（解决优先级反转） 不支持
递归获取 支持（需使用递归互斥量） 不支持
中断中使用 ❌ 不允许 ✅ 可以

6.2 创建互斥量

SemaphoreHandle_t xSemaphoreCreateMutex(void);

6.3 使用示例

保护共享串口资源

SemaphoreHandle_t xUartMutex;

void vTask1(void *pvParameters) {
    while(1) {
        // 获取互斥量
        if (xSemaphoreTake(xUartMutex, pdMS_TO_TICKS(100)) == pdPASS) {
            // 安全地使用串口
            printf("Task1: Hello from Task1\n");
            
            // 释放互斥量
            xSemaphoreGive(xUartMutex);
        }
        vTaskDelay(pdMS_TO_TICKS(500));
    }
}

void vTask2(void *pvParameters) {
    while(1) {
        if (xSemaphoreTake(xUartMutex, pdMS_TO_TICKS(100)) == pdPASS) {
            printf("Task2: Hello from Task2\n");
            xSemaphoreGive(xUartMutex);
        }
        vTaskDelay(pdMS_TO_TICKS(300));
    }
}

6.4 优先级反转问题

优先级反转是多任务系统中常见的问题，可能导致高优先级任务被低优先级任务阻塞。

优先级反转场景：

假设有任务优先级：高 > 中 > 低

1. 低优先级任务获取互斥量，准备访问资源

2. 高优先级任务就绪，抢占低优先级任务

3. 高优先级任务尝试获取同一互斥量，被阻塞

4. 中优先级任务就绪，抢占低优先级任务

5. 中优先级任务运行，低优先级任务无法释放互斥量

6. 高优先级任务被中优先级任务间接阻塞！

优先级反转时序

时间 →
┌──────────────────────────────────────────────────────────────┐
│ 高优先级任务  │      │██████│等待  │      │████████████████│
├──────────────────────────────────────────────────────────────┤
│ 中优先级任务  │      │      │      │██████│                │
├──────────────────────────────────────────────────────────────┤
│ 低优先级任务  │██████│      │      │      │ 恢复 │ 释放    │
│ (持有互斥量)  │ 获取 │ 被抢 │ 被抢 │ 被抢 │ 互斥 │ 互斥量  │
└──────────────────────────────────────────────────────────────┘
                ↑
          高优先级任务被中优先级任务
          间接阻塞，无法运行！

6.5 优先级继承

FreeRTOS的互斥量实现了优先级继承机制来缓解优先级反转。

优先级继承原理：

当高优先级任务因等待互斥量而阻塞时，持有互斥量的低优先级任务会临时继承高优先级任务的优先级，从而避免被中优先级任务抢占。

优先级继承后的时序

时间 →
┌──────────────────────────────────────────────────────────────┐
│ 高优先级任务  │      │██████│等待  │      │████████████████│
├──────────────────────────────────────────────────────────────┤
│ 中优先级任务  │      │      │      │      │                │
│               │      │      │      │      │ 无法抢占！     │
├──────────────────────────────────────────────────────────────┤
│ 低优先级任务  │██████│ 继承 │██████│ 释放 │                │
│ (持有互斥量)  │ 获取 │ 高优 │ 执行 │ 互斥 │                │
│               │      │ 先级 │      │      │                │
└──────────────────────────────────────────────────────────────┘
                        ↑
                 低优先级任务继承高优先级，
                 中优先级任务无法抢占！

优先级继承的限制：

• 只能缓解优先级反转，不能完全消除

• 如果有多层嵌套的互斥量，问题仍然可能发生

• 临界区时间应尽可能短，减少阻塞时间

6.6 何时用互斥量 vs 信号量

✅ 使用互斥量

  • 保护共享资源（变量、外设等）
  • 需要优先级继承的场景
  • 同一任务需要多次获取同一锁
  • 互斥访问时间较长

✅ 使用信号量

  • 任务同步（ISR通知任务）
  • 事件计数
  • 资源池管理
  • 需要在ISR中操作

7. 软件定时器

软件定时器允许你在指定时间后执行回调函数，无需占用硬件定时器资源。

7.1 创建定时器 xTimerCreate

函数原型

TimerHandle_t xTimerCreate(
    const char * const pcTimerName,   // 定时器名称
    const TickType_t xTimerPeriodInTicks, // 周期（tick）
    const UBaseType_t uxAutoReload,   // 自动重载（周期）
    void * const pvTimerID,           // 定时器ID
    TimerCallbackFunction_t pxCallbackFunction // 回调函数
);
参数 说明
pcTimerName 定时器名称，用于调试
xTimerPeriodInTicks 定时周期，使用 pdMS_TO_TICKS(ms) 转换
uxAutoReload pdTRUE = 周期定时器，pdFALSE = 单次定时器
pvTimerID 用户自定义ID，可用于区分多个定时器
pxCallbackFunction 回调函数，签名：void callback(TimerHandle_t)

7.2 单次 vs 周期定时器

单次定时器 (One-shot)

触发一次后停止，需要手动重新启动。

  • 超时执行某个操作
  • 看门狗喂狗
  • 延时执行
uxAutoReload = pdFALSE

周期定时器 (Auto-reload)

每次触发后自动重新启动，持续运行。

  • 周期性采样
  • LED闪烁
  • 状态检测
uxAutoReload = pdTRUE

7.4 完整示例

周期定时器控制LED闪烁

#include "timers.h"

TimerHandle_t xLedTimer;
TimerHandle_t xTimeoutTimer;

// LED闪烁回调函数

    // 切换LED状态
    HAL_GPIO_TogglePin(LED_GPIO_Port, LED_Pin);
}

// 超时回调函数
void vTimeoutCallback(TimerHandle_t xTimer) {
    printf("Timeout occurred!\n");
    
    // 单次定时器执行后自动停止
}

int main(void) {
    HAL_Init();
    // ...
    
    // 创建周期定时器：每500ms闪烁一次
    xLedTimer = xTimerCreate(
        "LedTimer",                 // 名称
        pdMS_TO_TICKS(500),          // 周期500ms
        pdTRUE,                        // 自动重载（周期）
        (void*)0,                     // ID
        vLedTimerCallback              // 回调函数
    );
    
    // 创建单次定时器：5秒后触发
    xTimeoutTimer = xTimerCreate(
        "Timeout",
        pdMS_TO_TICKS(5000),         // 5秒
        pdFALSE,                       // 单次
        (void*)1,
        vTimeoutCallback
    );

    xTimerStart(xTimeoutTimer, 0);

    while(1);
}

7.5 回调函数规则

⚠️ 定时器回调函数必须遵守的规则：

• 不能阻塞！ 不能调用 vTaskDelay()、xQueueReceive() 等阻塞函数

• 执行时间要尽可能短，避免影响其他定时器

• 可以发送信号量、向队列发送数据来通知任务处理

• 不能调用 xTimerStart/Stop 在同一个定时器上（会死锁），使用 xTimerPendFunctionCallFromISR 替代

正确做法：回调中通知任务

SemaphoreHandle_t xTimerSem;

void vTimerCallback(TimerHandle_t xTimer) {
    // 不做耗时操作，只发送信号量
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xSemaphoreGiveFromISR(xTimerSem, &xHigherPriorityTaskWoken);
}

void vHandlerTask(void *pvParameters) {
    while(1) {
        if (xSemaphoreTake(xTimerSem, portMAX_DELAY) == pdPASS) {
            // 在任务中执行耗时操作
            ProcessData();
        }
    }
}

8. 中断中的FreeRTOS

在中断服务函数（ISR）中使用FreeRTOS API有特殊规则，必须严格遵守。

8.1 FromISR 函数

所有在中断中调用的FreeRTOS API都有 FromISR 后缀版本：

任务上下文 中断上下文
xQueueSend() xQueueSendFromISR()
xSemaphoreGive() xSemaphoreGiveFromISR()
xTaskNotifyGive() vTaskNotifyGiveFromISR()
xTimerStart() xTimerStartFromISR()

8.2 portYIELD_FROM_ISR

中断可能唤醒更高优先级的任务，此时需要请求上下文切换。

标准ISR模板

void USART1_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    
    // 检查中断标志
    if (__HAL_UART_GET_FLAG(&huart1, UART_FLAG_RXNE)) {
        uint8_t data = (uint8_t)(huart1.Instance->DR & 0xFF);
        
        // 使用FromISR函数发送数据
        xQueueSendFromISR(xUartQueue, &data, &xHigherPriorityTaskWoken);
    }
    
    // 如果唤醒了更高优先级任务，请求切换
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}

xHigherPriorityTaskWoken 工作原理：

1. 初始化为 pdFALSE

2. 如果 FromISR 函数唤醒了更高优先级任务，会将其设为 pdTRUE

3. portYIELD_FROM_ISR(pdTRUE) 会在ISR退出后立即触发上下文切换

4. portYIELD_FROM_ISR(pdFALSE) 不会切换，返回被中断的任务

8.3 中断安全规则

⚠️ ISR中禁止的操作：

• 不能调用不带 FromISR 的API

• 不能调用 vTaskDelay() 或任何阻塞函数

• 不能获取互斥量（xSemaphoreTake）

• ISR执行时间应尽可能短

8.4 中断优先级配置

FreeRTOS需要配置中断优先级，确保内核中断正确工作。

FreeRTOSConfig.h 配置

// Cortex-M 中断优先级配置
// 数值越大，优先级越低

// 内核中断最低优先级（必须最低）
#define configKERNEL_INTERRUPT_PRIORITY         255

// 可调用FreeRTOS API的最高中断优先级
#define configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY 5

// 实际使用的优先级值
#define configMAX_SYSCALL_INTERRUPT_PRIORITY \
    ( configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY << (8 - configPRIO_BITS) )

中断优先级分组规则

  • 优先级 ≥ configMAX_SYSCALL_INTERRUPT_PRIORITY：可以调用 FromISR 函数，但会被内核临界区屏蔽
  • 优先级 < configMAX_SYSCALL_INTERRUPT_PRIORITY：不能调用任何FreeRTOS API，但永远不会被内核屏蔽
  • 建议：将实时性要求极高的中断（如电机控制）设为最高优先级（数值最小），且不调用FreeRTOS API

9. 内存管理

FreeRTOS提供多种内存管理方案，适应不同的应用场景。

9.1 heap_1 ~ heap_5 方案对比

方案 特点 适用场景
heap_1 只分配不释放 任务创建后永不删除
heap_2 可分配释放，不合并碎片 固定大小块频繁分配释放（已过时）
heap_3 包装标准库malloc/free 需要使用系统堆（不推荐）
heap_4 推荐 可分配释放，自动合并碎片 通用场景，最常用
heap_5 支持多块不连续内存 内存分布在多个区域

heap_4 详解（推荐使用）

heap_4 特点：

  • 内存分配使用"首次适应算法"
  • 释放相邻块会自动合并，减少碎片
  • 所有内存从 configTOTAL_HEAP_SIZE 定义的区域分配

9.2 栈溢出检测

任务栈溢出是常见的错误来源，FreeRTOS提供检测机制。

启用栈溢出检测

// 在 FreeRTOSConfig.h 中配置

// 方法1：检查栈指针是否越界
#define configCHECK_FOR_STACK_OVERFLOW 1

// 方法2：检查栈指针+栈末尾标记（更可靠）
#define configCHECK_FOR_STACK_OVERFLOW 2

实现栈溢出钩子函数

void vApplicationStackOverflowHook(TaskHandle_t xTask, char *pcTaskName) {
    // 栈溢出时调用
    printf("Stack overflow in task: %s\n", pcTaskName);
    
    // 可以在这里设置断点或重启
    while(1);  // 挂起系统
}

栈大小估算技巧：

1. 初始设置较大的栈（如512字节）

2. 运行程序，调用 uxTaskGetStackHighWaterMark() 检查剩余

3. 根据实际使用量设置合理大小（预留50%余量）

10. 与HAL共存

STM32 HAL库与FreeRTOS配合使用时，需要注意一些关键配置。

10.1 SysTick_Handler 修改

FreeRTOS需要独占SysTick定时器，HAL的 HAL_IncTick() 必须在FreeRTOS钩子中调用。

修改 stm32xxx_it.c

// 原始代码
void SysTick_Handler(void) {
    HAL_IncTick();
}

// 使用FreeRTOS后，CubeMX会自动修改为：
void SysTick_Handler(void) {
    if (xTaskGetSchedulerState() != taskSCHEDULER_NOT_STARTED) {
        xPortSysTickHandler();
    }
    HAL_IncTick();
}

CubeMX会自动处理这个修改，手动移植时需要注意。

10.2 HAL_Delay 替代方案

⚠️ HAL_Delay 在FreeRTOS中的问题：

HAL_Delay() 使用轮询等待，会阻塞整个系统。在任务中应使用 vTaskDelay()。

替代方案

// 方案1：任务中使用 vTaskDelay
void vTaskDelay(uint32_t ms) {
    vTaskDelay(pdMS_TO_TICKS(ms));
}

// 方案2：封装兼容函数
void Delay_ms(uint32_t ms) {
    if (xTaskGetSchedulerState() == taskSCHEDULER_RUNNING) {
        vTaskDelay(pdMS_TO_TICKS(ms));
    } else {
        HAL_Delay(ms);  // 调度器未启动时使用HAL_Delay
    }
}

// 方案3：中断中使用 HAL_Delay（不推荐，改用任务通知）
// 中断中不应有延时操作！

11.3 死锁排查

死锁常见原因

  • 互斥量获取后忘记释放
  • 多个互斥量获取顺序不一致
  • 中断中尝试获取互斥量
  • 优先级反转导致无限等待

死锁排查方法：

1. 在 xSemaphoreTake() 设置超时值而非无限等待

2. 超时后打印调试信息

3. 使用调试器查看任务状态

4. 检查所有互斥量获取/释放是否成对

11.4 定时器溢出检测

    uint32_t start = HAL_GetTick();

    ProcessTimer();

❓ 常见问题 FAQ

Q: FreeRTOS和裸机开发哪个好？

A: 3个以上并发任务用RTOS更清晰。1~2个任务裸机+状态机更简单。不要为了用RTOS而用RTOS。

Q: 任务栈大小怎么设？

A: 初期设256或512字（word），用uxTaskGetStackHighWaterMark()查看实际用量，再减半+20%余量。

Q: 优先级反转怎么办？

A: 使用互斥量(Mutex)替代二值信号量，FreeRTOS的Mutex自带优先级继承机制。

Q: vTaskDelay和vTaskDelayUntil区别？

A: vTaskDelay是相对延时（从调用时刻起），vTaskDelayUntil是绝对周期（适合精确周期任务）。推荐用DelayUntil做采样循环。
