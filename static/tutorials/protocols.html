📡 UART 详解

UART是最基础的串行通信协议，详见串口通信教程，这里补充要点。

数据帧结构

 空闲(高) ┌───┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─────┐
          │   ││D0││D1││D2││D3││D4││D5││D6││D7││ 停止 │
 ─────────┘   └┘└┘└┘└┘└┘└┘└┘└┘└┘└───┘
          ↑起始位    ↑数据位(LSB先发)   ↑校验  ↑停止位

常见配置 "8N1" = 8数据位 + 无校验 + 1停止位

STM32 HAL UART代码

// 初始化（CubeMX自动生成）
UART_HandleTypeDef huart1;
huart1.Instance = USART1;
huart1.Init.BaudRate = 115200;
huart1.Init.WordLength = UART_WORDLENGTH_8B;
huart1.Init.StopBits = UART_STOPBITS_1;
huart1.Init.Parity = UART_PARITY_NONE;
HAL_UART_Init(&huart1);

// 阻塞发送
HAL_UART_Transmit(&huart1, (uint8_t*)"Hello", 5, 1000);

// 中断接收
uint8_t rx_byte;
HAL_UART_Receive_IT(&huart1, &rx_byte, 1);

// 回调函数
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    if (huart->Instance == USART1) {
        // 处理接收到的字节
        HAL_UART_Receive_IT(huart, &rx_byte, 1);  // 重新开启接收
    }
}

// DMA空闲中断接收（推荐，解决粘包问题）
uint8_t rx_buf[256];
void UART_StartReceive(void) {
    HAL_UARTEx_ReceiveToIdle_DMA(&huart1, rx_buf, sizeof(rx_buf));
}

void HAL_UARTEx_RxEventCallback(UART_HandleTypeDef *huart, uint16_t Size) {
    // Size = 实际接收到的字节数
    process_data(rx_buf, Size);
    HAL_UARTEx_ReceiveToIdle_DMA(&huart1, rx_buf, sizeof(rx_buf));  // 重启
}

🔗 I2C 详解

I2C（Inter-Integrated Circuit）用2根线连接多个设备，是传感器/OLED最常用的协议。

硬件连接

          VCC (3.3V)
           │
    ┌──────┤──────┐
    │      │      │
   [RP]   [RP]   [RP]   ← 上拉电阻（4.7KΩ典型值）
    │      │      │
    ├──────┼──────┤──── SDA
    │      │      │
    ├──────┼──────┤──── SCL
    │      │      │
  [MCU] [传感器] [OLED]
    │      │      │
    └──────┴──────┘
           │
          GND

关键：SDA和SCL必须各接一个上拉电阻！
⚠️ 没有上拉电阻I2C不工作！I2C是开漏输出，没有上拉总线永远是低电平。有些模块自带上拉，检查原理图。没上的话外接4.7KΩ到VCC。

I2C通信流程

// 写寄存器：MCU → 传感器
// 主机发送：START → 从机地址(7bit)+W(0) → ACK → 寄存器地址 → ACK → 数据 → ACK → STOP

// 读寄存器：MCU ← 传感器
// 主机先写后读：
// START → 从机地址+W → ACK → 寄存器地址 → ACK →
// RESTART → 从机地址+R(1) → ACK → 数据 → NACK → STOP

// 示例：读取BMP280温度传感器
// 从机地址 = 0x76（SDO接GND）或0x77（SDO接VCC）
// 温度寄存器 = 0xFA

STM32 HAL I2C代码

// 写1字节到寄存器
HAL_I2C_Mem_Write(&hi2c1, 0x76<<1, 0xF4, 1, &data, 1, 100);
//              句柄   设备地址左移1位  寄存器地址  数据  超时

// 从寄存器读N字节
uint8_t buf[6];
HAL_I2C_Mem_Read(&hi2c1, 0x76<<1, 0xFA, 1, buf, 6, 100);

// 检测设备是否在线
HAL_StatusTypeDef ret = HAL_I2C_IsDeviceReady(&hi2c1, 0x76<<1, 3, 100);
if (ret == HAL_OK) {
    printf("BMP280 found at 0x76\n");
}

// 扫描总线上所有设备（调试利器）
void i2c_scan(I2C_HandleTypeDef *hi2c) {
    printf("扫描I2C总线...\n");
    for (uint8_t addr = 1; addr < 127; addr++) {
        if (HAL_I2C_IsDeviceReady(hi2c, addr << 1, 1, 10) == HAL_OK) {
            printf("  找到设备: 0x%02X\n", addr);
        }
    }
}

I2C常见问题

❌ HAL_I2C_MEM_Read 返回 HAL_TIMEOUT

原因：①设备地址错 ②上拉电阻缺失 ③SCL/SDA接反 ④设备未上电

排查：用i2c_scan()先看设备在不在

⚡ SPI 详解

SPI（Serial Peripheral Interface）是最快的串行协议，适合高速设备（Flash/屏幕/SD卡）。

硬件连接（4线SPI）

  MCU          从机1        从机2
  MOSI ──────→ MOSI        MOSI
  MISO ←────── MISO        MISO
  SCLK ──────→ SCLK        SCLK
  CS0  ──────→ CS          │
  CS1  ──────────────────→ CS
  
  每个从机需要一根独立的CS线！

SPI四根线的作用

信号方向作用
MOSI主→从主出从入（Master Out Slave In）
MISO从→主主入从出（Master In Slave Out）
SCLK主→从时钟信号（由主机产生）
CS/SS主→从片选（低有效，选中从机）

SPI极性和相位（CPOL/CPHA）

模式CPOLCPHA空闲SCK采样边沿常见设备
000低上升沿大多数传感器
101低下降沿部分LCD
210高下降沿少数设备
311高上升沿SD卡
⚠️ CPOL/CPHA必须匹配！主从双方必须用同一种模式，否则数据全是乱的。查设备数据手册确认。

STM32 HAL SPI代码

// CubeMX配置：SPI1 → Mode: Full-Duplex Master → Prescaler: 8 (9MHz)
// CPOL=Low, CPHA=1Edge, 8bit, MSB First

// 发送数据
uint8_t tx_buf[] = {0x9F};  // W25Q Flash: Read JEDEC ID
uint8_t rx_buf[4] = {0};

HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET);  // CS拉低
HAL_SPI_TransmitReceive(&hspi1, tx_buf, rx_buf, 4, 100);
HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET);    // CS拉高

printf("Flash ID: %02X %02X %02X\n", rx_buf[1], rx_buf[2], rx_buf[3]);
// W25Q32: EF 40 16

// 阻塞发送（单向，如控制SPI屏幕）
HAL_SPI_Transmit(&hspi1, pixel_data, len, 100);

// DMA发送（不阻塞CPU，推荐大屏使用）
HAL_SPI_Transmit_DMA(&hspi1, pixel_data, len);

软件SPI（任意GPIO模拟）

// 当硬件SPI不够用时，可以用GPIO模拟
#define SOFT_SCK  GPIO_PIN_5
#define SOFT_MOSI GPIO_PIN_7
#define SOFT_CS   GPIO_PIN_4

void SPI_Soft_Write(uint8_t data) {
    HAL_GPIO_WritePin(GPIOA, SOFT_CS, GPIO_PIN_RESET);
    for (int i = 7; i >= 0; i--) {
        HAL_GPIO_WritePin(GPIOA, SOFT_SCK, GPIO_PIN_RESET);
        HAL_GPIO_WritePin(GPIOA, SOFT_MOSI, (data >> i) & 1);
        HAL_GPIO_WritePin(GPIOA, SOFT_SCK, GPIO_PIN_SET);  // 上升沿采样
    }
    HAL_GPIO_WritePin(GPIOA, SOFT_CS, GPIO_PIN_SET);
}

🚗 CAN 总线详解

CAN（Controller Area Network）是汽车和工业领域的主流总线，抗干扰极强，支持多主。

硬件连接

  MCU1        MCU2        MCU3
  TXD→┐       TXD→┐       TXD→┐
       │            │            │
  [TJA1050]    [TJA1050]    [TJA1050]   ← CAN收发器
       │            │            │
  CANH ├────────────┼────────────┤   ← 差分信号
  CANL ├────────────┼────────────┤
       │            │            │
      [120Ω]                  [120Ω]   ← 终端电阻（两端各一个）
       │            │            │
      GND          GND          GND
⚠️ CAN必须加收发器和终端电阻！MCU的CAN引脚是逻辑电平，不能直接接CANH/CANL。必须通过TJA1050/SN65HVD230等收发器转换。总线两端各需120Ω终端电阻，没有的话信号反射导致通信失败。

CAN帧结构

┌──────┬──────┬────────────┬──────┬──────────────┬─────┬──────┬──────┐
│ 帧起始│仲裁场│  控制场     │数据场 │   CRC场      │ACK场│帧结束│帧间隔│
│ 1bit │12bit │  6bit      │0-8B  │  16bit       │2bit │7bit  │3bit  │
└──────┴──────┴────────────┴──────┴──────────────┴─────┴──────┴──────┘
仲裁场 = 11位ID + 1位RTR（0=数据帧, 1=远程帧）
ID越小优先级越高！

STM32 HAL CAN代码

// CAN初始化（STM32F4，500kbps）
CAN_HandleTypeDef hcan1;
hcan1.Instance = CAN1;
hcan1.Init.Prescaler = 8;           // APB1CLK / Prescaler / (1+BS1+BS2)
hcan1.Init.TimeSeg1 = CAN_BS1_14TQ; // 位时序段1
hcan1.Init.TimeSeg2 = CAN_BS2_3TQ;  // 位时序段2
hcan1.Init.SyncJumpWidth = CAN_SJW_1TQ;
hcan1.Init.Mode = CAN_MODE_NORMAL;
HAL_CAN_Init(&hcan1);

// 配置过滤器（必须！否则收不到消息）
CAN_FilterTypeDef filter;
filter.FilterBank = 0;
filter.FilterMode = CAN_FILTERMODE_IDMASK;
filter.FilterScale = CAN_FILTERSCALE_32BIT;
filter.FilterIdHigh = 0x0000;
filter.FilterIdLow = 0x0000;
filter.FilterMaskIdHigh = 0x0000;
filter.FilterMaskIdLow = 0x0000;  // 全部通过
filter.FilterFIFOAssignment = CAN_RX_FIFO0;
filter.FilterActivation = ENABLE;
HAL_CAN_ConfigFilter(&hcan1, &filter);

// 启动CAN
HAL_CAN_Start(&hcan1);
HAL_CAN_ActivateNotification(&hcan1, CAN_IT_RX_FIFO0_MSG_PENDING);

// 发送消息
CAN_TxHeaderTypeDef tx_header;
uint8_t tx_data[8] = {0x01, 0x02, 0x03, 0x04, 0x05, 0x06, 0x07, 0x08};
uint32_t tx_mailbox;

tx_header.StdId = 0x123;      // 标准ID
tx_header.ExtId = 0;           // 扩展ID（本例不用）
tx_header.IDE = CAN_ID_STD;    // 标准帧
tx_header.RTR = CAN_RTR_DATA;  // 数据帧
tx_header.DLC = 8;             // 数据长度

HAL_CAN_AddTxMessage(&hcan1, &tx_header, tx_data, &tx_mailbox);

// 接收回调
void HAL_CAN_RxFifo0MsgPendingCallback(CAN_HandleTypeDef *hcan) {
    CAN_RxHeaderTypeDef rx_header;
    uint8_t rx_data[8];
    HAL_CAN_GetRxMessage(hcan, CAN_RX_FIFO0, &rx_header, rx_data);
    printf("CAN ID:0x%03X DLC:%d Data:", rx_header.StdId, rx_header.DLC);
    for (int i = 0; i < rx_header.DLC; i++) {
        printf(" %02X", rx_data[i]);
    }
    printf("\n");
}

CAN波特率计算

// CAN波特率 = APB时钟 / Prescaler / (1 + TimeSeg1 + TimeSeg2)
// 例：APB1 = 42MHz, 500kbps
// 42MHz / 8 / (1 + 14 + 3) = 42,000,000 / 8 / 18 = 291,666 ≈ 291kbps ❌
// 改：42MHz / 6 / (1 + 9 + 4) = 42,000,000 / 6 / 14 = 500,000 = 500kbps ✅

// 常用组合：
// 500kbps: Prescaler=6,  BS1=9,  BS2=4   (APB1=42MHz)
// 250kbps: Prescaler=12, BS1=9,  BS2=4   (APB1=42MHz)
// 1Mbps:   Prescaler=3,  BS1=9,  BS2=4   (APB1=42MHz)

📡 信号完整性检查清单

排查通信问题的10步checklist：
  1. 共地了吗？两个设备必须有公共地线！(最常见错误)
  2. BAUD率误差在允许范围内吗？(标准UART容忍±2%, 内部HSI温度漂移可达±5% → 必须校准或用HSE晶振)
  3. 逻辑电平匹配吗？3.3V↔5V需要转换
  4. 线序对吗？TX↔RX交叉、MOSI↔MOSI直连、SDA/SDA直连
  5. 上拉电阻？I2C必须外接(4.7kΩ@3.3V), SPI的CS需要上拉防浮空
  6. 终端电阻？RS485/RS422/CAN需要在总线两端加120Ω
  7. 线长？UART≤2m, I2C≤30cm(PCB上), SPI≤10cm, RS485≤1200m
  8. 干扰源？电机/继电器/开关电源与信号线保持5cm以上距离
  9. 共模电压？RS485共模范围-7V~+12V, 超出会永久损坏
  10. 总线冲突？RS485多个发送器同时发送→短路→需要方向控制(DE/RE)

// 硬件: 8通道/24MHz采样率的CY7C68013A方案(¥25-40)

// 关键设置:

// - 全部为0xFF: 线断了或对方未发送
// - 偶发错误字节: 波特率偏差(对照波形测量位宽)
// - 只有0x00: TX/RX接反了(检查交叉)
//
// I2C:
// - SDA始终低: 从设备拉死(地址错误或时钟停)
// - 有SCK无SDA: 上拉电阻缺失或太大
// - 前7bit对后8bit不对: 地址左移1位？(有些库已做, 有些未做)
//
// SPI:
// - MISO全0: 从设备未正确响应(检查CS时序)
// - 数据移位: CPOL/CPHA配置错误
// - 第一个字节错误: CS建立时间不够(在CS拉低后delay 10μs)

🔧 通信协议的选择决策树

需要通信的两个设备...

├── 都在同一块PCB上?
│   ├── 需要多从设备? → I2C (2线, 最多127个从设备)
│   ├── 需要高速? → SPI (4线, 可达50Mbps)
│   └── 最简单、只有2个设备? → UART (2线, 常用115200bps)
│
├── 距离 2-10米?
│   ├── 抗干扰要求高? → RS485 (差分, 1200m)
│   └── 简单点对点? → UART + MAX232 (RS232, 15m)
│
├── 距离 10-50米?
│   └── RS485
│
├── 工业/汽车场景?
│   ├── 汽车内部 → CAN (差分, 多主, 无损仲裁)
│   ├── 简单传感器/执行器 → LIN (单线, 19.2kbps, 便宜)
│   └── 工业自动化 → RS485 + Modbus RTU
│
└── 无线?
    ├── 10m内 → BLE (nRF52/ESP32)
    ├── 100m-几km → LoRa (SX1278)
    └── 需要自组网 → Zigbee

❓ 常见问题 FAQ

Q: I2C和SPI选哪个？

A: 速度优先选SPI(可达几十MHz)，引脚少选I2C(只需2根)。传感器多选I2C，Flash/屏选SPI。

Q: I2C总线卡死怎么办？

A: 从机拉低SDA不释放。解决：1)重新初始化I2C 2)发9个SCL时钟脉冲 3)GPIO模拟I2C复位序列 4)硬件上加拉电阻4.7kΩ

Q: CAN和RS485哪个好？

A: CAN有仲裁机制和错误处理，适合多主高可靠场景（汽车）；RS485简单便宜，适合主从式长距离通信。
