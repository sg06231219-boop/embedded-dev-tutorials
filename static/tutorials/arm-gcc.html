🔧 ARM GCC 编译工具链 细致教程

ARM GCC 是嵌入式开发的核心编译工具，把C代码变成STM32/ESP32能运行的机器码。本教程从安装到精通，一步步带你搞懂。

💡 类比：ARM GCC就像一个翻译官——你写C语言（人类能读的），它翻译成机器码（MCU能执行的）。没有它，你写的代码就是纯文本，芯片看不懂。

📦 安装 ARM GCC

方法1：winget安装（推荐，最快）

winget install ARM.GCC

方法2：手动下载

步骤1：访问 developer.arm.com/tools-and-software/open-source-software/developer-tools/gnu-toolchain
步骤2：下载 arm-gnu-toolchain-14.2.rel1-mingw-w64-i686-arm-none-eabi.exe（Windows x86安装器）
步骤3：运行安装器，记住安装路径（默认 C:\Program Files (x86)\Arm GNU Toolchain arm-none-eabi\14.2 rel1）
步骤4：勾选"Add to PATH"
⚠️ 网络问题：ARM官网下载可能很慢或被墙。winget方式通常更快。如果都下不了，可以找国内镜像或同学帮忙传安装包（约200MB）。

方法3：便携版（免安装）

# 下载zip版，解压到 D:\EmbeddedTools\arm-gcc\
# 手动添加到PATH：
# PowerShell（管理员）：
[Environment]::SetEnvironmentVariable(
    "Path",
    [Environment]::GetEnvironmentVariable("Path", "User") + ";D:\EmbeddedTools\arm-gcc\bin",
    "User"
)
# 重启Powerhell后生效

✅ 验证安装

# 检查版本（能输出版本号就说明安装成功）
arm-none-eabi-gcc --version

# arm-none-eabi-gcc (Arm GNU Toolchain 14.2.Rel1 (Build arm-14-5679)) 14.2.0

🔧 工具链命令详解

ARM GCC不只是gcc一个命令，它是一套工具。以下是每个命令的用途：

命令用途常用场景
arm-none-eabi-gccC/C++编译器编译.c文件 → .o目标文件
arm-none-eabi-g++C++编译器编译.cpp文件
arm-none-eabi-as汇编器编译.s/.S汇编文件
arm-none-eabi-ld链接器把.o文件链接成.elf（通常通过gcc调用）
arm-none-eabi-objcopy格式转换.elf → .bin / .hex
arm-none-eabi-objdump反汇编/查看查看.elf的反汇编代码、段信息
arm-none-eabi-size段大小统计查看Flash/RAM占用
arm-none-eabi-nm符号表查看查看函数/变量的地址映射
arm-none-eabi-readelfELF文件分析查看ELF头、段表、调试信息
arm-none-eabi-gdb调试器在线调试（配合OpenOCD/st-util）
arm-none-eabi-strip去调试信息减小.elf体积（发布时用）
arm-none-eabi-strings提取字符串从.bin中找可读字符串

💡 命名规则

所有命令前缀 arm-none-eabi- 的含义：

部分含义说明
arm目标架构ARM架构（Cortex-M系列）
none操作系统裸机（无操作系统）
eabiABI接口嵌入式应用二进制接口
💡 为什么不直接用gcc？你电脑上的gcc编译出来的是x86机器码（给CPU跑的），arm-none-eabi-gcc编译出来的是ARM机器码（给STM32跑的）。架构不同，必须用对应的编译器。

🏗️ 完整编译流程 🟢入门

从C代码到可烧录文件，要经过4个阶段：

编译四阶段核心知识
阶段输入 → 输出命令做什么
① 预处理.c → .igcc -E展开#include、#define、#ifdef
② 编译.i → .sgcc -SC代码 → ARM汇编代码
③ 汇编.s → .ogcc -c / as汇编代码 → 机器码目标文件
④ 链接.o → .elfgcc / ld合并所有.o + 启动文件 + 库 → 可执行文件

🚀 最简编译示例

# 项目结构
project/
├── src/
│   ├── main.c          # 你的代码
│   ├── startup.c       # 启动文件（向量表+Reset_Handler）
│   └── system.c        # SystemInit等系统函数
├── ld/
│   └── STM32F103C8.ld  # 链接脚本
└── inc/                # 头文件目录

# 步骤1：编译每个.c文件（生成.o）
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -Os -Wall -Iinc -c src/main.c -o build/main.o
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -Os -Wall -Iinc -c src/startup.c -o build/startup.o
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -Os -Wall -Iinc -c src/system.c -o build/system.o

# 步骤2：链接所有.o（生成.elf）
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -Tld/STM32F103C8.ld \
    -Wl,--gc-sections -Wl,-Map=build/app.map \
    build/main.o build/startup.o build/system.o \
    -o build/app.elf

# 步骤3：生成.bin（用于st-flash烧录）
arm-none-eabi-objcopy -O binary build/app.elf build/app.bin

# 步骤4：生成.hex（用于串口烧录/J-Link）
arm-none-eabi-objcopy -O ihex build/app.elf build/app.hex

# 步骤5：查看Flash/RAM占用
arm-none-eabi-size build/app.elf
#    text    data     bss     dec     hex filename
#   1234      20     200    1454     5ae build/app.elf
#   ↑Flash   ↑Flash  ↑RAM
#   Flash占 = text + data = 1254 字节
#   RAM占 = data + bss = 220 字节
💡 类比：编译就像做饭——
① 预处理 = 准备食材（展开头文件、宏）
② 编译 = 看菜谱写出步骤（C→汇编）
③ 汇编 = 按步骤加工食材（汇编→机器码）
④ 链接 = 把所有菜摆上桌（合并所有.o → .elf）
objcopy = 打包外卖（.elf → .bin/.hex）

⚙️ 编译参数详解 🟡进阶

必选参数（每次都要加）

参数含义说明
-mcpu=cortex-m3指定CPU型号STM32F1=M3, F4=M4, F0/G0=M0, H7=M7
-mthumb使用Thumb指令集Cortex-M只支持Thumb，必须加
-T xxx.ld指定链接脚本告诉链接器内存布局（Flash/RAM起止地址）

CPU型号对应表

STM32系列CPU参数代表型号
STM32F0/G0-mcpu=cortex-m0F030, G070
STM32F1-mcpu=cortex-m3F103C8（蓝色药丸）
STM32F4/F3/L4-mcpu=cortex-m4F407, L476
STM32F7-mcpu=cortex-m7F767
STM32H7-mcpu=cortex-m7H743
ESP32不用ARM GCC用xtensa-esp32-elf-gcc或ESP-IDF

优化参数

参数含义适用场景
-O0不优化调试用（断点准确，变量可查看）
-O1基本优化调试+少量优化
-O2标准优化发布常用
-Os体积优化嵌入式最常用，省Flash空间
-Og调试优化GDB调试体验好，代码不过度内联
💡 建议：开发阶段用-Og（调试友好），发布时用-Os（省空间）。绝不要用-O0，因为有些bug只在优化后出现，-Og是最佳平衡点。

警告参数

参数含义
-Wall开启常见警告
-Wextra更多警告
-Werror警告当错误处理（严格模式）
-Wno-unused-parameter忽略"未使用参数"警告（HAL回调常用）

链接参数（-Wl,xxx）

参数含义说明
-Wl,--gc-sections删除未用代码段配合-ffunction-sections -fdata-sections，大幅减小体积
-Wl,-Map=app.map生成map文件记录每个函数/变量的地址，排错用
-Wl,--print-memory-usage打印内存使用链接时显示Flash/RAM占用百分比
-Wl,--wrap=malloc包装函数可以拦截malloc调用做内存追踪

实用参数组合

# 调试版（开发阶段）
CFLAGS = -mcpu=cortex-m3 -mthumb -Og -g3 -Wall -Wextra \
         -ffunction-sections -fdata-sections -fno-common

# 发布版（烧录到产品）
CFLAGS = -mcpu=cortex-m3 -mthumb -Os -Wall -Wextra \
         -ffunction-sections -fdata-sections -fno-common \
         -DNDEBUG

# 链接参数
LDFLAGS = -mcpu=cortex-m3 -mthumb -TSTM32F103C8.ld \
          -Wl,--gc-sections -Wl,-Map=app.map \
          --specs=nano.specs --specs=nosys.specs

specs文件说明

参数含义效果
--specs=nano.specs使用精简C库printf/scanf体积减半，但浮点打印需额外配置
--specs=nosys.specs空系统调用提供空的_exit/_sbrk等，裸机必需
-u _printf_float启用浮点打印nano.specs默认不支持%f，加这个开启（体积+8KB）

📐 链接脚本（Linker Script）详解 🔴高级

链接脚本告诉编译器：Flash从哪开始、RAM从哪开始、各个段怎么排。这是嵌入式最核心的配置文件之一。

STM32F103C8 链接脚本

/* STM32F103C8T6: 64KB Flash, 20KB RAM */

/* 入口点（芯片上电后执行的第一条指令的地址） */
ENTRY(Reset_Handler)

/* 内存区域定义 */
MEMORY
{
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 64K
    RAM   (rwx) : ORIGIN = 0x20000000, LENGTH = 20K
}

/* 段布局 */
SECTIONS
{
    /* 中断向量表（必须放在Flash最开头！） */
    .isr_vector :
    {
        . = ALIGN(4);
        KEEP(*(.isr_vector))
        . = ALIGN(4);
    } > FLASH

    /* 代码段 */
    .text :
    {
        *(.text)
        *(.text*)
        *(.rodata)       /* 只读数据（const变量、字符串） */
        *(.rodata*)
        . = ALIGN(4);
        _etext = .;      /* 代码段结束标记 */
    } > FLASH

    /* 已初始化数据（Flash中存副本，启动时复制到RAM） */
    .data :
    {
        . = ALIGN(4);
        _sdata = .;      /* 数据段RAM起始 */
        *(.data)
        *(.data*)
        . = ALIGN(4);
        _edata = .;      /* 数据段RAM结束 */
    } > RAM AT > FLASH   /* 运行在RAM，加载在Flash */

    /* 未初始化数据（BSS段，启动时清零） */
    .bss :
    {
        . = ALIGN(4);
        _sbss = .;       /* BSS段起始 */
        *(.bss)
        *(.bss*)
        *(COMMON)
        . = ALIGN(4);
        _ebss = .;       /* BSS段结束 */
    } > RAM

    /* 堆栈定义 */
    ._user_heap_stack :
    {
        . = ALIGN(8);
        PROVIDE(end = .);       /* 堆起始 */
        . = . + 2K;            /* 堆大小 */
        . = ALIGN(8);
    } > RAM

    /* 丢弃不需要的段 */
    /DISCARD/ :
    {
        *(.ARM.attributes)
        *(.comment)
        *(.ARM.exidx*)
    }
}

关键符号的作用

符号含义谁用
_sdata / _edata.data段在RAM中的起止startup.c中把Flash数据复制到RAM
_sbss / _ebss.bss段的起止startup.c中把BSS段清零
_etext代码段结束地址知道Flash中数据副本的起始位置
end堆起始地址malloc使用的起始点
⚠️ AT > FLASH 是什么意思？.data段的运行地址在RAM（变量在RAM中读写），但加载地址在Flash（掉电不丢）。启动代码负责把Flash中的初始值复制到RAM。这就是startup.c中那段"看起来没用"的memcpy代码的作用。

不同MCU的内存地址

芯片Flash起始Flash大小RAM起始RAM大小
STM32F103C80x0800000064K0x2000000020K
STM32F103RE0x08000000512K0x2000000064K
STM32F407VG0x080000001M0x20000000192K
STM32F030F40x0800000016K0x200000004K
STM32H7430x080000002M0x200000001M
💡 规律：所有STM32的Flash都从0x08000000开始，RAM都从0x20000000开始。不同型号只是大小不同。改链接脚本只需改LENGTH。

🚀 启动文件（Startup Code）详解 🔴高级

启动文件是MCU上电后执行的第一段代码，在main()之前运行。它负责：初始化堆栈 → 复制.data → 清零.bss → 调用main()。

/* startup.c - 最小启动文件（STM32F103） */
#include <stdint.h>

/* 链接脚本中定义的符号 */
extern uint32_t _estack;    /* 栈顶（RAM末尾） */
extern uint32_t _etext;     /* Flash中代码段结束 */
extern uint32_t _sdata;     /* .data段RAM起始 */
extern uint32_t _edata;     /* .data段RAM结束 */
extern uint32_t _sbss;      /* .bss段起始 */
extern uint32_t _ebss;      /* .bss段结束 */

/* 主函数声明 */
int main(void);

/* 默认中断处理函数 */
void Default_Handler(void) {
    while(1);  /* 死循环，调试时断点会停在这里 */
}

/* Cortex-M3核心中断 */
void Reset_Handler(void);     /* 复位，入口点 */
void NMI_Handler(void)        __attribute__((weak, alias("Default_Handler")));
void HardFault_Handler(void)  __attribute__((weak, alias("Default_Handler")));
void MemManage_Handler(void)  __attribute__((weak, alias("Default_Handler")));
void BusFault_Handler(void)   __attribute__((weak, alias("Default_Handler")));
void UsageFault_Handler(void) __attribute__((weak, alias("Default_Handler")));
void SVC_Handler(void)        __attribute__((weak, alias("Default_Handler")));
void PendSV_Handler(void)     __attribute__((weak, alias("Default_Handler")));
void SysTick_Handler(void)    __attribute__((weak, alias("Default_Handler")));

/* 外设中断（STM32F103C8，按编号排列） */
void WWDG_IRQHandler(void)       __attribute__((weak, alias("Default_Handler")));
void PVD_IRQHandler(void)        __attribute__((weak, alias("Default_Handler")));
void TAMPER_IRQHandler(void)     __attribute__((weak, alias("Default_Handler")));
void RTC_IRQHandler(void)        __attribute__((weak, alias("Default_Handler")));
void FLASH_IRQHandler(void)      __attribute__((weak, alias("Default_Handler")));
void RCC_IRQHandler(void)        __attribute__((weak, alias("Default_Handler")));
void EXTI0_IRQHandler(void)      __attribute__((weak, alias("Default_Handler")));
void EXTI1_IRQHandler(void)      __attribute__((weak, alias("Default_Handler")));
void EXTI2_IRQHandler(void)      __attribute__((weak, alias("Default_Handler")));
void EXTI3_IRQHandler(void)      __attribute__((weak, alias("Default_Handler")));
void EXTI4_IRQHandler(void)      __attribute__((weak, alias("Default_Handler")));
void DMA1_Channel1_IRQHandler(void) __attribute__((weak, alias("Default_Handler")));
/* ... 其他中断省略，完整版约40个 ... */
void USART1_IRQHandler(void)     __attribute__((weak, alias("Default_Handler")));
void USART2_IRQHandler(void)     __attribute__((weak, alias("Default_Handler")));
void TIM1_UP_IRQHandler(void)    __attribute__((weak, alias("Default_Handler")));
void TIM2_IRQHandler(void)       __attribute__((weak, alias("Default_Handler")));
void TIM3_IRQHandler(void)       __attribute__((weak, alias("Default_Handler")));
void I2C1_EV_IRQHandler(void)    __attribute__((weak, alias("Default_Handler")));
void I2C1_ER_IRQHandler(void)    __attribute__((weak, alias("Default_Handler")));
void SPI1_IRQHandler(void)       __attribute__((weak, alias("Default_Handler")));

/* 中断向量表（必须放在Flash最开头，链接脚本中用.isr_vector段） */
__attribute__((section(".isr_vector")))
const void (* const vector_table[])(void) = {
    (void *)&_estack,          /* 栈顶指针（MSP初始值） */
    Reset_Handler,              /* 复位 */
    NMI_Handler,                /* NMI */
    HardFault_Handler,          /* HardFault */
    MemManage_Handler,          /* MemManage */
    BusFault_Handler,           /* BusFault */
    UsageFault_Handler,         /* UsageFault */
    0, 0, 0, 0,                 /* 保留 */
    SVC_Handler,                /* SVCall */
    DebugMon_Handler,           /* Debug Monitor - 可省略 */
    0,                          /* 保留 */
    PendSV_Handler,             /* PendSV */
    SysTick_Handler,            /* SysTick */
    /* 外设中断，从IRQ0开始... */
    WWDG_IRQHandler,            /* IRQ0 */
    PVD_IRQHandler,             /* IRQ1 */
    TAMPER_IRQHandler,          /* IRQ2 */
    RTC_IRQHandler,             /* IRQ3 */
    FLASH_IRQHandler,           /* IRQ4 */
    RCC_IRQHandler,             /* IRQ5 */
    EXTI0_IRQHandler,           /* IRQ6 */
    EXTI1_IRQHandler,           /* IRQ7 */
    /* ... 完整列表需查参考手册 ... */
};

/* Reset_Handler - 上电后第一个执行的函数 */
void Reset_Handler(void) {
    uint32_t *src, *dst;

    /* 1. 把.data段从Flash复制到RAM */
    src = &_etext;          /* Flash中数据副本的起始 */
    dst = &_sdata;          /* RAM中.data的起始 */
    while (dst < &_edata) {
        *dst++ = *src++;
    }

    /* 2. 把.bss段清零 */
    dst = &_sbss;
    while (dst < &_ebss) {
        *dst++ = 0;
    }

    /* 3. 调用SystemInit（如果有的话，配置时钟） */
    extern void SystemInit(void);
    SystemInit();

    /* 4. 调用libc初始化（如果使用nano.specs） */
    extern void __libc_init_array(void);
    __libc_init_array();

    /* 5. 终于进入main！ */
    main();

    /* 6. main不应该返回，如果返回则死循环 */
    while(1);
}
⚠️ 不要自己写startup.c！CubeMX和芯片厂商都提供了现成的启动文件（通常是.s汇编格式）。手写只是为了理解原理。实际项目直接用现成的。
💡 类比：启动文件就像开店前的准备工作——
① 设好栈顶 = 摆好收银台
② 复制.data = 把商品从仓库搬到货架
③ 清零.bss = 把空柜台擦干净
④ SystemInit = 检查水电
⑤ main() = 开门营业！

🔍 常见问题与排错 🟠实战

问题1：undefined reference to `_exit'

# 原因：没有指定nosys.specs，裸机缺少系统调用
# 解决：链接时加参数
--specs=nosys.specs

问题2：region `FLASH' overflowed

# 原因：代码太大，Flash装不下
# 解决1：改用-Os优化（体积优先）
# 解决2：加上段清理参数
-ffunction-sections -fdata-sections -Wl,--gc-sections
# 解决3：用nano.specs（精简C库，省数KB）
--specs=nano.specs
# 解决4：检查是否有大的const数组，考虑用PROGMEM/放在外部Flash

问题3：region `RAM' overflowed

# 原因：全局变量+栈太大，RAM装不下
# 解决1：减小栈大小（链接脚本中的栈定义）
# 解决2：减少全局数组，改用局部变量或动态分配
# 解决3：查看map文件找大头：哪个变量最占空间
arm-none-eabi-nm --size-sort --print-size app.elf | Select-Object -Last 20

问题4：Hard Fault（程序崩溃）

# 查看崩溃地址
arm-none-eabi-addr2line -e app.elf 0x08001234
# 输出：main.c:42（崩溃在第42行！）

# 查看反汇编（定位具体指令）
arm-none-eabi-objdump -d app.elf | Select-String "8001234"

问题5：.bin文件太大（比Flash大）

# .bin不是实际大小！它包含从0到最高地址的所有内容
# 如果高地址有数据（如OTA分区），.bin会很大
# 正确查看大小：
arm-none-eabi-size app.elf
# 看 text+data 列 = 实际Flash占用

问题6：printf不输出浮点数

# nano.specs默认省略浮点支持以节省空间
# 解决：链接时加上
-u _printf_float
# 注意：会增大约8KB Flash占用

🛠️ 实战案例：从零构建 STM32F103 LED 闪烁项目

下面演示如何只用 ARM GCC（不用 IDE）从零编译一个完整的 LED 闪烁程序：

// main.c - STM32F103 LED 闪烁（寄存器版本，无 HAL 依赖）
#include <stdint.h>

// RCC 寄存器基址（复位和时钟控制）
#define RCC_BASE      0x40021000U
// GPIOC 寄存器基址（LED 在 PC13）
#define GPIOC_BASE    0x40011000U

// RCC APB2 外设时钟使能寄存器
#define RCC_APB2ENR   (*(volatile uint32_t *)(RCC_BASE + 0x18))
// GPIOC 端口配置寄存器（高16位对应 Pin8-15）
#define GPIOC_CRH     (*(volatile uint32_t *)(GPIOC_BASE + 0x04))
// GPIOC 端口输出数据寄存器
#define GPIOC_ODR     (*(volatile uint32_t *)(GPIOC_BASE + 0x0C))

// 简单延时函数（约 500ms @72MHz）
static void delay(void) {
    for (volatile uint32_t i = 0; i < 500000; i++) {
        __asm volatile("nop");  // 插入空指令，防止被优化掉
    }
}

int main(void) {
    // 步骤1：使能 GPIOC 时钟（APB2 总线上第4位）
    RCC_APB2ENR |= (1 << 4);

    // 步骤2：配置 PC13 为推挽输出，最大速度 50MHz
    // CRH 控制 Pin8-15，Pin13 对应 [23:20] 四位
    // 通用推挽输出 = 0b0011，50MHz = CNF=00, MODE=11
    GPIOC_CRH &= ~(0xF << 20);  // 先清零 Pin13 的4位配置
    GPIOC_CRH |=  (0x3 << 20);  // 设置为输出模式，50MHz

    // 步骤3：循环闪烁 LED
    while (1) {
        GPIOC_ODR ^= (1 << 13);  // 翻转 PC13（异或操作）
        delay();                   // 延时约 500ms
    }
}

# 编译命令（逐步执行）

# 1. 编译 main.c → main.o
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb -Os -Wall \
    -ffunction-sections -fdata-sections \
    -c main.c -o main.o

# 2. 链接（使用最小启动代码）→ main.elf
arm-none-eabi-gcc -mcpu=cortex-m3 -mthumb \
    -nostartfiles -T STM32F103C8.ld \
    -Wl,--gc-sections \
    main.o -o main.elf

# 3. 生成 .bin 和 .hex
arm-none-eabi-objcopy -O binary main.elf main.bin
arm-none-eabi-objcopy -O ihex main.elf main.hex

# 4. 检查大小
arm-none-eabi-size main.elf
#   text    data     bss     dec     hex filename
#   184       0       0     184      b8 main.elf   ← 极小！
✅ 成果：纯寄存器操作，无 HAL 依赖，编译后仅 184 字节！这是理解底层硬件的最佳起点。后续可以用 STM32 HAL 库教程学习更高级的开发方式。
⚠️ 常见坑：如果 LED 不亮，检查：①是否使能了 GPIOC 时钟（最常遗漏）②PC13 配置是否正确（CRH 不是 CRL）③链接脚本中 Flash 起始地址是否为 0x08000000。更多排错方法见 烧录与调试教程。

❓ 常见问题 FAQ

ARM GCC 常见问题FAQ

Q1：arm-none-eabi-gcc 和 arm-linux-gnueabihf-gcc 有什么区别？

arm-none-eabi- 是 裸机工具链，用于无操作系统的嵌入式开发（STM32、ESP32 等）。arm-linux-gnueabihf- 是 Linux 工具链，用于开发运行在 ARM Linux 上的应用程序，依赖 glibc 和 Linux 系统调用。嵌入式开发必须选 arm-none-eabi-。

Q2：编译时出现 multiple definition of `xxx' 怎么办？

通常是因为头文件中定义了变量（不是声明）。正确做法：头文件中用 extern 声明，在 .c 文件中定义。GCC 14.x 默认使用 -fno-common，不再允许多次定义合并。

Q3：-Os 和 -O2 该选哪个？

嵌入式一般选 -Os（优化代码大小），因为 Flash 空间有限。-O2 优化速度但体积更大。如果对性能敏感的中断处理函数，可以对特定文件用 -O2：在 Makefile 中用 CFLAGS_isr = -O2 单独设置。

Q4：如何查看编译后各个函数占用的 Flash 大小？

# 生成详细段大小信息
arm-none-eabi-objdump -h app.elf

# 按符号大小排序（找出最占空间的函数）
arm-none-eabi-nm --size-sort --print-size app.elf | tail -20

# 生成 map 文件查看详细内存映射
arm-none-eabi-gcc ... -Wl,-Map=app.map ...

Q5：链接脚本中 VMA 和 LMA 有什么区别？

VMA（Virtual Memory Address）是程序运行时的地址；LMA（Load Memory Address）是程序存储在 Flash 中的地址。通常 VMA = LMA（代码直接在 Flash 运行），但 重定位代码（如从 Flash 拷贝到 RAM 执行）时 VMA ≠ LMA。详见 烧录与调试教程中的 IAP 章节和 Makefile 教程中的链接脚本配置。

Q6：GCC 14.x 升级后代码变大了怎么办？

GCC 14 默认启用了 -fstack-protector-strong，每个函数添加栈保护代码。可用以下方法优化：① -fno-stack-protector 关闭（牺牲安全性）② -fstack-protector 改用轻量版 ③ 仅对关键函数单独启用。相关内容参见 CMake 教程中的 per-target 编译选项。
